# 하단 탭 `prefetch` 활성화 정책 (Write-up)

## 목적
- 탭 전환 시 “클릭 후 첫 화면이 뜨는 시간”을 줄여 사용자 체감 속도를 높인다.
- 성능 개선은 서버-클라이언트 데이터 구조 개선과 별개로, 라우트 전환 UX(대기/깜빡임)를 함께 최적화한다.

## 배경: `prefetch`가 하는 일
- Next.js App Router에서 `next/link`는 기본적으로 `prefetch`를 수행할 수 있다.
- `prefetch`는 “사용자가 그 탭으로 이동하기 전에”, 해당 라우트에 필요한 데이터/번들을 백그라운드로 준비하는 동작이다.
- 따라서 탭을 실제로 누를 때, 준비가 되어 있으면 첫 렌더까지 대기 시간이 줄어든다.

## 왜 활성화했나?
- 우리 화면 구조에서 하단 탭바(`홈`, `대회`, `랭킹`, `프로필`)는 대부분의 사용 흐름에서 항상 노출된다.
- 탭바가 보이는 동안 Next의 기본 프리페치 동작을 허용하면, 사용자가 자주 이동하는 라우트 전환 체감이 전반적으로 좋아질 가능성이 높다.
- 이전에 “사용자 입장에서 홈탭 개선이 체감되지 않는다”는 학습이 있었고, 서버 데이터 구조 개선 외에 “전환 UX 병목(라우트 준비 대기)”이 남아있을 수 있다고 판단했다.

## 우리 소스에서 어떤 부분에서 이득을 보나?
- 변경 위치: [`components/bottom-tab-bar.tsx`](components/bottom-tab-bar.tsx)
- 기존 코드: 탭 `Link`에 `prefetch={false}`가 지정되어 Next의 기본 프리페치가 꺼져 있었다.
- 변경: `prefetch={false}`를 제거하여 Next 기본 프리페치를 활성화한다.

## 기대 효과(사용자 체감)
- 탭 클릭 이후 “본문 데이터 렌더 + 다음 페인트”까지의 대기 시간이 감소한다.
- `components/perf/tab-load-probe.tsx`의 `TabLoadProbe` 로그를 통해 (전/후 비교) 탭 전환 체감 개선을 확인할 수 있다.

## 단점/리스크(추후 문제가 될 수 있는 지점)
- 프리페치가 켜지면 사용자가 보이지 않는 탭 경로도 백그라운드로 준비될 수 있어,
  - 초기 네트워크 요청량 증가
  - Supabase 호출/서버 리소스 사용 증가
  - 모바일/저데이터 모드에서 데이터 사용량 증가
  같은 문제가 생길 수 있다.

## 대응 전략(문제가 확인되면 어떻게 조정할까?)
- 측정 근거 기반으로 단계적 완화:
  - 백그라운드 요청이 실제로 과도한 경우, 탭 전체 프리페치 대신 “우선순위 탭 1~2개만” 프리페치하도록 조건을 둔다.
  - 네트워크 상태가 `save-data`거나 저대역폭으로 판단되면 프리페치를 줄인다(예: `prefetch={false}`를 조건부 적용).
  - 라우트 레벨 로딩(`app/(main)/loading.tsx`) 같은 UX 장치를 함께 사용하면, 프리페치 실패/캐시 미스 상황에서도 체감이 덜 거슬리게 만들 수 있다.

## 검증 방법(권장)
- 동일한 브라우저/네트워크 환경에서 탭 전환을 여러 번 수행하고 `TabLoadProbe` 로그를 비교한다.
- 비교는 최소한 “첫 진입/캐시 미스”와 “재진입/캐시 히트”를 분리해 보는 것을 권장한다.
