# `(main)` 라우트 로딩 스켈레톤 추가 Write-up

## 목적
- 탭 전환 시 “콘텐츠가 로드되기 전까지의 공백/지연”을 사용자에게 즉시 피드백으로 바꾸기.
- 실제 데이터 준비 시간이 같아도, 사용자 체감 대기를 줄인다.

## 배경: 라우트 레벨 로딩이 필요한 이유
- Next.js App Router는 `app/(segment)/loading.tsx`가 존재하면, 해당 세그먼트로 이동할 때 라우트 수준에서 `loading UI`를 먼저 보여줄 수 있다.
- 우리 프로젝트는 `app/(info)/`, `app/(protected)/`, `app/auth/`에는 `loading.tsx`가 이미 있었지만, **`app/(main)`에는 없었다.**
- 결과적으로 탭 전환 시 라우트 수준의 로딩 UI가 뜨지 않고, 페이지 내부 `Suspense`/서버 데이터 완료까지 기다리는 형태가 되어 사용자 입장에서 “멈춘 느낌”이 날 수 있었다.

## 변경 내용(소스)
- 추가 파일: [`app/(main)/loading.tsx`](app/(main)/loading.tsx)
  - `MainLoadingSkeleton` 컴포넌트를 기본 로딩 UI로 반환.
  - 하단 탭 공통 레이아웃 상단 제목 영역 + 카드 영역 스켈레톤을 포함해 메인 탭 전환 시 시각적 공백을 최소화.

## 우리 소스에서 얻는 이득
- 탭 전환 이후 `loading.tsx`가 먼저 노출되어, 사용자에게 “지금 로딩 중”임을 즉시 전달.
- `components/perf/tab-load-probe.tsx`의 `TabLoadProbe` 측정 관점에서도 “탭 클릭 → 콘텐츠 표시(ms)”와 함께 “탭 클릭 → 로딩 노출” 시간을 분리해서 체감 차이를 확인할 수 있다.

## 기대 효과(사용자 체감)
- 네트워크가 느리거나 캐시 미스일 때도 탭 전환 직후 스켈레톤이 보이므로, 빈 화면/지연 체감이 감소.
- 특히 우리처럼 하단 탭 전환이 잦은 앱에서 “반응성”이 더 좋아 보인다.

## 단점/리스크(추후 문제 가능 지점)
- 스켈레톤이 실제 레이아웃과 다르면 레이아웃 점프(shift)나 “스켈레톤이 깜빡이는 느낌”이 생길 수 있다.
- 페이지 내부에서 이미 `Suspense fallback`을 사용하고 있다면, 특정 상황에서 스켈레톤이 중복으로 보일 수 있다.
- 캐시 히트 상황에서도 아주 짧게 로딩 UI가 깜빡일 가능성(환경/렌더링 타이밍 의존)이 있다.

## 대응 전략
- 검증 결과 스켈레톤이 “깜빡임/중복”이 문제면:
  - (1) 페이지 내부 `Suspense fallback`의 범위를 조정하거나,
  - (2) `loading.tsx` 스켈레톤의 컴포넌트 형태를 더 단순/공통화해서 레이아웃 미스매치를 줄이거나,
  - (3) 스켈레톤 노출 빈도가 높다면 라우트 내부 구조를 스트리밍 친화적으로 재조정한다.
- 레이아웃 점프가 관측되면 스켈레톤의 높이/여백을 실제 첫 화면 구성과 더 가깝게 맞춘다.

## 검증 방법(권장)
- `TabLoadProbe` 로그 기준으로 탭 전환 전/후 비교:
  - 탭 클릭 직후 로딩 UI가 얼마나 빨리 보였는지(체감/시각)
  - “콘텐츠 표시” ms가 어떻게 변했는지
- 홈/대회/랭킹/프로필 탭에서 각각 2~3회씩 확인해 중복/깜빡임이 없는지 확인.
