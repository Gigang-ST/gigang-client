1. **`act_dt` 컬럼 추가.** 원안엔 `crt_at`만 있었다. 팻말 하단에 찍히는 날짜는 **뛴 날**이지

2. **RPC를 `get_team_story_feed`에 얹지 않고 `get_team_posts`로 분리.** §4는 `recent_pledges`
   CTE를 갈아끼우거나 `posts` 키를 추가하는 두 안을 놓고 열어뒀는데, 셋째 길을 택했다. 그 함수는
   이미 CTE 10개+이고 운영 중이라 존을 얹을 때마다 재배포하는 위험이 이득보다 크고, 무효화 주기가
   다르다 — 자랑은 올린 즉시(`story-posts` 태그), 피드 본문은 5분이면 된다. 한 태그면 자랑 한 건이
   피드 전체 캐시를 끌고 내려간다.
3. **`pledge` 이관은 하지 않았다.** `post_mst`는 두 타입을 담을 형태로 만들되 실제로는
   `record_flex`만 쓴다. 각오는 그대로 `pldg_mst`에 있다 — 같은 날 각오 UI를 종이비행기로
   교체한 참이라 같은 경로를 연달아 두 번 흔들지 않았다. §6-1/§6-3은 그대로 열려 있다.
4. **팻말의 본문은 "제목"이 아니라 "한마디"**(`cmnt_txt`). 상한은 200자로 마일리지런
   `review`와 맞췄다 — 두 경로가 같은 컬럼에 담기므로 길이가 갈리면 자동 유입분만 잘려 보인다.

### 안 된 것 (다음)

- **§3 마일리지런 자동 유입 트리거** — 미구현. 방식은 **A안(트리거 동기화)으로 확정**(2026-07-24 결정).
- **중복 게시 우려의 결론**: 별도 안내 문구나 중복 감지 UI를 만들지 않기로 했다. 자동 유입분이
  반쪽(사진 없음)이라 사용자가 다시 올릴 유인이 생기는 게 문제의 뿌리인데, 그건 **마일리지런 기록
  입력에도 사진 첨부를 추가해 원천에서 정합을 맞추는 것**으로 푼다(후속). 그때까지는 "전광판에서
  기록을 올릴 수 있고, 마일리지런 기록도 여기 같이 보인다" 수준의 인지면 충분하다는 판단.
  → §6-4(자동 유입분 편집 권한)는 이 방향과 함께 다시 볼 것.

## 1. 배경 — 왜 "각오 확장"인가

지금 `pldg_mst`는 "각오(한 줄 다짐)" 전용이다. 사용자는 기강이야기(전광판)에 멤버가 **러닝 기록을
자랑하는** 기능을 원한다. 두 콘텐츠는 형태가 다르다:

| | 각오 | 기록 자랑 |
|---|---|---|
| 필수 요소 | 텍스트(40자) | 사진 + 기록(거리/시간 등) + 한마디 |
| 갱신 빈도 | 낮음, 누적 | 마일리지런 기록 시 자동 발생 가능 |
| 의미 | "다짐" | "결과 보고" |

**방향(확정)**: `pldg_mst`를 독립 유지하지 않고, "포스트" 계열로 일반화한다. 각오와 기록자랑을
같은 테이블 계열의 두 `post_type`으로 담아 전광판이 하나의 피드 소스에서 시간순으로 섞어 보여줄 수
있게 한다. 이유:

1. **UI 결이 같다** — 둘 다 "멤버가 자발적으로 남긴 한마디"이고 전광판의 같은 시간대(`pledges` 존
   또는 그 확장)에 뜬다. 별개 테이블 + 별개 UNION을 story-feed RPC에 계속 추가하는 것보다, 애초에
   하나의 조회 경로로 묶는 편이 RPC를 덜 비대하게 만든다(`get_team_story_feed`는 이미 CTE 10개+).
2. **마일리지런 자동 유입과 자연 결합** — 마일리지런 기록의 `review`(한마디)를 "기록 자랑"으로
   흘려보내려면 어차피 "사진 없는 기록 자랑" 형태를 지원해야 한다. 포스트 테이블이 사진을
   nullable로 두면 마일리지런 유입분과 사용자가 직접 올리는 사진 포함 게시물이 같은 셰이프를 공유한다.

## 2. 스키마 설계 (DDL 초안 — 텍스트 참고용, 미적용)

### 2.1 명명 결정

`pldg_mst`를 리네임하지 않고 **새 테이블 `post_mst`를 신설**하고 `pldg_mst`는 당분간 유지하는 안과,
`pldg_mst`를 확장해 재사용하는 안 두 가지를 검토했다.

**결정: 신규 테이블 `post_mst` + 기존 `pldg_mst` 데이터 마이그레이션 후 폐기(추후 별도 마이그레이션)**

근거:
- `pldg_mst`라는 이름 자체가 "각오"에 고정돼 있어 기록자랑까지 담으면 이름이 실체와 어긋난다
  (`AGENTS.md`의 네이밍 컨벤션 — `_mst` 접미사는 유지하되 어근을 실체에 맞게 바꿔야 함).
- 이미 운영 중인 테이블(row 7건, RLS 정책 3개, 인덱스 2개, story-feed RPC가 이미 조회 중)을
  in-place로 컬럼 추가하며 의미를 바꾸면, 기존 `pldg_mst_insert` 정책의 `CHECK` 제약(40자 텍스트
  전용)과 새 포스트 타입의 제약이 뒤섞여 정책이 복잡해진다.
- 신규 테이블로 가면 `pldg_mst`의 기존 7건을 `INSERT ... SELECT`로 이관하는 **일회성 백필
  마이그레이션**만 추가하면 되고, story-feed RPC의 `recent_pledges` CTE를 `post_mst` 조회로
  갈아끼우는 변경 범위가 명확해진다. 이 백필/전환 자체는 이번 설계 스콥 밖(별도 작업으로 분리)이며,
  이 문서는 `post_mst` 신설과 마일리지런 자동 유입 방식까지만 다룬다.

### 2.2 `post_mst` (신규)

```sql
CREATE TYPE public.post_type_enm AS ENUM (
  'pledge',      -- 각오(한 줄 다짐) — pldg_mst 후신
  'record_flex'  -- 기록 자랑(사진 + 기록 + 한마디)
);

CREATE TYPE public.post_src_enm AS ENUM (
  'manual',   -- 사용자가 전광판에서 직접 작성
  'mlg_auto'  -- 마일리지런 기록 등록 시 자동 생성
);

CREATE TABLE public.post_mst (
  post_id       uuid          PRIMARY KEY DEFAULT gen_random_uuid(),
  team_id       uuid          NOT NULL REFERENCES public.team_mst(team_id),
  mem_id        uuid          NOT NULL REFERENCES public.mem_mst(mem_id),
  post_type_enm public.post_type_enm NOT NULL,
  src_enm       public.post_src_enm  NOT NULL DEFAULT 'manual',

  -- 각오(pledge)일 때만 사용 — record_flex는 NULL
  pldg_txt      text,

  -- 기록자랑(record_flex)일 때만 사용
  photo_url     text,          -- Storage 공개 URL, nullable(마일리지런 자동 유입분은 사진 없음)
  cmnt_txt      text,          -- 한마디 (마일리지런 review 필드와 동일 역할)
  dst_km        numeric,       -- 거리 — 마일리지런/수동 둘 다 채움
  sprt_enm      text,          -- 종목 표기 (마일리지런 evt_mlg_sprt_enm 값 그대로 문자열 저장, §3.2)

  -- 자동 유입 원천 역참조 (감사·중복 방지). src_enm='mlg_auto'일 때만 값 존재.
  ref_type_txt  text,          -- 'mlg_act' 고정(현재는), 추후 대회기록 자동유입 등 확장 대비
  ref_id        uuid,          -- evt_mlg_act_hist.act_id

  del_yn        boolean       NOT NULL DEFAULT false,
  crt_at        timestamptz   NOT NULL DEFAULT now(),
  upd_at        timestamptz   NOT NULL DEFAULT now(),

  CONSTRAINT ck_post_mst_pldg_txt_len
    CHECK (post_type_enm <> 'pledge' OR char_length(pldg_txt) BETWEEN 1 AND 40),
  CONSTRAINT ck_post_mst_record_flex_fields
    CHECK (post_type_enm <> 'record_flex' OR cmnt_txt IS NOT NULL),
  -- 같은 마일리지런 기록에서 중복 유입 방지 (§3.3)
  CONSTRAINT uq_post_mst_mlg_ref UNIQUE (ref_type_txt, ref_id)
);

COMMENT ON TABLE public.post_mst IS
  '전광판 포스트 통합 — 각오(pledge)와 기록자랑(record_flex)을 함께 담는다. pldg_mst 후신.';
COMMENT ON COLUMN public.post_mst.src_enm IS
  'manual=사용자가 직접 작성, mlg_auto=마일리지런 기록 등록 시 자동 생성(§3 참조)';
COMMENT ON COLUMN public.post_mst.ref_id IS
  '자동 유입 원천 PK. src_enm=mlg_auto면 evt_mlg_act_hist.act_id — 원본 기록 삭제 시 이 포스트도 함께 정리(§3.4)';

CREATE INDEX ix_post_mst_team_recent
  ON public.post_mst (team_id, crt_at DESC)
  WHERE del_yn = false;

CREATE INDEX ix_post_mst_mem
  ON public.post_mst (mem_id)
  WHERE del_yn = false;

ALTER TABLE public.post_mst ENABLE ROW LEVEL SECURITY;

-- SELECT: 전광판 공개 노출 (pldg_mst_select와 동일 정책)
CREATE POLICY post_mst_select ON public.post_mst
  FOR SELECT USING (del_yn = false);

-- INSERT: 로그인 + 본인 mem_id + 소속 팀. mlg_auto는 트리거(SECURITY DEFINER)가 쓰므로
-- 이 정책은 사실상 manual(사용자 직접 작성) 경로만 통과한다 — 트리거는 RLS를 우회한다.
CREATE POLICY post_mst_insert ON public.post_mst
  FOR INSERT WITH CHECK (
    mem_id = public.v2_rls_resolve_mem_id()
    AND public.v2_rls_auth_in_team(team_id)
    AND del_yn = false
    AND src_enm = 'manual'  -- 클라이언트발 INSERT는 항상 manual만 허용
  );

-- UPDATE: 본인 포스트만 (pldg_mst_update와 동일)
CREATE POLICY post_mst_update ON public.post_mst
  FOR UPDATE USING (mem_id = public.v2_rls_resolve_mem_id())
  WITH CHECK (
    mem_id = public.v2_rls_resolve_mem_id()
    AND public.v2_rls_auth_in_team(team_id)
  );
```

**미확정 사항(§4에서 결정 필요)**: `photo_url`을 여러 장 허용할지(현재는 1장, text 단일 컬럼).
여러 장이 필요하면 `post_photo_rel(post_id, photo_url, sort_ord)` 별도 테이블로 분리.

## 3. 마일리지런 자동 유입 방식

### 3.1 요구사항

`evt_mlg_act_hist.review`(한마디)가 있는 활동 기록은 사용자가 별도 조작 없이 전광판 포스트
피드에 "기록 자랑"으로 나타나야 한다.

### 3.2 후보 비교

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. INSERT 트리거로 post_mst 행 생성** | `evt_mlg_act_hist` AFTER INSERT/UPDATE/DELETE 트리거가 `post_mst`에 `src_enm='mlg_auto'` 행을 동기화 | 조회는 `post_mst` 하나만 보면 됨. story-feed RPC가 단순해짐. `del_yn` 소프트삭제·수정 이력 관리가 자연스러움(pt_txn_hist 트리거와 동일 컨벤션) | 원본(`evt_mlg_act_hist`)과 사본(`post_mst`)의 동기화 책임이 트리거에 생김 — UPDATE 시 review가 지워지면 포스트도 지워야 하는 등 케이스가 늘어남 |
| **B. 조회 시 UNION (뷰 또는 RPC 내 CTE)** | `post_mst`엔 manual만 저장. story-feed RPC(또는 별도 뷰)가 조회 시점에 `evt_mlg_act_hist WHERE review IS NOT NULL`을 UNION | 원본 단일 진실 소스 — 동기화 버그 원천 차단. 스키마 변경 없이 review 편집이 즉시 반영됨 | RPC가 더 복잡해짐(이미 CTE 10개+인 get_team_story_feed에 조인 추가). "포스트 좋아요/댓글" 같은 포스트 고유 기능이 생기면 UNION된 가짜 행에는 post_id가 없어 확장이 막힘 |
| **C. 뷰(view)로 통합 후 RPC는 뷰만 조회** | `post_feed_v`라는 뷰가 `post_mst UNION ALL (evt_mlg_act_hist WHERE review IS NOT NULL)`를 미리 정의, RPC는 이 뷰 하나만 SELECT | B보다 RPC 본문은 깔끔 | 근본적으로 B와 동일한 한계(가짜 post_id, 포스트 고유 기능 확장 불가) + 뷰 계층이 하나 늘어 디버깅 시 추적할 곳이 많아짐 |

### 3.3 결정: **A. 트리거 동기화**

**이 코드베이스의 결과 맞는 이유**:

1. `2026-07-04-기강포인트제도.md`가 이미 정확히 같은 문제(원천 테이블 여러 곳에 흩어진 쓰기 →
   파생 원장 자동 반영)를 **트리거**로 풀기로 결정했고, 그 근거(§4.1 "쓰기 경로가 흩어져 있음",
   "유실 없음 — fire-and-forget 앱 훅과 달리 원천과 같은 트랜잭션")가 이 케이스에도 동일하게
   적용된다. `evt_mlg_act_hist`는 이미 `trg_pt_evt_mlg_act_hist` 트리거가 붙어 있어 INSERT 시
   포인트를 적립 중 — 같은 테이블에 트리거를 하나 더 얹는 게(또는 같은 트리거 함수를 확장하는 게)
   이 프로젝트의 기존 결이다.
2. 포스트가 "좋아요/응원(`rctn_mst`)·댓글" 등 자기 고유 기능을 갖게 될 가능성이 높다(전광판은
   이미 `records`/`newbies`/`races`에 리액션을 붙이는 패턴을 갖고 있다 — `rctn_mst`가 `entity_type`
   범용 설계다). UNION 방식(B/C)은 가짜 행에 안정적인 `entity_id`가 없어 이 확장을 막는다.
   트리거 방식(A)은 `post_mst.post_id`가 실제 PK라 `rctn_mst`의 `entity_id`로 바로 쓸 수 있다.
3. 동기화 버그 리스크(A의 단점)는 마일리지런 기록의 **수정 빈도가 낮고**(기록 입력 후 거의
   불변), 트리거 로직을 "review 유무로 존재 여부 판정"이라는 단순 규칙 하나로 고정하면 관리
   가능하다(§3.4).

### 3.4 트리거 동작 규칙

```text
evt_mlg_act_hist AFTER INSERT:
  review IS NOT NULL AND btrim(review) <> '' 이면
    post_mst INSERT (src_enm='mlg_auto', ref_type_txt='mlg_act', ref_id=NEW.act_id,
                      post_type_enm='record_flex', mem_id=<prt_id로 evt_team_prt_rel 조인>,
                      team_id=<evt_id로 evt_team_mst 조인>,
                      cmnt_txt=NEW.review, dst_km=NEW.dst_km, sprt_enm=NEW.sprt_enm::text,
                      photo_url=NULL, crt_at=NEW.created_at)
    -- uq_post_mst_mlg_ref가 중복 INSERT를 막아준다(멱등)

evt_mlg_act_hist AFTER UPDATE:
  review가 NULL/빈문자로 바뀜  → 기존 post_mst 행(ref_id=NEW.act_id) soft delete(del_yn=true)
  review가 새로 채워짐(이전엔 없었음) → post_mst INSERT (INSERT 케이스와 동일)
  review 내용만 바뀜(계속 존재)  → post_mst UPDATE (cmnt_txt=NEW.review, upd_at=now())
  dst_km/sprt_enm 변경          → 대응 post_mst 행도 함께 UPDATE (표시값 정합성)

evt_mlg_act_hist AFTER DELETE:
  대응 post_mst 행(ref_id=OLD.act_id) soft delete(del_yn=true)
  -- 원본 기록 자체가 지워졌으니 포스트도 사라져야 자연스럽다. hard delete가 아니라 soft인 이유는
  -- pldg_mst/post_mst 전체 컨벤션(soft delete)과 일치시키기 위함 — 감사 추적도 겸함.
```

- **team_id/mem_id 조인 경로**: 작업 1에서 확정한 것과 동일 —
  `evt_mlg_act_hist.prt_id → evt_team_prt_rel(mem_id, evt_id) → evt_team_mst.team_id`.
- **작성자 표시**: `mem_id`는 마일리지런 참여자 본인 그대로 — "자동 유입"이어도 전광판에는
  본인이 쓴 것처럼 보인다(실제로 본인이 입력한 review이므로 자연스럽다).
- **manual 포스트와 구분 표시가 필요한가**: UI 스콥이라 이 문서에선 결정하지 않음. `src_enm`
  컬럼으로 프론트가 원하면 "마일리지런에서" 같은 배지를 붙일 수 있다(§4 열린 결정).

## 4. story-feed 연결 지점

현재 `get_team_story_feed`의 `recent_pledges` CTE는 `pldg_mst`를 직접 조회해
`pledges` 키로 반환한다. `post_mst` 전환 후:

- `recent_pledges` CTE의 FROM을 `pldg_mst` → `post_mst`로 교체, `WHERE post_type_enm = 'pledge'`
  조건 추가하거나, 아예 `pledges` 키를 없애고 **`posts`라는 새 키**로 pledge+record_flex를
  함께 반환(시간순 통합 피드)하는 두 가지 안이 있다 — 이는 §5 열린 결정사항.
- `StoryFeed` 타입(`lib/queries/story-feed.ts`)의 `StoryPledge`도 이에 맞춰 `StoryPost`로
  일반화하거나 유니온 타입(`StoryPost = StoryPledgePost | StoryRecordFlexPost`)이 필요.
  **이 파일은 프론트엔드 연결 지점이라 이번 스콥에서 건드리지 않는다** — 구현 단계에서
  프론트 담당이 반영.
- 리액션(`rctn_mst`) 연동을 원하면 `entity_type`에 `'post'`(또는 `'record_flex'`)를 추가하고
  `RCTN_CODES`(`lib/queries/story-feed.ts`)에 이미 있는 6종 중 적합한 것(`fire`?)을 기본
  리액션으로 매핑 — `records`가 `fire`를 쓰는 것과 통일감 있게.

## 5. 사진 업로드 — Storage 패턴

기존 아바타 업로드(`app/actions/update-profile.ts`)가 유일한 선례다. 그대로 따를 것:

| 항목 | 아바타 패턴 | 기록자랑 적용안 |
|------|------------|----------------|
| 버킷 | `avatars` (public, RLS: `authenticated`면 전체 insert/update/delete 허용, SELECT는 완전 공개) | **신규 버킷 `post-photos`** 신설 — avatars와 목적이 다르고(자랑 사진은 원본 화질·다양한 비율 요구), 버킷을 분리해야 향후 용량 정책(파일 크기 제한 등)을 따로 걸 수 있다 |
| 경로 규칙 | `{member.id}/{Date.now()}.webp` | `{member.id}/{post_id 또는 Date.now()}.webp` — 아바타처럼 멤버별 폴더 |
| 리사이즈 | `sharp`로 512x512 cover + webp 80% | 기록자랑은 정사각 크롭이 아니라 원본 비율 유지가 자연스러움 — `sharp .resize({width: 1080, withoutEnlargement: true}).webp({quality: 80})` 정도로 폭만 제한 (세부는 프론트/디자인 결정 필요) |
| HEIC 변환 | `heic-convert` 라이브러리 사용 | 동일하게 재사용 |
| 업로드 주체 | 서버 액션 내부에서 `withActive`로 인증된 `supabase`(사용자 세션) 클라이언트로 업로드 | 동일 패턴 — 별도 admin 클라이언트 불필요, 세션 클라이언트가 RLS 통과 |
| 파일 정리 | DB 커밋 성공 후에만 기존 파일 삭제(트랜잭션처럼 순서 보장) | 기록자랑은 "기존 파일 대체"가 없으므로(포스트마다 새 사진) 이 로직은 불필요. 대신 포스트 soft-delete 시 Storage 파일을 즉시 지울지, 남겨둘지는 §6 열린 결정 |

**RLS 정책 초안(참고용, 신규 버킷 생성 시)**:

```sql
-- storage.objects, bucket_id = 'post-photos'
CREATE POLICY post_photos_public_read ON storage.objects
  FOR SELECT USING (bucket_id = 'post-photos');

CREATE POLICY post_photos_auth_insert ON storage.objects
  FOR INSERT WITH CHECK (bucket_id = 'post-photos' AND auth.role() = 'authenticated');

CREATE POLICY post_photos_auth_delete ON storage.objects
  FOR DELETE USING (bucket_id = 'post-photos' AND auth.role() = 'authenticated');
-- UPDATE 정책은 불필요할 수 있음 — 아바타와 달리 "같은 경로 덮어쓰기"가 없다면
```

## 6. 열린 결정사항 (구현 전 확정 필요)

1. **`pledges` → `posts` 통합 방식**: story-feed RPC가 pledge/record_flex를 하나의 시간순 배열로
   섞어 낼지, 아니면 지금처럼 별도 키로 유지할지. 섞으면 프론트 전광판 컴포넌트가 타입 분기를
   렌더링 단에서 처리해야 함.
2. **사진 여러 장 허용 여부**: `photo_url` 단일 컬럼 vs `post_photo_rel` 별도 테이블. 마일리지런
   앱(있다면) 원본 사진 여러 장을 이미 갖고 있는지도 확인 필요(§3의 자동 유입은 현재 스키마상
   `evt_mlg_act_hist`에 사진 컬럼이 없어 항상 `photo_url = NULL`).
3. **`pldg_mst` 폐기 시점과 방식**: 이 문서는 `post_mst` 신설까지만 다룬다. 기존 7건 이관 +
   `pldg_mst` 드롭 + `create-pledge.ts` 서버 액션의 대상 테이블 교체는 후속 마이그레이션/PR로
   분리할지, 같은 PR에 포함할지.
4. **`src_enm='mlg_auto'` 포스트의 편집/삭제 권한**: 사용자가 마일리지런 기록의 review를
   수정하면 트리거가 포스트도 갱신하는데(§3.4), 반대로 **전광판에서 포스트를 직접 수정**하면
   원본 `evt_mlg_act_hist.review`에 역으로 반영해야 하는지, 아니면 자동 유입 포스트는
   전광판에서 읽기 전용으로 둘지.
5. **리액션 연동 여부와 매핑 리액션 코드**: §4에서 언급한 `entity_type` 확장을 이번 기능에
   포함할지, 후속으로 미룰지.
6. **사진 리사이즈 파라미터**: 폭 제한값(1080px 등), 품질(80%), 최대 용량(아바타는 10MB 제한 —
   기록자랑 사진도 동일 상한을 둘지)은 디자인/프론트 협의 필요.
7. **마일리지런 자동 유입에 "감사 로그"가 필요한가**: `2026-07-04-기강포인트제도.md`의
   포인트 원장처럼 append-only가 필요한 성격은 아니라고 판단해 이 문서는 소프트삭제+UPDATE로
   설계했으나, 추후 "포스트 수정 이력"이 요구되면 별도 이력 테이블이 필요.
