# 크루 진행 차트(마일리지/달성률) — “11명 뷰” 설계

## 배경 / 목적

- 동시에 많은 선을 그리면 색·범례·툴팁 가독성이 급격히 떨어진다.
- 따라서 **차트(라인)에서는 최대 11명만 표시**하고, 전체 데이터는 별도 **통계 탭**에서 제공한다.
- 본 문서는 `components/projects/crew-progress-chart.tsx` 및 관련 서버 래퍼(`crew-progress-chart-server.tsx`) 구현 시 **단일 기준(Single Source of Truth)** 으로 삼는다.

## 용어

- **멤버**: `mem_id` + 표시명(`mem_nm`)
- **모드**
  - **마일리지 모드**: 누적 km 기반
  - **달성률 모드**: 달성률 % 기반
- **기준일(`d_ref`)**: “상/하위/주변/제외” 판단에 사용하는 일자. 별도의 ‘스냅샷 엔티티’가 DB에 존재한다는 뜻이 아니라, **렌더링 시점에 계산 기준으로 삼는 날짜**를 의미한다.
- **표시 대상(11명)**: `d_ref` 시점의 값으로 선별된 멤버 집합

## 범위(이번 설계가 다루는 것)

- 라인 차트에서 **표시 멤버 수를 최대 11명으로 제한**하는 규칙
- **0km 제외** 규칙(“기록이 없는 사람”을 차트에서 제외)과 그에 따른 엣지케이스
- 마일리지/달성률 모드별 **랭킹 기준 분리**
- 동률 처리(소수 첫째 자리까지 동일 → 이름순)
- “내 주변(위2/아래2)” 규칙의 **부족 슬롯 보충** 규칙
- 색상은 **최대 11개 고정 팔레트**(운영자가 직접 선정)로 매핑

## 비범위(이번 설계에서 명시적으로 하지 않는 것)

- 통계 탭 UI/쿼리 상세(별도 문서/스펙으로 분리)
- 서버/DB에 순위를 저장하는 방식(필요 시 추후 확장)

## 데이터 정의(현 구현과의 정합)

### 누적 km (마일리지)

- 일자 `d`에 대한 멤버 `m`의 누적 km는 월 내 로그를 일자 오름차순으로 합산한 값이다.
- `d`에 기록이 없으면 **직전 기록일의 누적값으로 carry-forward** 한다. (현 컴포넌트 로직과 동일)

### 달성률 % (달성률)

- 멤버의 목표 km를 `goalKm`으로 둔다.
  - 우선순위: `evt_mlg_goal_cfg.goal_val` (해당 월) → 없으면 `init_goal`
- `goalKm <= 0` 인 경우 달성률은 **정의 불가**로 취급한다. (아래 “달성률 모드 랭킹 후보” 참고)

## 기준일(`d_ref`) — “스냅샷”이 아니라 “지금 이 화면에서 판단하는 날”

요구사항: 매일 기록이 들어가므로 **19일엔 A가 1등이었다가 20일엔 B가 1등이 될 수 있다**. 즉, 별도의 스냅샷 테이블 없이 **데이터가 변하면 순위도 변한다**.

### `d_ref`를 무엇으로 둘지(구현 전 합의)

아래 중 하나를 택한다. (둘 다 타당하지만 UX가 다름)

#### 옵션 1 (권장, 단순): “오늘(KST)까지 반영된 월 누적”

- 현재 보고 있는 `month`가 ‘이번 달’이면 `d_ref = min(오늘 일자, totalDays)`
- 과거 달을 보면 `d_ref = totalDays` (그 달의 마지막 상태로 고정)

장점: 사용자가 기대하는 “지금 순위”와 가장 잘 맞는다.

단점: X축 전체 월을 보여주더라도, **선택된 11명은 `d_ref` 기준**이라는 설명이 필요하다.

#### 옵션 2: “차트에서 사용자가 선택한 일”

- `d_ref`를 UI에서 고르게 한다(예: 슬라이더/날짜 선택).

장점: “그날의 순위/주변”을 정확히 볼 수 있다.

단점: UI 복잡도가 늘고, 모바일에서 조작성 이슈가 생길 수 있다.

### 성능(부하) 관점

이 계산은 보통 **O(멤버 수)** 정도로 충분히 가볍다. (정렬 + 주변 슬라이스)

다만 다음이 커지면 체감이 생길 수 있다:

- 멤버 수가 매우 크고(수백~수천)
- 페이지에서 같은 계산을 **초당 여러 번** 반복(불필요한 리렌더)

권장 최적화(코드 레벨):

- `d_ref`와 입력 데이터가 같으면 `useMemo`로 **선별 결과를 메모이즈**
- “랭킹 탭 캐시”는 **네트워크/DB 비용**을 줄이는 도구이고, 이 11명 선별은 대부분 **클라이언트 CPU** 문제라서 캐시가 정답은 아닐 수 있다.
- 서버에서 월별 집계 테이블/머티리얼라이즈드 뷰를 두는 건 “전체 통계/랭킹” 쪽에 더 잘 맞는다.

## 데이터 로딩 전략(통계 탭과의 관계)

요구사항: **전체 참가자 기록은 통계 탭에서 본다**. 따라서 프로젝트 탭의 차트는 “전체를 항상 같이 무겁게 로드”할 필요가 없다.

### 선택지 A — 한 번에 가져와서(현재 구조와 유사) 클라이언트에서 11명만 고른다

- 서버/클라이언트가 이미 월 로그를 넓게 들고 있다면, **11명 선별은 메모리에서** 하면 된다(추가 DB 왕복 없음).
- 장점: 구현 단순, 통계 탭이 같은 데이터를 재사용하기 쉬움
- 단점: 프로젝트 탭 초기 로딩이 느린 경우(체감상) **“차트만 가볍게” 목표와 충돌**할 수 있음

### 선택지 B — 차트는 가볍게, 통계 탭은 무겁게(권장 방향)

- 프로젝트 탭(차트): “차트 렌더에 필요한 최소 데이터”만
  - 예: **선택된 11명**의 시계열(누적/달성률) + (선택) 나의 reference line용 목표
- 통계 탭: 전체 인원/전체 로그/추가 지표
  - 탭 진입 시 로드(지연 로딩)하거나, 별도 라우트에서 로드

### 선택지 C — DB에서 11명만 뽑아 내려준다

- 장점: 네트워크 payload 감소 가능
- 단점: “월 전체 라인 차트”를 그리려면 결국 **11명의 일자별 시계열**이 필요하므로,
  - (1) 11명 id만 받고 클라이언트가 또 조회하거나
  - (2) 서버/RPC가 시계열까지 한 번에 만들어 내려줘야 한다
- 즉 “11명만 뽑는 쿼리” 자체가 마법처럼 가볍게 만들지는 못하고, **집계 형태를 어떻게 내려줄지**가 핵심이다.

### 캐시를 어디에 두는 게 맞나

- `unstable_cache` + `revalidateTag`는 **DB/RPC 비용**을 줄이는 데 특히 효과가 있다. (레코드 페이지의 패턴과 유사)
- 반면 11명 선별 자체는 대개 **CPU 비용**이라 캐시 이득이 작다.
- 따라서 느림이 “프로젝트 탭 전체”라면 우선순위는 보통:
  1) 페이지에서 동시에 도는 쿼리 수/중복 조회 줄이기(병렬화/공용 fetch/집계 RPC)
  2) 차트/통계의 데이터 분리(지연 로딩)
  3) 월별 rollup 캐시(태그 기반 무효화 포함)

## 0km 제외 규칙

### 정의

- **차트에 표시되는 멤버 후보**에서 `d_ref` 기준 누적 km가 **0인 멤버는 제외**한다.
  - 의미: “아직 아무 기록도 없는 사람”은 비교군에서 빼고 혼잡도를 줄인다.

### 달성률 모드에서의 해석

- 사용자 관점에서 “0km면 달성률도 0%”가 대부분이므로, **동일하게 제외**해도 제품 의도와 잘 맞는다.
- 단, 구현상으로는 `goalKm <= 0`이면 달성률이 **0으로 찍히거나 의미가 없는 값**이 될 수 있으니(현 구현도 그렇게 될 수 있음) **후보에서 제외**하는 처리는 별도로 명시한다.

## 랭킹(모드별로 독립)

### 공통 tie-break

1. 비교 값(아래 모드별 값)을 소수 **첫째 자리**까지 반올림해 비교한다.
2. 동률이면 **`mem_nm` 오름차순(가나다/ABC)** 으로 정렬한다. (요구사항)
3. 그래도 동일하면(사실상 동명이인 등) **`mem_id` 문자열 오름차순**으로 최종 tie-break 한다.

### 마일리지 모드 랭킹

- 랭킹 키: `rankKey = round1(cumKm@d_ref)`
- 정렬: `rankKey` 내림차순 → tie-break

### 달성률 모드 랭킹

- 후보: `goalKm > 0` 인 멤버만
- 랭킹 키: `rankKey = round1(percent@d_ref)` (이미 컴포넌트는 100 cap)
- 정렬: `rankKey` 내림차순 → tie-break

## 11명 선별 알고리즘(핵심)

입력:

- `membersRanked[]`: 위 랭킹 규칙으로 정렬된 배열(길이 N)
- `me`: 로그인 사용자의 `mem_id` (없을 수 있음)

출력:

- 기본 목표는 **11명**의 `mem_id` 집합 `selected` 이다.
- 다만 후보 인원 `N`이 11보다 작으면 **가능한 최대 `N`명**만 선택한다.

중요: “상위3 + 하위3 + 나 + 주변4”는 **서로 중복될 수 있다**. 중복이 생기면 합이 11보다 작아지므로, **부족분만큼 추가 인원을 채워 11명을 맞춘다**.

### 케이스 A: `me`가 없음(비로그인/식별 불가)

- 기본: **상위 11명**
- (선택) UI에 “로그인하면 내 기준 뷰” 안내

### 케이스 B: `me`가 랭킹 후보에 없음

- 예: 0km 제외로 필터되어 빠짐, goal 없음으로 달성률 후보에서 빠짐
- 기본: **상위 11명**
- (선택) `me`가 필터 밖이면 별도 안내 배너(“기록이 없어 비교 뷰에 포함되지 않았어요”)

### 케이스 C: `me`가 후보에 있음

목표 구성:

- **상위 3명**
- **하위 3명**
- **나(1명)**
- **내 주변 4명**: 랭킹에서 나의 위치를 기준으로 **위 2 + 아래 2**
- (중복 제거 후) **11명이 되도록 추가 멤버를 채움**

#### 내 주변 4명 — 슬롯 부족 보충 규칙(요구사항 반영)

- 기본: `above = 2`, `below = 2`
- `below`가 부족하면 부족분만큼 `above`를 늘린다. (예: 아래 사람이 없으면 위 4)
- 대칭적으로 `above`가 부족하면 `below`를 늘린다.

구체 절차(의사코드):

```
wantAbove = 2
wantBelow = 2

while (availableAbove < wantAbove || availableBelow < wantBelow):
  if availableBelow < wantBelow:
    wantAbove += (wantBelow - availableBelow)
    wantBelow = availableBelow
  else if availableAbove < wantAbove:
    wantBelow += (wantAbove - availableAbove)
    wantAbove = availableAbove
  else:
    break
```

- `availableAbove/availableBelow`는 **본인을 제외한** 인접 인원 수다.

#### 중복 제거 후 11명 미만이면 “추가로 가까운 사람”을 채운다

아래는 사용자가 제시한 예시를 일반화한 규칙이다.

- 먼저 다음 집합을 만든다(중복 제거):
  - `Top = 상위 3`
  - `Bottom = 하위 3`
  - `Me = {나}`
  - `Near = 주변(위2/아래2 + 부족 보충 규칙)`

- `U = union(Top, Bottom, Me, Near)`
- `|U| < 11` 이면 `need = 11 - |U|` 만큼 추가한다.

추가 후보(기본):

- **랭킹에서 나와 가장 가까운 사람들**을 우선한다.
  - “가깝다”는 기본적으로 **랭크 거리(|rank(me) - rank(x)|)** 가 작은 순**이다.

추가 시 제외 규칙(예시를 코드 정책으로 고정):

- 사용자 예시: “내가 꼴지면 하위3에 내가 포함되니, 하위3을 제외한 채로 아래 방향(더 꼴지 쪽)으로 갈 사람이 없으니 **아래쪽에서(= 더 꼴지 방향) 5명**을 채운다”
- 일반화:
  - `Bottom`에 포함된 인원은 **추가 후보에서 제외**할 수 있다(사용자 의도: 하위3은 ‘꼴지 프레임’이므로 Near 확장과 섞이지 않게).
  - 단, `Me ∈ Bottom`이면 `Me`는 예외로 항상 포함한다.

> 이 “Bottom 제외” 규칙은 제품 UX를 크게 바꾸므로 구현 시 상수로 박고, QA 체크리스트에 넣는다.

## 차트 렌더링 규칙

### 라인 수

- 최대 11개 `Line`만 렌더링한다.

### 색상

- 최대 11색 팔레트를 운영자가 지정한다.
- 매핑은 **역할 기반**이 가장 설명 가능하다(권장):
  - 상위 3: 고정 3색
  - 하위 3: 고정 3색
  - 나: 강조 1색
  - 주변 4: 고정 4색(또는 2색 교차 등)
- “멤버 고정색”을 유지하려는 요구가 없다면(이번 설계 방향), **역할 기반 매핑이 UX상 유리**하다.

## 전체 통계 탭 표 스펙(확정)

정렬 기준:

- 기본 정렬은 `누적거리` 내림차순
- 동률은 기존 공통 규칙과 동일하게 소수 첫째 자리 기준 후 이름순 정렬

컬럼 순서(확정):

1. `순위`
2. `이름`
3. `목표거리`
4. `누적거리`
5. `목표` (텍스트 목표: 예. 하프완주, 풀코스완주, 서브3, 서브4) — **현재 보류**
6. `달성률`
7. `추천거리(일)`

계산/표시 규칙:

- 달성률은 기존 로직을 그대로 사용한다. (`누적거리 / 당월목표`)
- 추천거리(일)은 기존 “일일 필요” 계산식을 그대로 사용한다.
- 달성률/추천거리 계산은 중복 구현하지 않고 **공용 함수**로 통일한다.
  - 예: `computeAchievementPercent(...)`, `computeDailyRequiredDistance(...)`
- 공용 함수는 차트/표/툴팁에서 동일하게 재사용해 값 불일치를 방지한다.
- `목표거리`는 가입 시 필수 입력값이므로 **비어있지 않다**는 전제로 처리한다.
- `목표`(텍스트)는 **현재 DB 컬럼이 없어 표에서 제외(보류)** 한다.
  - 추후 DB 컬럼 추가 시, 표에 `목표` 컬럼만 확장해서 노출 가능하도록 설계한다.

현재 구현 기준 표시 컬럼(보류 반영):

- `순위 / 이름 / 목표거리 / 누적거리 / 달성률 / 추천거리(일)`

## 테스트 체크리스트(수동)

- N=5 이하에서도 크래시 없이 선택되는가
- 내가 1등/꼴등일 때 주변 슬롯 보충이 기대대로인가
- 동률(소수 첫째 자리까지 동일)에서 이름순이 고정되는가
- 마일리지/달성률 모드 전환 시 랭킹이 기대대로 바뀌는가(`d_ref` 라벨 포함)
- `goalKm=0` 멤버가 달성률 모드에서 **후보 제외/표시 0%** 중 어떤 동작이 되는가(구현 전 합의)

## 구현 메모(파일 단위)

- 클라이언트: `components/projects/crew-progress-chart.tsx`
  - 멤버 필터링/랭킹/선별을 함수로 분리해 테스트 가능하게 한다.
- 서버 래퍼: `components/projects/crew-progress-chart-server.tsx`
  - 이 파일은 “프리패치(prefetch)”라고 부르기 쉬운데, 정확히는 **서버에서 월 데이터를 미리 조회해 `initialData`로 내려주는 역할**이다.
  - 11명 뷰를 도입하면 “서버가 내려주는 `members`가 항상 전체”일 수 있다. 이는 문제는 아니지만, **선별 로직을 클라이언트에만 두면** 서버/클라이언트가 미묘하게 어긋날 여지가 있다(예: 추후 서버에서도 요약을 만들 때).
  - 권장: 선별/랭킹/ tie-break 규칙을 `lib/...` 유틸로 분리해 **서버/클라이언트가 동일 함수를 import** 하게 한다.

## 리팩토링 원칙(사용자 체감 성능 중심)

목표:

- “정확한 실시간 반영(기록/순위)”을 유지하면서, **페이지가 멈춘 듯 보이는 구간을 제거**한다.
- 정적 캐시에 의존하지 않고도, 초기 체감 속도를 높인다.

핵심 철학:

1. **정확성 우선**: 순위/11명 선별은 현재 데이터 기준으로 계산(개인화 유지)
2. **블로킹 최소화**: 느린 네트워크 응답이 페이지 전체 렌더를 막지 않게 분리
3. **점진적 렌더링**: 계산/데이터는 늦어도 되지만, 화면 골격은 즉시 노출
4. **단일 계산식**: 달성률/일일 필요(추천거리) 계산은 공용 함수로 통일

### 아키텍처 가이드

- 프로젝트 페이지는 “한 번에 다 준비된 결과”를 기다리지 않는다.
- 기본 탭(마일리지)은 우선 노출하고, 달성률/전체 통계는 **자동 백그라운드 로딩**한다.
  - 사용자가 버튼을 눌러야 계산/조회가 시작되는 구조는 피한다.
- Suspense 경계를 세분화해, 섹션 단위로 독립 로딩한다.
  - 헤더/월 네비/탭 껍데기
  - 마일리지 본문
  - 달성률/전체 통계 데이터

### 네트워크 병목 대응 원칙

- 병목이 계산이 아니라 네트워크면, “연산 축소”보다 “대기 구조 개선”을 우선한다.
- 즉, 전체 데이터를 조금 덜 가져오는 것만으로 기대만큼 빨라지지 않을 수 있다.
- 먼저 적용할 우선순위:
  1) 페이지 블로킹 제거(전체 대기 금지)
  2) 데이터 영역 분리 로딩(부분 스켈레톤)
  3) 중복/불필요 요청 제거
  4) 이후 필요 시 집계/RPC/캐시 최적화

### UI/UX 가이드(로딩 상태)

- “큰 스켈레톤 하나” 대신 **영역별 스켈레톤** 사용
- 탭/레이아웃은 즉시 표시하고, 데이터 영역만 지연 표시
- 기본 탭은 우선 완성, 나머지 탭은 준비되면 즉시 전환 가능 상태로
- 로딩 중에도 화면이 멈춘 것처럼 보이지 않게 작은 상태 표시(예: 동기화 중)

### 구현 작업 순서(본 문서 기반 착수 순서)

1. 공용 유틸 정리
   - 순위/선별/달성률/추천거리 계산식을 `lib` 유틸로 분리
2. 데이터 경계 분리
   - 서버 컴포넌트는 초기 렌더에 필요한 최소 구조 전달
   - 클라이언트는 마운트 직후 비기본 탭 데이터를 자동 프리로드
3. 렌더 경계 개선
   - Suspense 경계/스켈레톤을 섹션 단위로 재배치
4. 회귀 검증
   - 기록 등록 직후 순위/차트 반영, 탭 전환 체감, 동률 규칙, 11명 선별 규칙 확인
