## 1. 개요 — 문제와 전략

**문제**: 신규 인원은 간간히 들어오지만 유령회원 확률이 매우 높다 = 첫 모임 참석까지의 심리적 진입장벽이 세다.

**전략**: "가입은 쉽게, 의도는 명확히"

| 장치 | 노리는 효과 |
|------|------------|
| 가입 목적 칩 + 자유입력 | 뉴비가 뭘 원하는지 파악 → 운영 대응 가능 |
| 유입 경로 칩 (단일선택) | 홍보 채널 효과 측정, 경로별 유령률 분석 |
| 가까운 역 (선택) | 접근성 파악, 향후 코스·벙 안내에 활용 |
| 평균 거리·페이스 (선택) | "나 너무 느린데" 심리장벽 해소 — 입력값 기반 개인화 환영 멘트 |
| **참석 약속 (필수 클릭)** | implementation intention — 추상적 서약이 아니라 실제 열린 모임을 보여주고 바로 신청까지 연결 |
| 가입 D+14 미참석 넛지 푸시 | 약속 리마인드, 유령 전환 전 마지막 개입 |

**원칙**:

- 신규 필수 입력은 **유입 경로 1개 + 목적 칩 1개 이상 + 참석 약속 클릭** 뿐. 역·거리·페이스는 전부 선택.
- 토스식 다단계 위저드 — 한 화면에 한 주제, 상단 진행바가 빠르게 차오르는 느낌.
- 참석을 강요하지 않는다 — 참가 없이 넘어가는 버튼도 긍정문("일정 확인하고 꼭 참석할게요")으로.

## 2. 가입 위저드 재구성

### 2.1 진행바 확장 (3 → 6단계)

`SignupProgress`의 `TOTAL_STEPS`를 6으로 확장. 세그먼트가 얇아지고 스텝당 입력이 적어
"빠르게 진행되는" 체감을 만든다.

| 단계 | 라벨 | 화면 | 필수 여부 |
|------|------|------|----------|
| 1 | 시작 | newbie 안내 (기존) | — |
| 2 | 로그인 | OAuth 로그인 (기존) | — |
| 3 | 기본 정보 | 연락처 확인 → 이름·성별·생년월일 (OAuth에 이메일 없을 때만 이메일 선택 입력) | 필수 |
| 4 | 러닝 프로필 | 가까운 역 / 평균 거리 / 평균 페이스 | **전부 선택** (건너뛰기 가능) |
| 5 | 가입 목적 | 유입 경로 칩(단일선택) + 목적 칩 멀티선택 + 자유입력 | 유입 경로 1개 + 목적 칩 1개 이상 필수 |
| 6 | 참석 약속 | 안내문 + "네, 참석할게요!" 버튼 → 모임 선택 화면으로 전환 | **버튼 클릭 필수** |

- 기존 `stage` state("phone"/"details"/…)를 확장하는 형태로 `MemberOnboardingForm` 내부에서 처리 (라우트 분리 없음).
- **연락처 확인 분기는 3단계에서 그대로**: 기존 회원 연동(link)·재가입(pending)이면 4~6단계를 타지 않고 기존 플로우대로 종료. 신규(new)만 위저드 계속.
- **계좌 입력은 온보딩에서 제거** (기존 접이식 선택정보 폐지). 가입 시점엔 계좌가 필요한 맥락이 없어 거부감만 주므로, 완료 화면의 "계좌 정보 지금 등록하기" 링크(기존)와 `/profile/bank`로 충분. 회비 안내 시점에 자연스럽게 받는다.

### 2.2 회원 생성 시점 — 마지막에 한 번 (결정)

현행은 3단계 제출 시 `onboardingCreateMember`가 즉시 회원을 생성한다. 개편 후에는
**4~6단계 입력을 클라이언트에 보관했다가, 6단계 참석 약속 완료 시점에 한 번에 생성**한다.

- 근거: "참석 약속 필수"의 의미가 성립하려면 약속 전에 회원이 만들어지면 안 됨. 중간 이탈 시 미가입 상태로 남아 재시도 가능(고아 프로필 없음).
- `onboardingCreateMember` 확장: 기존 파라미터 + `onbdProfile`(역·거리·페이스·목적·서약시각) + `pledgeGthrId`(팝업에서 참가 선택한 모임, 없으면 null).
- 서버 액션 처리 순서: `mem_mst` INSERT → `mem_onbd_prf` INSERT → (`pledgeGthrId` 있으면) `gthr_attd_rel` INSERT.
  - 참석 INSERT 시 기강포인트 트리거가 자동 적립됨 — 정상 동작이며 별도 처리 불필요.
  - 참석 INSERT는 **기존 모임 참석 신청 로직 재사용** — 6단계 도달 시점엔 페이지 로드 때 조회한 모임 데이터가 오래됐을 수 있으므로(그 사이 정원 마감·삭제), 신청 시점에 정원을 재검증하는 기존 경로를 그대로 탄다.
  - 참석 INSERT 실패(정원 마감 등)는 가입 자체를 롤백하지 않는다 — 가입 성공 + 완료 화면에서 "모임 신청은 실패했어요, 일정에서 다시 신청해 주세요" 안내.
  - `mem_onbd_prf` INSERT 실패도 **비치명 처리** — 서버 액션은 문장별 실행이라 트랜잭션으로 묶이지 않는데, 프로필 저장 실패 때문에 이미 생성된 회원을 되돌리지 않는다(가입 성공 처리 + 에러 로깅). 부작용은 해당 회원이 넛지 대상에서 빠지는 정도로 수용 가능.
- 위저드 뒤로가기: 4~6단계는 이전 단계로 돌아갈 수 있어야 한다(입력값 보존). 3단계 안에서도 "번호 다시 입력"(기존 기능)으로 연락처 재입력 가능 — 잘못 입력한 번호를 고칠 수 있어야 한다. 돌아갈 수 없는 곳은 1~2단계(시작·로그인)뿐 — OAuth가 이미 완료된 상태라 되돌아갈 이유가 없다.

## 3. 입력 항목 스펙

### 3.1 가입 목적 (5단계, 칩 1개 이상 필수)

- UI: 칩(토글 버튼) 멀티선택 + 하단 자유입력 textarea(선택, "더 하고 싶은 말이 있다면").
- 칩 목록은 공통코드(`cmm_cd_grp_mst`/`cmm_cd_mst`) 그룹 **`JOIN_PURP`** 로 관리 — 추가/수정에 배포 불필요.

| 코드 | 라벨 |
|------|------|
| `RUN_MATE` | 같이 달릴 사람이 필요해요 |
| `COACH` | 자세·훈련 코칭을 받고 싶어요 |
| `TRAINING` | 인터벌 같은 훈련을 같이 하고 싶어요 |
| `NEW_SPORT` | 안 해본 운동을 해보고 싶어요 |
| `RACE` | 대회에 같이 나가고 싶어요 |
| `FRIENDS` | 새로운 친구를 만나고 싶어요 |
| `HABIT` | 운동 습관을 만들고 싶어요 |

- 저장: `mem_onbd_prf.join_purp_cds text[]` + `join_purp_txt text null`.

### 3.2 가까운 역 (4단계, 선택)

- 대상: 서울·인천·경기 지하철역 전체 (약 800역). **외부 API 없이 정적 JSON 번들** — `lib/data/subway-stations.json`.
  - 공공데이터포털 전국 도시철도 역 목록에서 수도권만 1회 추출. 형식: `{ name: "강남", lines: ["2호선", "신분당선"] }[]` — 환승역은 역명 기준 1건으로 병합.
  - 데이터 갱신은 사실상 불필요(신규 역 개통 시에만 수동 갱신).
- UI: 기존 `components/ui/command.tsx` + `popover.tsx` 조합 콤보박스. 초성/부분 일치 검색.
- 역 데이터 JSON은 **4단계 진입 시 dynamic import** — 초기 번들에 포함하지 않는다.
- 저장: `near_stn_nm varchar null` (역명만, 호선은 표시용).

### 3.3 평균 러닝 거리 (4단계, 선택)

- UI: 숫자 입력(km 단위, `inputMode="decimal"`), placeholder "예: 5".
- 저장: `avg_run_dist_km numeric(4,1) null`. 범위 검증 0.5~100.

### 3.4 평균 페이스 (4단계, 선택)

- UI: 드롭다운. 3'30"~7'30" 30초 단위 9개 + 경계 옵션 2개.

| 값(`pace_cd`) | 라벨 |
|---------------|------|
| `P330` ~ `P730` | 3'30" / 4'00" / … / 7'30" (30초 단위 9개) |
| `P730_OVER` | 7'30"보다 여유롭게 |
| `UNKNOWN` | 잘 모르겠어요 |

- 저장: `avg_pace_cd varchar null` + CHECK 제약. 미입력 = null.
- 초 단위 숫자 대신 코드로 저장하는 이유: "여유롭게"/"모르겠어요"에 가짜 숫자를 부여하지 않기 위함. 집계는 코드 GROUP BY로 충분.

### 3.5 유입 경로 (5단계, 단일선택 필수)

- UI: 가입 목적 화면 상단에 배치 — "기강은 어떻게 알게 되셨어요?" 단일선택 칩. `ETC` 선택 시 자유입력 노출.
- 칩 목록은 공통코드 그룹 **`JOIN_SRC`** 로 관리.

| 코드 | 라벨 |
|------|------|
| `FRIEND` | 지인 소개 |
| `INSTA` | 인스타그램 |
| `SOMOIM` | 소모임 |
| `DAANGN` | 당근 |
| `ETC` | 기타 (직접 입력) |

- 저장: `join_src_cd varchar` + `join_src_txt text null` (기타 상세). 온보딩에서는 앱 레벨 필수, 컬럼은 nullable(프로필 편집으로 row를 만드는 기존 회원은 유입 경로가 없음).
- 용도: 홍보 채널 효과 측정 + 유입 경로별 유령률 상관 분석 (§9). 일회성 사실이라 프로필 편집 대상은 아님.

### 3.6 참석 약속 (6단계, 필수)

- 화면 구성: 안내문 + 큰 버튼 하나.
  > **가입 후 한 달 안에 모임에 한번 나와주세요** 🙏
  > 정기런(격주 수요일)을 추천해요. 처음엔 다들 어색하지만, 한 번 나오면 달라져요.
  >
  > [ 네, 참석할게요! ]
- 버튼 클릭 → 같은 단계 안에서 모임 선택 화면(§4)으로 전환. 화면에서 참가 or "일정 확인하고…"를 선택하면 가입 최종 제출.
- 저장: `attd_pldg_at` (버튼 클릭 시각 — 온보딩 경로는 항상 기록), `pldg_gthr_id` (모임 선택 화면에서 참가 선택한 모임, 없으면 null).

## 4. 참석 약속 — 모임 선택 화면

> **구현 변경(2026-07-09)**: 아래 설계는 휠 피커(§4.2)를 전제했으나, 최종 구현은 **모임을 세로 목록으로 나열하고 각 행에 "참석" 버튼**을 두는 방식으로 바뀌었다(행에서 바로 참가 신청, 하단에 "일정을 확인하고 나중에 꼭 참석할게요" 링크). 휠 피커·`components/common/wheel-picker.tsx`는 폐기. 6단계 "이전" 버튼도 제거(마지막 단계라 앞으로만 진행). §4.2는 원설계 기록으로 남긴다.

### 4.1 동작

- 컨테이너: 별도 팝업이 아니라 **6단계 안에서 안내 화면 → 모임 선택 화면으로 전환** (위저드 화면 전환과 동일한 UX). 휠 피커가 화면을 넓게 쓸 수 있고 진행바 흐름도 유지된다.
- 데이터: 열려있는 모임 = `gthr_mst`에서 `stt_at > now()` AND `del_yn = false` AND 정원 미달(`max_prt_cnt` null 또는 참석수 < 정원). 온보딩 페이지(서버)에서 조회해 props로 전달 — 미가입 상태 RLS 이슈를 피하려면 기존 public RPC(`get_public_team_gatherings`) 계열 재사용.
- 분기:

| 열린 모임 수 | 표시 (실제 구현) |
|--------------|------|
| 1개 이상 | **세로 목록**: 각 행에 모임명·날짜·시간·참석 인원 + 행별 "참석" 버튼(즉시 신청). 하단에 "일정을 확인하고 나중에 꼭 참석할게요" 링크 |
| 0개 | 안내문: "다음 정기런이 열리면 가장 먼저 알려드릴게요 🏃 정기런은 **격주 수요일 저녁**에 열려요." + "확인했어요, 꼭 참석할게요" |

> 원설계는 6개 이상=휠 피커 / 1~5개=라디오 목록으로 분기했으나, 구현에서 개수 무관 세로 목록(행별 참석)으로 통일(§4 상단 노트).

- 항목 표시 정보(간단하게): 모임명 · 날짜(`M/D(ddd)`) · 장소.
- CTA:
  - 주 버튼 **"이 모임 참가하기"** — 선택(휠 가운데/라디오) 모임의 `gthr_id`를 `pldgGthrId`로 확정.
  - 하단 회색 텍스트 버튼 **"일정 확인하고 꼭 참석할게요"** — 참가 없이 진행. (거절도 참석 의사로 프레이밍 — "고민"이라는 단어 금지)
- 어느 경로든 선택 즉시 §2.2의 최종 제출 실행 → 완료 화면.

### 4.2 휠 피커 UI 스펙

iOS 시간 선택기 스타일의 세로 휠:

```text
   (흐림·작게)   4. 7/9(수) 정기런
   ┌─────────────────────────────┐
   │ 5. 7/12(토) 한강 벙          │
   │ 6. 7/16(수) 정기런           │
   │ ▶ 7. 7/19(토) 트랙 훈련 ◀   │  ← 가운데(선택됨) 가장 크고 진하게
   │ 1. 7/23(수) 정기런           │
   │ 2. 7/26(토) 남산 벙          │
   └─────────────────────────────┘
   (흐림·작게)   3. 7/30(수) 정기런
```

- 가운데 5개 정도가 크고 진하게, 그 위아래로 흐린 항목이 하나씩 보이며 **무한 루프**로 순환.

## 6. DB 스키마

### 6.1 `mem_onbd_prf` — 온보딩 프로필 (신규)

멤버 1:1 위성 테이블 (`mem_utmb_prf` 패턴). `mem_mst`에 컬럼을 늘리지 않는다.

```sql
CREATE TABLE mem_onbd_prf (
  mem_id           UUID PRIMARY KEY REFERENCES mem_mst(mem_id),
  near_stn_nm      VARCHAR,                 -- 가까운 역 (선택)
  avg_run_dist_km  NUMERIC(4,1),            -- 평균 러닝 거리 km (선택)
  avg_pace_cd      VARCHAR                  -- P330~P730 / P730_OVER / UNKNOWN (선택)
                   CHECK (avg_pace_cd IS NULL OR avg_pace_cd ~ '^(P[3-7](00|30)|P730_OVER|UNKNOWN)$'),
  join_purp_cds    TEXT[] NOT NULL DEFAULT '{}',  -- 가입 목적 칩 (온보딩은 앱 레벨에서 1개 이상 강제)
  join_purp_txt    TEXT,                    -- 목적 자유입력 (선택)
  join_src_cd      VARCHAR,                 -- 유입 경로 칩 (온보딩은 앱 레벨에서 필수, 기존 회원 upsert는 null 허용)
  join_src_txt     TEXT,                    -- 유입 경로 기타 상세 (선택)
  attd_pldg_at     TIMESTAMPTZ,             -- 참석 약속 클릭 시각 (온보딩 경로에서만 기록)
  pldg_gthr_id     UUID REFERENCES gthr_mst(gthr_id),  -- 약속 시 참가 신청한 모임
  crt_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  upd_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- RLS: 본인 INSERT/SELECT/UPDATE, 관리자 SELECT. (`attd_pldg_at`·`pldg_gthr_id`·`join_src_cd`는 프로필 편집 폼에서 다루지 않음 — 온보딩 서버 액션만 기록)
- 공통코드: `JOIN_PURP` 그룹 7종 (§3.1) + `JOIN_SRC` 그룹 5종 (§3.5) 시드.
- 기존 회원도 프로필 편집(§6.3)에서 입력하면 row가 생긴다(upsert). 따라서 **"개편 후 신규 가입자" 식별자는 row 존재가 아니라 `attd_pldg_at IS NOT NULL`** — 서약은 온보딩에서만 발생하므로 넛지 대상 선별이 안전하다.

### 6.2 마이그레이션

| 파일 | 내용 |
|------|------|
| `2026070XXXXXXX_mem_onbd_prf.sql` | 테이블 + RLS + `JOIN_PURP`·`JOIN_SRC` 공통코드 시드 |

### 6.3 프로필 편집 개방 (1차 범위 포함)

> **2026-07-10 축소 결정**: 온보딩 답변(평균 거리·페이스·가입 목적)은 가입 시점 스냅샷이라 사후 수정 가치가 없다고 판단, 편집 대상에서 제외. 회원 프로필 성격인 **가까운 역만** 수정 개방. 별도 "러닝 프로필" 섹션·저장 버튼도 제거하고 메인 프로필 폼에 통합(저장 시 수정된 섹션만 각자 액션 호출).

| 항목 | 수정 위치 |
|------|----------|
| 이름·성별·생년월일·이메일 | `/profile/edit` (기존 그대로) |
| 계좌 | `/profile/bank` (기존 그대로) |
| 가까운 역 | `/profile/edit` 메인 폼에 통합 (`updateNearStation` 액션) |

- 저장은 `mem_onbd_prf` upsert(near_stn_nm만) — 개편 전 가입한 기존 회원도 여기서 처음 입력 가능(소급 입력이 자연 해결됨).
- 평균 거리·페이스·가입 목적·`attd_pldg_at`·`pldg_gthr_id`는 편집 대상 아님 (온보딩 스냅샷/이력이므로 폼에 노출하지 않음). 테이블·온보딩 위저드는 그대로 유지.

## 7. 미참석 넛지 푸시

### 7.1 발송 조건

매일 1회(KST 오전 9시) 크론이 판정:

| 넛지 | 대상 | 메시지(안) |
|------|------|-----------|
| D+14 | `mem_onbd_prf.attd_pldg_at IS NOT NULL`(개편 후 온보딩 가입자) AND 가입(`crt_at`) 14일 경과 AND `gthr_attd_rel` 0건 AND D+14 넛지 미발송 | "가입하고 2주가 지났어요! 약속하신 첫 모임, 이번 정기런은 어때요? 🏃" |
| D+28 | 위와 같고 28일 경과 AND D+28 넛지 미발송 | "한 달 약속이 코앞이에요. 이번 주가 딱 좋은 타이밍! 부담 없이 나와요 🙌" |

- 공통 전제: **활성 회원만** — `mem_mst.del_yn = false` AND 팀 소속 상태 active(탈퇴·비활성 제외).
- 중복 방지: `noti_mst`에 해당 타입 발송 이력 존재 여부로 판정 (별도 상태 컬럼 불필요).
- 발송: `insertNoti()` 호출 → 인앱 알림 + 웹푸시 자동 발송 (기존 인프라 그대로). 알림 클릭 랜딩은 홈(일정 섹션).
- 알림 타입 추가: `newbie_nudge_14` / `newbie_nudge_28` (`lib/notifications/insert-noti.ts` 타입 체계에 추가 — 타입이 DB enum/CHECK로 강제되는지 구현 시 확인, 그렇다면 마이그레이션에 포함).
- 멘트 개선(선택): 발송 시점에 다가오는 정기런이 있으면 날짜를 박아 발송 — "약속하신 첫 모임, **이번 주 수요일(7/22) 정기런** 어때요?" / 없으면 위 기본 멘트 폴백.

## 9. 성과 지표

핵심 지표: **가입 후 30일 내 첫 모임 참석률** (월 코호트).

```sql
-- 코호트: 가입 월별, 30일 내 첫 참석 여부
SELECT date_trunc('month', m.crt_at AT TIME ZONE 'Asia/Seoul') AS cohort,
       count(*) AS joined,
       count(*) FILTER (WHERE a.first_attd IS NOT NULL
                        AND a.first_attd < m.crt_at + interval '30 days') AS attended_30d
FROM mem_mst m
LEFT JOIN (SELECT mem_id, min(crt_at) AS first_attd FROM gthr_attd_rel GROUP BY mem_id) a USING (mem_id)
WHERE m.del_yn = false
GROUP BY 1 ORDER BY 1;
```

- 개편 전 코호트가 자연 대조군. 개편 전/후 구분은 `attd_pldg_at IS NOT NULL` (§6.1과 동일 기준).
- 보조 지표: 모임 선택 화면에서 즉시 참가 신청 비율(`pldg_gthr_id NOT NULL` 비율), 넛지 발송 후 7일 내 참석 전환율, 유입 경로(`join_src_cd`)별 30일 참석률.
