## 1. 개요

기강 크루의 월 회비 납부·관리 시스템입니다.
카카오뱅크 계좌와 서버를 직접 연동할 수 없으므로, 모임장이 CSV를 주기적으로 업로드하면 서버가 이를 파싱하여 회원별 납부 내역을 관리합니다.

## 2. 회비 정책

### 기본 정책

| 항목 | 내용 |
|------|------|
| 월 회비 | 2,000원 (변경 시 `dues_config` 테이블에 이력 관리) |
| 납부 수단 | 카카오뱅크 계좌 입금 (3333-09-6788223) |
| 입금자명 | 본인 실명 (띄어쓰기 없이) |
| 회비 환불 | 불가 |
| 예치금 환불 | 가능 (관리자 처리) |
| 내역 공개 | 비공개 (관리자만 전체 열람 가능) |
| 납부 마감일 | 매월 말일 (strict deadline 없음. 선불 납부 권장. 미납 확정은 익월 자동 마감 시 처리) |

### 면제 정책

아래 상황에서 모임장이 개별 회원에 대해 특정 월 회비를 면제할 수 있습니다.

- 타지역 이사로 활동 불가
- 장기 부상·휴식
- 기타 모임장 판단

면제 시 **면제 사유, 면제 기간, 승인한 관리자 ID**를 반드시 기록합니다. (`member_due_records.exemption_reason`, `exempted_by` 컬럼)

### 신규 가입자 회비 시작 시점

- 가입 당월은 무조건 회비 면제 (`exempted` 상태, 자동 면제)
- 가입 다음 달부터 회비 부과 시작
- 기준 시점은 온보딩 완료(`members.status = 'active'` 전환) 시점

**예시**: 3월 15일 가입 완료 → 3월 면제, 4월부터 부과

### 탈퇴 시 예치금 처리

탈퇴 시 예치금 처리 의무 없음. 시스템이 강제로 안내하거나 환불 절차를 요구하지 않습니다.

- 회원이 **명시적으로 환불을 요청한 경우에만** 관리자가 `refund-deposit` 액션으로 처리
- 조용히 탈퇴하는 경우 예치금은 모임 잔액으로 남음
- 탈퇴 후에도 `member_deposits` 레코드는 삭제하지 않고 유지 (이력 보존)

### 예치금 시스템

회비 초과 납부 시 초과금은 예치금으로 적립되며, 이후 월 회비에 자동 차감됩니다.

**예시**: 1월에 12,000원 납부

| 월 | 처리 내용 | 예치금 잔액 |
|----|-----------|------------|
| 1월 | 2,000원 → 회비 / 10,000원 → 예치금 적립 | 10,000원 |
| 2월 | 예치금에서 2,000원 차감 | 8,000원 |
| 3월 | 예치금에서 2,000원 차감 | 6,000원 |
| ... | ... | ... |
| 6월 | 예치금에서 2,000원 차감 | 0원 |
| 7월 | 예치금 소진 → 미납 발생 | 0원 |

회원 프로필에서 예치금 차감 시 "예치금 자동 차감 (차감액: 2,000원, 잔액: N원)" 형식으로 표시합니다.

### 선불 납부 원칙

회비는 **선불** 개념입니다. 이번 달 초에 이번 달 회비를 납부합니다.

- N월 회비 = N월 중 납부 (기준: `member_due_records.year_month = 'YYYY-MM'`)
- 선불 납부를 권장하나 strict deadline 없음
- 월 마감(매월 1일 자동 실행) 시점에 전월 미납자 확정

**예시**: 3월 회비 → 3월 1~31일 중 납부

## 3. 데이터 수집 방식 (CSV 업로드)

### 흐름

```
카카오뱅크 앱 → 거래 내역 CSV 다운로드 (권장: 매월 초 전월 내역 다운로드)
→ 모임장(관리자) 업로드
→ 서버 파싱 및 회원 자동 매칭
→ 관리자 검토 및 수동 보정
→ 회원 회비 상태 반영
→ 월 마감 실행
```

### CSV 업로드 권장 주기

매월 초(1~3일) 전월 거래 내역을 업로드합니다. 관리자 대시보드에 마지막 업로드 일시를 표시하여 업로드 누락을 방지합니다.

### 파일 포맷 (카카오뱅크 거래내역)

카카오뱅크 앱에서 다운로드되는 파일은 **xlsx(Excel) 형식**입니다. 비밀번호로 암호화되어 있습니다.

**실물 확인 완료 (2026-03-26)**

| 컬럼 | 예시 | 비고 |
|------|------|------|
| 거래일시 | `2026.02.26 11:15:31` | 날짜+시간 통합 컬럼 (`YYYY.MM.DD HH:MM:SS`) |
| 구분 | `입금` / `출금` | |
| 거래금액 | `5,000` / `-17,200` | 쉼표 포함 문자열. 출금은 음수 |
| 거래 후 잔액 | `2,510,578` | 쉼표 포함 문자열. 파싱 시 무시 가능 |
| 거래구분 | `일반입금`, `일반입금(AI)`, `일반이체`, `예금이자`, `타행자동이체`, `간편이체`, `간편이체(이름)` | |
| 내용 | `이현근`, `동마 1차`, `싱글렛 대금 결제` | 입금자명 또는 적요. 회원 매칭에 사용 |
| 메모 | `호정 플루스 2월 3등` | 이체 시 추가 메모. 비어있는 경우 많음 |

**헤더 위치**: 11번째 행 (1~10행은 계좌 정보 및 안내 문구)

**파일 암호화**: 업로드 시 비밀번호를 입력하거나, 모임장이 미리 복호화하여 업로드하는 방식 중 선택 필요. (→ 9. 미결정 사항 참조)

**파싱 시 처리 사항**:
- `거래금액`의 쉼표 제거 후 정수 변환
- `거래일시`를 `transaction_date`(date)와 `transaction_time`(time)으로 분리
- `구분 = 출금`이면 `category = 'expense'` 후보 (관리자 확인)
- `거래구분 = '예금이자'` 또는 `'타행자동이체'`(소모임 구독료 등): `category = 'other'` 자동 분류

### 회비 명목 판단 기준

CSV 한 건이 회비 납부인지, 지출인지, 기타 입금인지 구분이 필요합니다.

| 조건 | 분류 | 비고 |
|------|------|------|
| 거래유형 = 입금 + 금액이 2,000원 단위 + 이름 매칭 가능 | 회비 or 예치금 | 자동 분류 후 관리자 확인 |
| 거래유형 = 출금 | 지출 | 회식비, 행사비 등 |
| 거래유형 = 입금 + 이름 매칭 불가 | 미확인 | 관리자 수동 처리 |

> 최종 카테고리 지정은 관리자가 수동 확인·보정합니다.

### 중복 업로드 방지

- 파일 내용의 SHA-256 해시(`content_hash`)를 `csv_uploads`에 저장
- 동일 해시 파일 재업로드 시 서버에서 오류 반환, 프론트에서 "이미 업로드된 파일입니다" 안내
- 잘못 업로드된 CSV는 `delete-csv-upload` 액션으로 관련 `dues_transactions` cascade delete 후 롤백 가능 (단, 이미 `member_due_records`에 반영된 경우 관리자 수동 확인 필요)

## 4. 회원 매칭 로직

### 자동 매칭

1. CSV 내 `내용` 필드에서 이름 추출
2. 현재 `active` 상태 회원 목록과 비교 (이름 기준, 완전 일치)
3. **정확히 1명 매칭** → `matched` 상태로 자동 연결
4. **0명 또는 2명 이상 매칭** → `unmatched` 또는 `ambiguous` 상태로 관리자 대기열에 올림

이름 정규화: 공백 제거 후 비교. 포함 검색은 사용하지 않고 완전 일치만 적용합니다.

### 동명이인 처리 (`ambiguous`)

자동 매칭하지 않고 `ambiguous` 상태로 남겨 관리자가 직접 지정합니다.
관리자 미매칭 처리 화면에서 후보 회원 목록(이름 + 전화번호 뒷자리 힌트)을 표시하여 선택할 수 있도록 합니다.

### 미매칭 거래 처리 (`unmatched`)

- 회비 관련이 아닌 것으로 판단되는 경우: `other` 카테고리로 분류, 관리자만 열람 가능
- 회비일 가능성이 있는 경우: 관리자가 전체 회원 검색 후 수동으로 회원 UUID 지정

> `ambiguous`(동명이인 후보 선택)와 `unmatched`(전체 검색 후 지정)는 화면에서 별도 UI로 처리합니다.

## 5. DB 스키마

> **백엔드 개발자 참고**
> - 모든 테이블은 Supabase SQL Editor 또는 마이그레이션 파일로 생성합니다.
> - `members` 테이블은 기존 테이블입니다. `id uuid`, `name text`, `admin boolean`, `status text` 컬럼이 있습니다.
> - ENUM 타입은 아래 순서대로 먼저 생성해야 합니다.
> - RLS는 테이블 생성 후 별도 섹션(8번)의 SQL을 실행합니다.

### ENUM 타입 (먼저 실행)

```sql
CREATE TYPE dues_transaction_category AS ENUM ('due', 'expense', 'other');
CREATE TYPE dues_match_status         AS ENUM ('matched', 'unmatched', 'ambiguous');
CREATE TYPE dues_upload_status        AS ENUM ('pending', 'confirmed', 'rolled_back');
CREATE TYPE dues_record_status        AS ENUM ('paid', 'unpaid', 'exempted', 'covered_by_deposit');
CREATE TYPE deposit_transaction_type  AS ENUM ('charge', 'deduction', 'refund');
```

### `dues_config` — 회비 금액 이력 (신규)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| effective_from | date | 해당 금액 적용 시작 월 (YYYY-MM-01) |
| amount | int | 월 회비 금액 (원) |
| created_by | uuid FK → members | 변경한 관리자 |
| created_at | timestamptz | |

> `close-month` 실행 시 해당 월 기준 `dues_config`에서 금액을 조회하여 하드코딩을 피합니다.

```sql
CREATE TABLE dues_config (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  effective_from date NOT NULL,          -- YYYY-MM-01 형식
  amount         int  NOT NULL CHECK (amount > 0),
  created_by     uuid NOT NULL REFERENCES members(id),
  created_at     timestamptz NOT NULL DEFAULT now()
);

-- 초기 데이터 (월 2,000원, 2026-01-01부터)
INSERT INTO dues_config (effective_from, amount, created_by)
VALUES ('2026-01-01', 2000, '<모임장 uuid>');
```

> 금액 조회 쿼리: `SELECT amount FROM dues_config WHERE effective_from <= '<YYYY-MM-01>' ORDER BY effective_from DESC LIMIT 1`

### `dues_expense_records` — 회비 지출 기록 (신규)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| transaction_id | uuid FK → dues_transactions (nullable) | CSV에서 파싱된 지출 거래 |
| expense_date | date | 지출일 |
| amount | int | 지출 금액 (양수) |
| title | text | 지출 항목명 (예: 2월 뒤풀이) |
| description | text (nullable) | 상세 내용 |
| created_by | uuid FK → members | 등록한 관리자 |
| created_at | timestamptz | |

```sql
CREATE TABLE dues_expense_records (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  transaction_id uuid REFERENCES dues_transactions(id) ON DELETE SET NULL,
  expense_date   date NOT NULL,
  amount         int  NOT NULL CHECK (amount > 0),
  title          text NOT NULL,
  description    text,
  created_by     uuid NOT NULL REFERENCES members(id),
  created_at     timestamptz NOT NULL DEFAULT now()
);
```

### `dues_month_close_log` — 월 마감 이력 (신규)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| year_month | text PK | `YYYY-MM` 형식, 월당 1건 |
| closed_at | timestamptz | 마감 실행 시각 |
| closed_by | uuid FK → members (nullable) | 마감 실행 관리자. null이면 Cron 자동 실행 |

> 이 테이블로 `close-month` 중복 실행을 방지합니다. 이미 레코드가 있는 월은 재마감 불가.
> 마감 후에는 해당 월 데이터 수정이 잠기며, 수정이 필요하면 관리자가 마감을 해제(레코드 삭제) 후 재처리합니다.

```sql
CREATE TABLE dues_month_close_log (
  year_month text PRIMARY KEY CHECK (year_month ~ '^\d{4}-\d{2}$'),
  closed_at  timestamptz NOT NULL DEFAULT now(),
  closed_by  uuid REFERENCES members(id)   -- null = Cron 자동 실행
);
```

### `xlsx_uploads` — xlsx 업로드 이력

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| uploaded_by | uuid FK → members | 업로드한 관리자 |
| file_name | text | 원본 파일명 |
| content_hash | text UNIQUE | SHA-256 해시 (중복 방지) |
| row_count | int | 파싱된 행 수 |
| status | dues_upload_status | `pending` / `confirmed` / `rolled_back` |
| uploaded_at | timestamptz | 업로드 시각 |

```sql
CREATE TABLE xlsx_uploads (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  uploaded_by  uuid NOT NULL REFERENCES members(id),
  file_name    text NOT NULL,
  content_hash text NOT NULL UNIQUE,
  row_count    int  NOT NULL DEFAULT 0,
  status       dues_upload_status NOT NULL DEFAULT 'pending',
  uploaded_at  timestamptz NOT NULL DEFAULT now()
);
```

### `dues_transactions` — xlsx 원시 거래 내역

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| xlsx_upload_id | uuid FK → xlsx_uploads | 출처 업로드 |
| transaction_date | date | 거래일 |
| transaction_time | time | 거래시간 |
| amount | int | 금액 (항상 양수. 입금/출금은 category로 구분) |
| raw_name | text | xlsx 내 이름/메모 원문 (`내용` 컬럼값) |
| raw_memo | text (nullable) | xlsx `메모` 컬럼값 |
| transaction_type | text | xlsx `거래구분` 컬럼값 (예: `일반입금`, `일반이체`) |
| category | dues_transaction_category | `due` / `expense` / `other` |
| matched_member_id | uuid FK → members (nullable) | 매칭된 회원 |
| match_status | dues_match_status | `matched` / `unmatched` / `ambiguous` |
| parse_error | text (nullable) | 파싱 오류 내용 (오류 행 기록용) |
| created_at | timestamptz | |

**category 초기값 정책**

> 기본적으로 **모든 입금은 `due`(회비)** 로 처리합니다.
> 관리자가 명시적으로 `expense` 또는 `other`로 변경한 경우에만 회비가 아닌 항목으로 처리됩니다.

| 조건 | 초기 category | 비고 |
|------|--------------|------|
| `구분 = 입금` | `due` | 기본값. 관리자가 나중에 변경 가능 |
| `구분 = 출금` | `expense` | 지출로 자동 분류 |
| `거래구분 = '예금이자'` | `other` | 이자 수입 |
| `거래구분 = '타행자동이체'` | `other` | 소모임 구독료 등 고정 출금 |

```sql
CREATE TABLE dues_transactions (
```

### `member_due_records` — 회원별 월 회비 납부 기록

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| member_id | uuid FK → members | |
| year_month | text | `YYYY-MM` 형식, CHECK(`year_month ~ '^\d{4}-\d{2}$'`) |
| status | enum | `paid` / `unpaid` / `exempted` / `covered_by_deposit` |
| transaction_id | uuid FK → dues_transactions (nullable) | 연결된 거래 |
| exemption_reason | text (nullable) | 면제 사유 (exempted 상태일 때 필수) |
| exempted_by | uuid FK → members (nullable) | 면제 승인한 관리자 |
| created_at | timestamptz | |
| updated_at | timestamptz | |

**UNIQUE 제약**: `(member_id, year_month)`

**status 값 설명**

| 값 | 의미 |
|----|------|
| `paid` | 직접 납부 |
| `unpaid` | 미납 |
| `exempted` | 모임장 면제 (가입 당월 자동 포함) |
| `covered_by_deposit` | 예치금으로 차감 처리 |

```sql
CREATE TABLE member_due_records (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id        uuid NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  year_month       text NOT NULL CHECK (year_month ~ '^\d{4}-\d{2}$'),
  status           dues_record_status NOT NULL DEFAULT 'unpaid',
  transaction_id   uuid REFERENCES dues_transactions(id) ON DELETE SET NULL,
  exemption_reason text,   -- exempted 상태일 때 사유 기록
  exempted_by      uuid REFERENCES members(id),
  created_at       timestamptz NOT NULL DEFAULT now(),
  updated_at       timestamptz NOT NULL DEFAULT now(),
  UNIQUE (member_id, year_month)
);

CREATE INDEX idx_due_records_member  ON member_due_records(member_id, year_month);
CREATE INDEX idx_due_records_month   ON member_due_records(year_month, status);  -- close-month 핵심 쿼리

-- updated_at 자동 갱신 트리거
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN NEW.updated_at = now(); RETURN NEW; END;
$$;

CREATE TRIGGER trg_due_records_updated_at
BEFORE UPDATE ON member_due_records
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

### `member_deposits` — 회원별 예치금 잔액

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid PK | |
| member_id | uuid FK → members UNIQUE | |
| balance | int | 현재 예치금 잔액 (원, 0 이상) |
| updated_at | timestamptz | |

> 신규 회원 온보딩 완료(`members.status = 'active'`) 시 `balance = 0`으로 자동 생성합니다.
> 트리거 또는 서버 액션에서 INSERT합니다.

```sql
CREATE TABLE member_deposits (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  member_id  uuid NOT NULL UNIQUE REFERENCES members(id) ON DELETE CASCADE,
  balance    int  NOT NULL DEFAULT 0 CHECK (balance >= 0),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- 잔액 갱신 시 updated_at 자동 처리
CREATE TRIGGER trg_member_deposits_updated_at
BEFORE UPDATE ON member_deposits
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

**예치금 차감 시 동시성 처리**: `balance` 갱신은 반드시 Supabase RPC(PostgreSQL Function) 안에서 `SELECT ... FOR UPDATE`를 사용해 race condition을 방지합니다.

```sql
-- 예시 RPC (close-month, refund-deposit에서 호출)
CREATE OR REPLACE FUNCTION deduct_deposit(
  p_member_id uuid,
  p_amount    int
) RETURNS void LANGUAGE plpgsql AS $$
BEGIN
  UPDATE member_deposits
  SET balance = balance - p_amount
  WHERE member_id = p_member_id AND balance >= p_amount
  FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION '예치금 잔액 부족';
  END IF;
END;
$$;
```

### 자동 매칭 로직 (upload-xlsx 내부)

```
xlsx 행 순회 (헤더 11행 이후부터)
  → 파싱 오류 시 parse_error 기록, 해당 행 skip (전체 롤백하지 않음)
  → category 초기값 결정 (위 표 기준)
  → category = 'due'인 행만 회원 매칭 시도
      → raw_name 공백 제거 후 active 회원 이름 완전 일치 검색
      → 매칭 1명: matched_member_id 설정, match_status = 'matched'
      → 매칭 0명: match_status = 'unmatched'
      → 매칭 2명 이상: match_status = 'ambiguous'
  → category = 'expense' / 'other': match_status = 'unmatched' (매칭 불필요)
```

> 파싱 중 오류가 발생한 행은 `parse_error`에 기록하고 건너뜁니다. 업로드 결과 요약에 "파싱 오류 N건"을 표시합니다.

### 월별 회비 마감 로직 (close-month)

마감은 **매월 1일 자동 실행** (전월 마감)합니다. 관리자가 수동으로도 실행 가능합니다.

**자동 실행 방식**: Vercel Cron Job (`vercel.json` 설정) → `POST /api/dues/close-month` API 라우트 호출

```
dues_month_close_log에 해당 year_month 존재 확인
  → 존재하면: 오류 반환 "이미 마감된 월입니다"
  → 없으면:
      dues_config에서 해당 월 회비 금액 조회
      해당 월 미납 회원 조회 (member_due_records.status = 'unpaid' or 레코드 없음)
        → 예치금 잔액 >= 회비 금액인 회원:
            SELECT FOR UPDATE (RPC) → balance -= 회비금액
            status = 'covered_by_deposit'
            deposit_transactions 차감 기록 (member_due_record_id 연결)
        → 예치금 없는 회원: status = 'unpaid' 확정
      dues_month_close_log INSERT (closed_by = null이면 시스템 자동, closed_at)
```

**Vercel Cron 설정 예시** (`vercel.json`):
```json
{
  "crons": [
    {
      "path": "/api/dues/close-month",
      "schedule": "0 1 1 * *"
    }
  ]
}
```
매월 1일 01:00 UTC(한국시간 10:00) 실행. API 라우트는 `CRON_SECRET` 환경변수로 인증합니다.

### 관리자 판별 (RLS)

`members` 테이블의 `admin boolean` 컬럼 기준:

```sql
CREATE POLICY "관리자 전체 조회"
ON dues_transactions FOR SELECT
USING (
  EXISTS (
    SELECT 1 FROM members
    WHERE members.id = auth.uid()
    AND members.admin = true
  )
);
```

## 8. RLS (Row Level Security) 방침

관리자 판별: `members.admin = true` 조건 기준.

| 테이블 | 본인 조회 | 관리자 전체 조회 | 관리자 수정 |
|--------|----------|----------------|------------|
| dues_transactions | ✗ | ✅ | ✅ |
| member_due_records | ✅ (본인 것만) | ✅ | ✅ |
| member_deposits | ✅ (본인 것만) | ✅ | ✅ |
| deposit_transactions | ✅ (본인 것만, display_reason만 표시) | ✅ (전체 컬럼) | ✅ |
| xlsx_uploads | ✗ | ✅ | ✅ |
| dues_config | ✗ | ✅ | ✅ |
| dues_expense_records | ✗ | ✅ | ✅ |
| dues_month_close_log | ✗ | ✅ | ✅ |

> 일반 회원이 자신의 납부일(`dues_transactions.transaction_date`)을 확인해야 할 때는 Supabase RPC(PostgreSQL Function)를 통해 필요한 컬럼만 반환합니다. 일반 회원에게 직접 `dues_transactions` 조회 권한을 부여하지 않습니다.

### RLS SQL (백엔드 개발자용)

아래 SQL을 테이블 생성 후 실행합니다. 모든 테이블에 RLS를 활성화한 뒤 정책을 추가합니다.

```sql
-- 관리자 판별 헬퍼 함수 (반복 사용)
CREATE OR REPLACE FUNCTION is_admin()
RETURNS boolean LANGUAGE sql SECURITY DEFINER AS $$
  SELECT EXISTS (
    SELECT 1 FROM members WHERE id = auth.uid() AND admin = true
  );
$$;

ALTER TABLE dues_transactions ENABLE ROW LEVEL SECURITY;
```

### 회원용 RPC (납부 내역 조회)

일반 회원이 프로필에서 납부 내역을 조회할 때 사용합니다. `dues_transactions`에 직접 접근하지 않고 이 함수를 통해 필요한 정보만 반환합니다.

```sql
-- 회원의 납부 거래 정보 (날짜, 금액만 반환)
CREATE OR REPLACE FUNCTION get_my_due_transaction(p_transaction_id uuid)
RETURNS TABLE (transaction_date date, amount int)
LANGUAGE sql SECURITY DEFINER AS $$
  SELECT transaction_date, amount
  FROM dues_transactions
  WHERE id = p_transaction_id;
$$;

-- 회원의 예치금 변동 내역 (display_reason만 노출)
CREATE OR REPLACE FUNCTION get_my_deposit_history(p_limit int DEFAULT 20)
RETURNS TABLE (amount int, type deposit_transaction_type, year_month text, display_reason text, created_at timestamptz)
LANGUAGE sql SECURITY DEFINER AS $$
  SELECT amount, type, year_month, display_reason, created_at
  FROM deposit_transactions
  WHERE member_id = auth.uid()
  ORDER BY created_at DESC
  LIMIT p_limit;
$$;
```
