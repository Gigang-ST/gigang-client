## 1. 개요

기강 크루의 월 회비 납부·관리 시스템입니다.
카카오뱅크 계좌와 서버를 직접 연동할 수 없으므로, 관리자가 거래내역 xlsx 파일을 비정기적으로 업로드하면 서버가 이를 파싱하여 회원별 납부 내역을 관리합니다.

- 회비 관리 기능은 **팀 admin 이상(`team_role_cd IN ('owner', 'admin')`)** 전용입니다.
- 예치금 환불은 회원이 카카오톡으로 직접 요청 시 관리자가 처리하는 오프라인 절차이며, 시스템에는 별도 UI가 없습니다.

## 2. 회비 정책

### 기본 정책

| 항목 | 내용 |
|------|------|
| 월 회비 | 2,000원 (변경 시 `fee_policy_cfg` 테이블에 이력 관리) |
| 납부 수단 | 카카오뱅크 계좌 입금 (3333-09-6788223) |
| 입금자명 | 본인 실명 (띄어쓰기 없이) |
| 회비 환불 | 불가 |
| 예치금 환불 | 가능 (회원이 카카오톡으로 요청 시 관리자가 수동 처리. 시스템 UI 없음) |
| 내역 공개 | 비공개 (admin 이상만 전체 열람 가능) |
| 납부 마감일 | 매월 말일 (strict deadline 없음. 선불 납부 권장) |

### 면제 정책

아래 상황에서 관리자가 개별 회원에 대해 특정 기간 회비를 면제할 수 있습니다.

- 타지역 이사로 활동 불가
- 장기 부상·휴식
- 기타 관리자 판단

면제는 `fee_due_exm_cfg` 테이블에 **면제 유형(`full`/`part`), 적용 기간, 사유, 등록자**를 기록합니다.
정산 실행 시 `fee_due_exm_hist`에 실제 반영 이력을 남깁니다.

### 신규 가입자 회비 시작 시점

- 가입 당월부터 회비 부과 시작
- 기준 시점은 `team_mem_rel.join_dt` (크루 가입 시각)

**예시**: 3월 15일 가입 완료 → 3월부터 부과

### 탈퇴 시 예치금 처리

탈퇴 시 예치금 처리 의무 없음.

- 회원이 명시적으로 환불을 요청한 경우에만 관리자가 오프라인으로 처리
- 조용히 탈퇴하는 경우 예치금은 모임 잔액으로 남음
- `fee_mem_bal_snap` 레코드는 삭제하지 않고 유지 (이력 보존)

### 잔액(예치금) 개념

`fee_mem_bal_snap.bal_amt`로 표현되는 회원별 누적 잔액.

```
bal_amt = 확정 납부 누적 + 면제 반영 − 가입일/정책 기준 부과 누적
```

- `bal_amt > 0`: 예치금 보유 (초과 납부)
- `bal_amt = 0`: 정확히 납부
- `bal_amt < 0`: 미납 발생

**예시**: 1월에 12,000원 납부

| 월 | 처리 내용 | 잔액 |
|----|-----------|------|
| 1월 | 2,000원 → 회비 / 10,000원 → 예치금 적립 | +10,000원 |
| 2~6월 | 매월 예치금 2,000원 차감 | +8,000 → 0원 |
| 7월 | 예치금 소진 → 미납 | -2,000원 |

## 3. 데이터 수집 방식 (xlsx 업로드)

### 흐름

```
카카오뱅크 앱 → 거래 내역 xlsx 다운로드
→ 관리자 업로드
→ 서버: 복호화(KAKAOBANK_XLSX_PASSWORD) → 파싱 → fee_txn_hist 저장 → 자동 매칭
→ 관리자 검토 및 수동 확정 (is_cfm_yn = true)
→ 확정 건만 fee_due_pay_hist에 반영
→ "재계산" 실행 → fee_mem_bal_snap 갱신
```

### 업로드 권장 주기

매월 초(1~3일) 전월 거래 내역을 업로드합니다. 관리자 대시보드에 마지막 업로드 일시를 표시하여 업로드 누락을 방지합니다.

### 파일 포맷 (카카오뱅크 거래내역 xlsx)

**실물 확인 완료 (2026-03-26)**

| 컬럼 | 예시 | 비고 |
|------|------|------|
| 거래일시 | `2026.02.26 11:15:31` | `YYYY.MM.DD HH:MM:SS` |
| 구분 | `입금` / `출금` | → `fee_txn_io_enm` |
| 거래금액 | `5,000` / `-17,200` | 쉼표 포함 문자열. 출금은 음수. 파싱 시 절대값만 사용 |
| 거래 후 잔액 | `2,510,578` | 파싱 시 무시 |
| 거래구분 | `일반입금`, `일반입금(AI)`, `일반이체`, `예금이자`, `타행자동이체`, `간편이체`, `간편이체(이름)` | → `txn_tp_txt` |
| 내용 | `이현근`, `동마 1차`, `싱글렛 대금 결제` | → `raw_name`. 회원 매칭에 사용 |
| 메모 | `호정 플루스 2월 3등` | → `raw_memo`. 비어있는 경우 많음 |

**헤더 위치**: 11번째 행 (1~10행은 계좌 정보 및 안내 문구)

**파일 암호화**: 서버 환경변수 `KAKAOBANK_XLSX_PASSWORD`로 자동 복호화. 프론트는 파일만 업로드.

**파싱 시 처리 사항**:
- `거래금액`의 쉼표 제거 후 절대값 정수 변환 → `txn_amt`
- `거래일시`를 `txn_dt`(date)와 `txn_tm`(time)으로 분리
- `구분 = 입금` → `txn_io_enm = 'deposit'`
- `구분 = 출금` → `txn_io_enm = 'withdrawal'`

### fee_item_cd 초기값 정책

| 조건 | 초기 `fee_item_cd` | 비고 |
|------|-------------------|------|
| `txn_io_enm = 'deposit'` | `due` | 기본값. 관리자가 확정 전 변경 가능 |
| `txn_io_enm = 'withdrawal'` | `expense` | 지출로 자동 분류 |
| `txn_tp_txt = '예금이자'` | `other` | 이자 수입 |
| `txn_tp_txt = '타행자동이체'` | `other` | 소모임 구독료 등 고정 출금 |

### 중복 업로드 처리

`fee_txn_hist`의 `uk_fee_txn_hist_dedup` UNIQUE 제약으로 처리:

```sql
UNIQUE (team_id, txn_dt, txn_tm, txn_amt, raw_name)
```

- 동일 파일 재업로드: `fee_xlsx_upd_hist.file_hash` (SHA-256) 체크로 거부
- 기간 겹침: 동일 거래는 `ON CONFLICT DO NOTHING`으로 skip. "중복 건너뜀 N건" 표시

## 4. 회원 매칭 로직

### 자동 매칭

1. `raw_name` 공백 제거 후 `mem_mst.mem_nm`과 완전 일치 비교 (포함 검색 사용 안 함)
2. 정확히 1명 매칭 → `match_st_cd = 'matched'`, `mem_id` 설정
3. 0명 매칭 → `match_st_cd = 'unmatched'`
4. 2명 이상 매칭 → `match_st_cd = 'ambiguous'`

매칭 시도는 `fee_item_cd = 'due'`인 입금 거래에만 수행합니다.

### 동명이인 처리 (`ambiguous`)

자동 매칭하지 않고 관리자가 직접 지정합니다.
관리자 미매칭 처리 화면에서 후보 회원 목록(이름 + 전화번호 뒷자리 힌트)을 표시합니다.

### 관리자 확정 (`is_cfm_yn`)

매칭 이후에도 **관리자가 `is_cfm_yn = true`로 확정**해야 `fee_due_pay_hist`에 반영됩니다.
자동으로 확정되지 않으며, 동명이인/미매칭 건은 절대 자동 확정하지 않습니다.

- `vers=0` = 현재 잔액 (정본)
- `vers>0` = 과거 정산 이력 (숫자 클수록 오래된 이력)
- 새 정산 실행 시: 기존 `vers=0` → `max(vers)+1`로 UPDATE 후 새 행 `vers=0`으로 INSERT

**정산 규칙**:
- **스냅샷(`vers=0`) 있는 경우**: `last_calc_dt` 이후 신규 납부·면제·부과만 증분 계산 → 기존 `vers=0`을 `max(vers)+1`로 밀고 새 `vers=0` INSERT
- **스냅샷 없는 경우 (신규 회원 첫 정산)**: `team_mem_rel.join_dt`부터 현재까지 `fee_policy_cfg` 기준 전체 계산 → `vers=0`으로 첫 INSERT
- 스냅샷 기준일 **이전** 데이터가 새로 들어오면 오류 반환 (역행 방지)
- 과거 정정 필요 시 스냅샷 초기화 후 전 구간 재계산


RLS도 동일 기준(`team_role_cd IN ('owner', 'admin')`)으로 설정되어 있습니다.

### 서버 액션 / API 목록

| 기능 | 경로 | 설명 |
|------|------|------|
| xlsx 업로드 및 파싱 | `actions/dues/upload-xlsx.ts` | 파일 수신 → `KAKAOBANK_XLSX_PASSWORD` 복호화 → SHA-256 중복 확인 → 파싱 → `fee_txn_hist` 저장 → 자동 매칭 |
| xlsx 롤백 | `actions/dues/rollback-xlsx.ts` | `fee_xlsx_upd_hist.upd_st_cd = 'rolled_back'`. `is_cfm_yn = false`인 연결 `fee_txn_hist` soft-delete. 이미 확정된 건은 롤백 불가 (관리자 안내) |
| 회원 수동 매칭 | `actions/dues/match-transaction.ts` | `match_st_cd`, `mem_id` 업데이트. 이미 `matched`인 건 덮어쓰기 가능 (확인 다이얼로그) |
| 카테고리 변경 | `actions/dues/update-fee-item.ts` | `fee_item_cd` 수정 (`is_cfm_yn = false` 인 건만) |
| 거래 확정 | `actions/dues/confirm-transaction.ts` | `is_cfm_yn = true`, `cfm_by_mem_id`, `cfm_at` 설정 → `fee_due_pay_hist` 삽입 |
| 면제 규칙 등록 | `actions/dues/create-exemption.ts` | `fee_due_exm_cfg` 삽입. 기간 중복 서비스 레이어 검증 |
| 잔액 재계산 | `actions/dues/recalculate-balance.ts` | `fee_due_pay_hist` + `fee_due_exm_hist` 기반 증분 또는 전체 재계산 → `fee_mem_bal_snap` 갱신 |
| 회비 정책 설정 | `actions/dues/update-policy.ts` | `fee_policy_cfg` 신규 기간 추가. 기간 중복 검증 |

### 자동 매칭 로직 (upload-xlsx 내부)

```
xlsx 행 순회 (헤더 11행 이후부터)
  → 파싱 오류 시 adm_memo_txt에 오류 기록, skip (전체 롤백하지 않음)
  → fee_item_cd 초기값 결정 (2절 표 기준)
  → fee_item_cd = 'due'인 입금 행만 회원 매칭 시도
      → raw_name 공백 제거 후 mem_mst.mem_nm 완전 일치 (vers=0, del_yn=false, mem_st_cd='active')
      → 매칭 1명: mem_id 설정, match_st_cd = 'matched'
      → 매칭 0명: match_st_cd = 'unmatched'
      → 매칭 2명 이상: match_st_cd = 'ambiguous'
  → fee_item_cd ≠ 'due': match_st_cd = 'unmatched' (매칭 불필요)
  → ON CONFLICT (uk_fee_txn_hist_dedup) DO NOTHING
업로드 결과 요약: 총 N건, 매칭 N건, 미매칭 N건, 동명이인 N건, 중복 skip N건, 오류 N건
```

### 거래 확정 → 납부 원장 반영 흐름

```
관리자가 fee_txn_hist 행 확인
  → fee_item_cd 최종 확인·수정
  → match_st_cd = 'matched' 확인
  → confirm-transaction 액션 호출
      → fee_txn_hist.is_cfm_yn = true
      → fee_due_pay_hist INSERT
          (team_id, mem_id, src_txn_id=txn_id, pay_amt=txn_amt, pay_dt=txn_dt, pay_st_cd='paid')
  → (선택) recalculate-balance 호출 → fee_mem_bal_snap 갱신
```

### 회원 프로필 — 회비 섹션 (진입점)

## 8. RLS (Row Level Security)

권한 기준: `team_mem_rel.team_role_cd IN ('owner', 'admin')` (admin 이상).

마이그레이션 `20260404093853_v2_wave5_fee_snap_rls.sql` 기준.

| 테이블 | SELECT (팀 소속 회원) | ALL (admin 이상) |
|--------|----------------------|-----------------|
| `fee_policy_cfg` | ✅ | ✅ |
| `fee_xlsx_upd_hist` | ✅ | ✅ |
| `fee_txn_hist` | ✅ | ✅ |
| `fee_due_pay_hist` | ✅ | ✅ |
| `fee_due_exm_cfg` | ✅ | ✅ |
| `fee_due_exm_hist` | ✅ | ✅ |
| `fee_mem_bal_snap` | ✅ | ✅ |

> **참고**: 현재 RLS는 팀 소속 회원 전체가 SELECT 가능하도록 설정되어 있습니다 (`team_mem_rel` 조인).
> 회원 본인 잔액만 프로필에 표시하는 것은 **쿼리 레벨에서 `mem_id = auth.uid()` 조건**으로 처리합니다.
> 타인의 상세 납부 내역은 쿼리에서 본인 것만 조회하도록 서버 액션에서 제한합니다.
