## 1. 개요

모임 출석을 독려하기 위해, 한 달 동안의 출석·개설 활동에 따라 **당월 회비를 자동 감면**하는 제도.
"퀘스트"처럼 진행도를 보여주고, 조건을 달성하면 면제 내역에 자동으로 반영한다.

- 월 회비: **2,000원** (`fee_policy_cfg.monthly_fee_amt`)
- **1단계 감면 = 회비의 50%(1,000원)** / **2단계 = 회비 전액(2,000원) 면제**
- 회원이 "내가 얼마나 참석해서 얼마나 감면받았는지"를 한눈에 볼 수 있어야 한다.

### 핵심 구조 한눈에

```text
[원천]  gthr_mst · gthr_attd_rel
   │
   ▼  (집계 RPC — 칭호·랭킹과 공유할 경계)
get_member_monthly_activity(team, mem, ym) → { 정기참석, 개설, 총참석 }
   │
   ├──▶ [규칙] calcExemption(stats, 회비단가) → 면제액   ← 코드(순수함수)+상수, DB 아님
   │
   ├──▶ [배치] 월 마감 후 전월 면제 INSERT (rflt_yn=false)  ← 확정·잔액반영용
   │
   └──▶ [퀘스트 카드] 당월 실시간 진행도 표시              ← 회원 안내용
                                          │
[재계산] rflt_yn=false 면제 전부 합산 → 잔액 반영 + rflt_yn=true 마킹
```

## 2. 감면 규칙 (확정)

### 2.1 게이트 조건 (둘 중 하나 충족 필수)

| 게이트 | 판정 |
|--------|------|
| 정모(정기) 1회 참석 | 해당 월 `gthr_type_enm = 'regular'` 모임에 **참석**(`gthr_attd_rel` row 존재) ≥ 1 |
| 벙 1회 개설 | 해당 월 본인이 **개설**(`gthr_mst.crt_by = 본인`)한 모임 ≥ 1 (타입 무관) |

> **게이트를 통과하지 못하면 참석 횟수가 아무리 많아도 감면 없음.**
> (예: 일반 벙 5번 참석 + 정모 0 + 개설 0 → 감면 0원)

### 2.2 참석 횟수 (게이트 통과를 전제로 적용)

| 누적 참석 | 감면액 |
|-----------|--------|
| 4회 이상 | 회비의 50% (현재 1,000원) |
| 8회 이상 | 회비 전액 (현재 2,000원) |

- **참석 횟수 = 해당 월 `gthr_attd_rel`에 본인 row가 있는, `stt_at`이 그 달에 속한 모든 모임 수** (타입 무관).
- 본인이 개설한 모임은 개설 시 자동으로 참석 처리되므로 **참석 1회에 자연히 포함**된다.
  → "벙 개설하고 7번 더 참석" = 개설한 모임(1) + 추가 참석(7) = **총 8회 → 전액 면제**.

### 2.3 예시

| 활동 | 게이트 | 참석 횟수 | 감면 |
|------|--------|-----------|------|
| 정모 참석 + 일반 벙 3회 참석 | ✅ 정모 | 4회 | 1,000원 |
| 벙 개설(=참석1) + 추가 참석 7회 | ✅ 개설 | 8회 | 2,000원(전액) |
| 일반 벙 5회 참석 (정모X, 개설X) | ❌ | 5회 | 0원 |
| 정모 참석 + 추가 참석 9회 | ✅ 정모 | 10회 | 2,000원(전액) |

> **감면은 단계형(누적 아님).** 8회 도달 시 2,000원이지 1,000+2,000=3,000이 아니다.
> 면제 금액은 월 회비를 초과할 수 없다(전액이 상한).

### 2.4 월 귀속 기준

- 모임의 **`stt_at`(KST)이 속한 월**로 귀속. (참석 토글한 시점이 아님)
- 미래 모임에 미리 참석을 눌러도 그 모임의 `stt_at` 월로 집계된다.

## 3. 규칙 정의 — 코드(순수 함수 + 상수)로 두고 "호출"한다

> **확정**: 규칙은 DB 컬럼·관리 UI로 빼지 않는다. **코드 모듈 한 곳**에 두고
> 배치·퀘스트 카드가 **같은 함수를 호출**한다. (규칙은 한 곳, 사용처는 여럿)

### 왜 코드인가

| 규칙 조각 | 위치 | 이유 |
|-----------|------|------|
| 게이트 로직 ("정기 참석 OR 개설") | **코드** (`calcExemption` 본문) | 불리언 조합이라 데이터화하면 룰엔진 수준 오버스펙 |
| 임계값·감면비율 (4/8회, 50%/100%) | **상수** (`lib/constants/dues-quest.ts`) | 파라미터지만 거의 안 바뀜. DB+관리UI는 과함 |
| 회비 단가 (2,000원) | **기존 DB** (`fee_policy_cfg.monthly_fee_amt`) | 감면액 = 단가 × 비율로 계산 → 회비 변동 자동 대응 |

> ⚠️ **"배치에 녹이지 말 것."** 규칙을 배치 함수 안에 직접 박으면 퀘스트 카드가 같은 판정을
> 복붙하게 되어 어긋난다. 반드시 **공유 모듈로 분리**하고 배치·카드가 호출만 한다.

### 모듈 구조

```text
lib/constants/dues-quest.ts   ─ 숫자 상수
lib/dues/calc-exemption.ts    ─ calcExemption() 순수 함수 (게이트 + 티어 판정)
        ▲                              ▲
  배치(8)가 호출              퀘스트 카드(7)가 호출 (동일 결과 보장)
```

```typescript
// lib/constants/dues-quest.ts
export const DUES_QUEST = {
  gate: { regularAttend: 1, hosted: 1 },   // 게이트 임계값
  tiers: [
    { attendCnt: 4, exmRatio: 0.5 },        // 4회 → 회비의 50%
    { attendCnt: 8, exmRatio: 1.0 },        // 8회 → 전액
  ],                                        // 높은 티어가 우선 (내림차순 평가)
} as const;

// lib/dues/calc-exemption.ts
export type AttendStats = { attendCnt: number; regularAttendCnt: number; hostedCnt: number };
export type ExemptionResult = {
  gatePassed: boolean;
  gateDetail: { regularAttend: number; hosted: number };
  attendCnt: number;
  exmAmt: number;
  nextTier?: { attendCnt: number; remaining: number; exmAmt: number };
  reason: string;
};

export function calcExemption(stats: AttendStats, monthlyFeeAmt: number): ExemptionResult {
  const amtFor = (ratio: number) => Math.min(Math.round(monthlyFeeAmt * ratio), monthlyFeeAmt);
  const gateDetail = { regularAttend: stats.regularAttendCnt, hosted: stats.hostedCnt };
  const tiersDesc = [...DUES_QUEST.tiers].sort((a, b) => b.attendCnt - a.attendCnt);   // 내림차순
  const tiersAsc = [...DUES_QUEST.tiers].sort((a, b) => a.attendCnt - b.attendCnt);    // 오름차순

  // 다음 티어(아직 못 넘은 가장 가까운 티어) — 게이트와 무관하게 진행도 안내용
  const next = tiersAsc.find((t) => stats.attendCnt < t.attendCnt);
  const nextTier = next
    ? { attendCnt: next.attendCnt, remaining: next.attendCnt - stats.attendCnt, exmAmt: amtFor(next.exmRatio) }
    : undefined;

  const gatePassed =
    stats.regularAttendCnt >= DUES_QUEST.gate.regularAttend ||
    stats.hostedCnt >= DUES_QUEST.gate.hosted;
  if (!gatePassed)
    return { gatePassed: false, gateDetail, attendCnt: stats.attendCnt, exmAmt: 0, nextTier, reason: "게이트 미충족" };

  const tier = tiersDesc.find((t) => stats.attendCnt >= t.attendCnt);
  const exmAmt = tier ? amtFor(tier.exmRatio) : 0;
  return {
    gatePassed, gateDetail, attendCnt: stats.attendCnt, exmAmt, nextTier,
    reason: tier ? `출석 ${stats.attendCnt}회 / ${tier.exmRatio === 1 ? "전액 면제" : "감면"}` : "참석 횟수 부족",
  };
}
```

> **반올림 경계**: 회비가 홀수(예 1,500원)면 50% = 750원. `Math.round`로 처리하되, **티어 금액 합이
> 회비를 넘지 않도록 `Math.min(..., monthlyFeeAmt)`로 클램프**(전액 티어가 상한). 단계형이라 누적 아님.
> 회비 0/음수는 `fee_policy_cfg` CHECK(`> 0`)으로 차단되어 입력 안 됨.

> **나중에 DB화가 필요해지면**: `DUES_QUEST`를 읽던 자리를 `fee_policy_cfg`(또는 신규 컬럼)에서
> 읽도록 바꾸면 끝. `calcExemption`은 "숫자를 인자로 받는" 함수라 호출부는 안 바뀐다. (§11 대안)

## 4. 집계 — `get_member_monthly_activity` RPC (칭호·랭킹과 공유)

> 출석·개설·참석 집계는 **회비 감면 전용이 아니라**, 곧 만들 **칭호·랭킹**도 쓸 공통 기반이다
> (`.claude/docs/gathering-design.md` §통계). 그래서 **집계의 인터페이스(RPC)를 지금 확정**하고,
> 내부 구현은 미룬다.

### 4.1 RPC 시그니처 (경계선)

```text
get_member_monthly_activity(p_team_id uuid, p_mem_id uuid, p_ym text)
  → { attend_cnt int, regular_attend_cnt int, hosted_cnt int }
```

- 이 함수가 **중립적인 숫자만** 내놓는다. 게이트 판정(정모/개설) 같은 지엽적 로직은
  여기 두지 않고 §3 `calcExemption`(소비처)에 둔다.
- 호출자: 출석 감면 배치(§8), 퀘스트 카드(§7). 추후 칭호 evaluator·랭킹이 재사용.
- `hosted_cnt`는 `gthr_attd_rel` JOIN과 **무관하게** `gthr_mst.crt_by`로만 센다(게이트 판정용: 개설 사실 자체, 참석 여부 무관).

### 4.2 내부 구현 (지금은 즉석 쿼리)

KST 월 경계는 **기존 헬퍼 `public.kst_day_start(date)` / `public.kst_day_end_excl(date)`**
([20260627180000_kst_date_boundary_helpers.sql](supabase/migrations/20260627180000_kst_date_boundary_helpers.sql))를 재사용한다.
월 시작 = `p_ym` 1일, 끝(exclusive) = **다음 달 1일의 시작**:

```sql
-- p_ym: 'YYYY-MM'  → 그 달 1일 date 로 변환
-- month_start := to_date(p_ym || '-01', 'YYYY-MM-DD')
-- next_month  := month_start + interval '1 month'
SELECT
  count(*) FILTER (WHERE a.mem_id = p_mem_id)                                      AS attend_cnt,
  count(*) FILTER (WHERE a.mem_id = p_mem_id AND g.gthr_type_enm = 'regular')      AS regular_attend_cnt,
  count(DISTINCT g.gthr_id) FILTER (WHERE g.crt_by = p_mem_id)                     AS hosted_cnt
FROM gthr_mst g
LEFT JOIN gthr_attd_rel a ON a.gthr_id = g.gthr_id AND a.mem_id = p_mem_id
WHERE g.team_id = p_team_id
  AND g.del_yn = false
  AND g.stt_at >= public.kst_day_start(month_start)          -- p_ym 1일 00:00 KST
  AND g.stt_at <  public.kst_day_start(next_month::date);    -- 다음 달 1일 00:00 KST (exclusive)
```

> `kst_day_end_excl`는 "특정 *날짜*의 익일"이라 월 끝엔 안 맞다(말일+1일 계산이 번거로움). 대신
> **다음 달 1일에 `kst_day_start`를 적용**하면 그게 곧 이번 달의 exclusive 끝이라 더 깔끔하다.

> **성능 승급 경로**: 칭호·랭킹까지 붙어 즉석 쿼리가 느려지면, 이 RPC **속만** 월별 집계
> 스냅샷 테이블(`mem_monthly_activity_stat`)을 읽도록 교체한다. **호출부(배치·카드)는 안 건드린다.**
> `gthr_*`는 원천 데이터라 과거가 보존되므로 어느 과거 월이든 정확히 소급 집계된다.

## 5. 면제 이력 — `fee_due_exm_hist` 재사용 + `rflt_yn` 컬럼 추가

### 5.1 테이블 재사용 + 컬럼 추가

`fee_due_exm_hist`를 그대로 쓰되, **반영여부 컬럼 `rflt_yn`을 추가**한다 (§6에서 사용).

```sql
ALTER TABLE public.fee_due_exm_hist
  ADD COLUMN rflt_yn boolean NOT NULL DEFAULT false;
COMMENT ON COLUMN public.fee_due_exm_hist.rflt_yn IS '잔액 반영 여부 (재계산이 합산하면 true)';

-- 기존 데이터(이미 잔액에 녹은 면제)는 true로 백필
UPDATE public.fee_due_exm_hist SET rflt_yn = true WHERE del_yn = false;

-- 퀘스트 면제 멱등 보장: 월당 1건 (배치 재실행 안전)
CREATE UNIQUE INDEX uk_fee_exm_hist_quest
  ON public.fee_due_exm_hist (team_id, mem_id, aply_ym)
  WHERE grant_src_enm = 'rule_attd_quest' AND del_yn = false;

-- 미반영 면제 합산용 인덱스
CREATE INDEX ix_fee_exm_hist_unrflt
  ON public.fee_due_exm_hist (team_id, mem_id)
  WHERE rflt_yn = false AND del_yn = false;
```

### 5.2 퀘스트 면제 row 값

| 컬럼 | 값 |
|------|-----|
| `aply_ym` | 대상 월 `YYYY-MM` |
| `exm_amt` | `calcExemption` 결과 (1000 또는 2000) |
| `grant_src_enm` | **`rule_attd_quest`** (신규 enum 값 — §5.3) |
| `exm_cfg_id` | NULL (규칙 면제와 달리 규칙 row 없음) |
| `rflt_yn` | **false** (생성 시) → 재계산이 `true`로 마킹 |
| `rsn_txt` | 예: `"출석 감면: 참석 8회/정모 참석"` |
| `aprv_by_mem_id`/`aprv_at` | 배치 실행 주체(시스템/관리자) |

> 퀘스트 면제 식별 키 = **`grant_src_enm='rule_attd_quest'`** (단일 enum 값으로 명확 식별).

### 5.3 신규 enum 값 `rule_attd_quest` — 분리 이유 (코드 검증)

> **확정**: `fee_grant_src_enm`에 `'rule_attd_quest'`를 추가한다. `rule_attd`(관리자 규칙 면제)와 **반드시 분리.**

코드 확인 결과, 퀘스트 면제를 기존 `rule_attd`로 두면 **기존 로직에 휩쓸리는 부작용**이 있다:

- 🔴 **`rollback-snapshot.ts`가 `rule_attd` 면제를 일괄 삭제**([rollback-snapshot.ts:35](app/actions/dues/rollback-snapshot.ts#L35)):
  스냅샷 롤백 시 `grant_src_enm='rule_attd'`인 면제를 `del_yn=true` 처리한다. 퀘스트 면제를 `rule_attd`로
  두면 **관리자가 스냅샷 롤백할 때 출석 감면까지 같이 지워진다.** → `rule_attd_quest`로 분리하면 안전.
- **`manual` 면제도 이미 `exm_cfg_id=null`을 쓴다**([manage-exemption-hist.ts:19-20](app/actions/dues/manage-exemption-hist.ts#L19-L20)):
  그래서 `exm_cfg_id IS NULL` 단독 식별은 `manual`과 모호. **enum 값으로 식별해야 깔끔.**

```sql
ALTER TYPE public.fee_grant_src_enm ADD VALUE 'rule_attd_quest';
COMMENT ON TYPE public.fee_grant_src_enm IS '회비 면제 출처: manual(수동) | rule_attd(관리자 규칙) | rule_attd_quest(출석 감면 퀘스트)';
-- ⚠️ ALTER TYPE ... ADD VALUE 는 트랜잭션 내에서 즉시 사용 불가(Postgres 제약) → 별도 마이그레이션으로 먼저 추가
-- 이후 database.types.ts 재생성 필요
```

> 분리 후 점검할 기존 로직: `rollback-snapshot.ts`(휩쓸림 방지 확인), 면제 목록 표시(`exemptions/page.tsx`,
> `dues-exemptions-client.tsx`의 "수동/규칙" 라벨에 "출석" 추가), 멱등 인덱스(`uk_fee_exm_hist_quest`)의
> `grant_src_enm = 'rule_attd_quest'` 조건으로 갱신.

## 6. 재계산 — `rflt_yn` 기반 합산 + 마킹 (핵심 수정)

> **문제 (대화에서 발견)**: 현재 재계산은 면제를 `aply_ym`(귀속 월) 기준으로만 합산해서,
> "기준월보다 과거에 나중에 꽂힌 면제"(배치를 늦게 돌린 경우)를 **영영 못 잡는다.**
> 예: 6/1 재계산(기준월=6월) 후, 7월에 4월치 배치를 돌리면 `aply_ym=4월`은 SELECT에서 누락 ❌

### 6.1 해결: 합산 기준을 `aply_ym` → `rflt_yn`으로 교체

현재 `recalculateBalance`의 면제 합산([recalculate-balance.ts](app/actions/dues/recalculate-balance.ts) `exmFromYm` / `.gte("aply_ym", ...)`)을 다음으로 바꾼다:

```text
-- AS-IS: aply_ym >= exmFromYm  (월 기준 → 과거 소급 면제 누락)
-- TO-BE: rflt_yn = false        (귀속월 무관, "아직 잔액에 안 들어간 것"을 전부)

1. 이 멤버의 rflt_yn=false 면제 전부 SELECT (관리자 규칙 면제 + 퀘스트 면제 공통)
2. totalExempted = Σ exm_amt
3. 잔액 스냅샷 INSERT (newBal = baseBal + totalPaid + totalExempted - totalCharged)
4. 방금 합산한 면제 row들을 rflt_yn=true 로 UPDATE   ← 마킹
```

- **귀속월(4월)이 과거든 당월(6월)이든 상관없이** "안 반영된 면제(`rflt_yn=false`)"는 무조건 한 번 합산 → 마킹. 배치를 2달 늦게 돌려도 다음 재계산이 잡는다.
- 같은 달 두 번 재계산? → 1차가 이미 `true`로 마킹했으니 2차는 그 row를 안 가져옴 → **이중 합산 없음.**

### 6.1.1 ⚠️ `baseBal`·`exmFromYm`과의 정합성 (구현 필수 검증)

현재 재계산은 **이전 스냅샷의 `bal_amt`를 `baseBal`로 이어받는다**([recalculate-balance.ts:69](app/actions/dues/recalculate-balance.ts#L69)).
즉 `baseBal`에는 **"직전 재계산까지 이미 합산된 면제"가 녹아 있다.** 그래서 `rflt_yn` 전환 시
**이중 합산을 막는 불변식**은 다음과 같다:

> **불변식**: `baseBal`에 녹은 면제 = `rflt_yn=true`인 면제. 재계산은 `rflt_yn=false`만 더한다.

- **백필 `true`가 안전한 근거**: 기존 면제는 전부 ⓐ재계산이 직접 INSERT한 규칙 면제(생성=그 재계산이 즉시 반영) 또는 ⓑ관리자 수동 면제(`manual`)뿐이고, 둘 다 **생성된 뒤 한 번이라도 재계산이 돌았다면 이미 `bal_amt`에 반영**됐다. → `del_yn=false` 전부 `true` 백필이 맞다.
  > ⚠️ **단 예외**: 백필 시점에 "생성됐지만 아직 재계산이 안 돈" 면제가 있다면(예: manual 면제 추가 직후 재계산 전), 그건 실제론 미반영인데 `true`로 잘못 칠해져 **영영 누락**된다. → **백필 직전 전체 재계산을 한 번 돌려 모든 면제를 반영시킨 뒤** 백필하거나, 백필을 `true`로 한 뒤 **전체 재계산으로 검증**(아래 회귀 테스트)해 차이 나는 멤버를 잡는다.
- **백필 누락 시**: 전 멤버 잔액이 면제만큼 이중 가산된다. **→ enum 추가 → 컬럼/백필 → 재계산 수정을 같은 배포 흐름에 묶고, 백필 후 첫 재계산 전 잔액 검증 필수.**
- **`exmFromYm`/면제 INSERT 루프(6.2 (가))는 그대로 두되**, INSERT 시 신규 row는 `rflt_yn=false`로 생성되어 같은 재계산의 (나)가 합산한다. (가)는 `aply_ym` 기준 INSERT, (나)는 `rflt_yn` 기준 합산 — 변수가 달라 안 충돌(6.2).
- **회귀 테스트(필수)**: 백필 직후 전체 재계산 → 모든 멤버 잔액이 백필 이전과 **동일**해야 한다(면제 이중 가산·누락 둘 다 없음). 차이 나는 멤버 = 위 예외에 걸린 케이스 → 수동 점검.

### 6.2 기존 면제규칙 INSERT 루프와의 독립성 (대화에서 검증)

재계산 안에서 면제는 **두 곳**에서 다뤄지며 서로 다른 변수를 봐서 **독립적**이다:

| | 보는 기준 | 동작 |
|---|---|---|
| (가) 면제규칙 INSERT 루프 | `calcFromMonth`(마지막 계산월 +1 ~ 오늘월) | 이 달 면제규칙 있으면 INSERT (없을 때만, 멱등) |
| (나) 면제 합산 | **`rflt_yn=false`** (← 6.1로 변경) | 미반영 면제 전부 합산 + 마킹 |

- (가)는 월 순회라 같은 달을 재방문하지 않으므로 중복 INSERT 없음 — **기존 그대로 둠.**
- (나)만 `rflt_yn` 기준으로 바뀜. (가)가 막 넣은 면제(`rflt_yn=false`)도 같은 재계산의 (나)가 함께 합산·마킹한다.
- **순서 보장 필수**: 재계산 내에서 **(가) 면제 INSERT(false) → (나) 미반영 전부 합산+마킹(true)** 순. 그러면 규칙 면제든 배치 면제든 **"rflt_yn=false면 합산, 하면 true"** 한 규칙으로 통일된다.

### 6.3 ⚠️ 원자성 — 마킹과 잔액 저장은 한 트랜잭션 (RPC 강제)

`rflt_yn=true` 마킹과 잔액 스냅샷 INSERT가 **원자적이어야** 한다. 마킹만 되고 잔액 저장이
실패하면 "반영됨인데 잔액엔 없는" 유령 면제가 생긴다.

> ⚠️ **Supabase JS 클라이언트는 여러 쿼리를 한 트랜잭션으로 묶지 못한다.** 현재 재계산은
> 멤버별로 `update`→`insert`를 순차 호출([recalculate-balance.ts:53-235](app/actions/dues/recalculate-balance.ts#L53-L235))하는데,
> `rflt_yn` 마킹을 추가하면 **반드시 DB 함수(RPC)로 원자화**해야 한다. JS에서 나눠 호출하면 중간 실패 시 정합성 깨짐.

**신규 RPC** (멤버 1명의 스냅샷 갱신 전체를 한 트랜잭션에서). 현재 코드([recalculate-balance.ts:203-232](app/actions/dues/recalculate-balance.ts#L203-L232))는 **vers 밀기 → 스냅샷 INSERT**가 이미 비원자적(중간 실패 시 깨짐)인데, 여기에 면제 마킹까지 더해지므로 **이 3단계를 RPC로 묶는다**:

```sql
-- recalc_member_balance(p_team_id, p_mem_id, p_now timestamptz, p_base_bal, p_total_paid, p_total_charged,
--                        p_last_calc_at, p_last_ref_pay_id) RETURNS bigint  -- new_bal
--  트랜잭션 본문:
--   ① SELECT exm_hist_id, exm_amt FROM fee_due_exm_hist
--        WHERE team,mem AND rflt_yn=false AND del_yn=false   -- 미반영 면제(규칙+퀘스트 공통)
--        → total_exempted, exm_ids[]
--   ② new_bal = p_base_bal + p_total_paid + total_exempted - p_total_charged
--   ③ 기존 스냅샷(vers=0) → vers = max(vers)+1 로 밀기   (현행 203-218 로직)
--   ④ 새 스냅샷 INSERT (vers=0, bal_amt=new_bal, last_calc_dt=p_now,
--        last_calc_at, last_ref_pay_id, last_ref_exm_hist_id = exm_ids 중 최신)
--   ⑤ UPDATE fee_due_exm_hist SET rflt_yn=true WHERE exm_hist_id = ANY(exm_ids)
--   → ③④⑤가 한 트랜잭션. 어느 하나 실패 시 전체 롤백(유령 면제·중복 vers 방지)
```

- **분업**: 서버 액션은 **멤버 루프 + 정책/입금/면제 *조회* (`totalPaid`·`totalCharged`·`getTxnInfo`/입금필터)**를 유지하고, **쓰기(③④⑤)만 RPC로** 넘긴다. 또는 면제 합산(①)까지 RPC로 넣어 입력 인자를 줄인다(구현 판단).
- **면제 SELECT 기준 변경**: 기존 `aply_ym`/`exmFromYm` 필터([recalculate-balance.ts:116-118,175-183](app/actions/dues/recalculate-balance.ts#L116-L118))를 **`rflt_yn=false`** 로 교체(§6.1). `exmFromYm` 관련 코드 제거.
- **chunk 병렬 주의**: 현재 멤버를 `Promise.all`로 병렬 처리([recalculate-balance.ts:55](app/actions/dues/recalculate-balance.ts#L55)) — RPC가 멤버 단위라 멤버 간 독립이면 병렬 유지 가능. 단 같은 멤버 동시 호출은 막아야(배치·수동 재계산 동시 실행 시 잠금/직렬화 고려).

### 6.4 마감 후 변동(참석 취소)은 자동 회수하지 않음 — 확정

> **정책**: 출석 감면은 **마감 시점 스냅샷으로 확정**된다. 마감 후 참석을 취소해도
> 이미 부여된 면제를 **자동으로 회수하지 않는다.** (참여 독려 목적, 사후 취소는 드묾.)

따라서 "이미 `rflt_yn=true`인 면제를 되돌리는" 복잡한 로직(역면제 보정 row·전체 재산정·rflt_yn 되돌림)은
**도입하지 않는다.** 명백한 오류로 조정이 필요한 드문 경우는 **관리자가 수동으로** 처리한다:

- `add-manual-transaction`(회비 내역에 -금액 거래 추가) 또는 `create-exemption`/면제 관리 도구로
  관리자가 임의 조정. → 자동화 비용 0, 운영 유연성 확보.

→ 재계산은 `rflt_yn=false` 면제를 **합산·마킹만** 하면 되고(이미 반영된 것은 다시 안 건드림), 회수 분기가 없어 단순하다.

## 8. 출석 감면 배치 — 월 마감 후 전월 확정

> **확정**: 출석은 그 달이 끝나야 확정된다. 그래서 잔액 재계산과 **별개로**,
> **"월 마감 후 전월 출석 감면만 면제 내역에 쌓는 배치"** 를 둔다.

### 8.1 배치 로직 (전월 1회 확정, 회수 없음)

> **정책 (확정)**: **한번 마감하면 확정.** 마감 후 참석을 취소해도 **자동 회수하지 않는다.**
> 참여 독려가 목적이고 사후 취소는 드물다. 명백한 오류 조정이 필요하면 **관리자가 수동으로**
> 회비 내역에서 -금액 거래/면제 조정을 추가한다(`add-manual-transaction` / `create-exemption` 기존 도구).
> → **마감월 재집계(3개월 룩백)·역면제 row·rflt_yn 되돌림 로직 전부 불필요.**

**대상 월** = 전월 1개월만 (예: 7/1 실행 → 2026-06).

**대상 멤버** = 대상 월에 활성이었던 멤버:
`team_mem_rel`에서 `team_id` + `vers=0` + `del_yn=false` + `mem_st_cd='active'`,
가입월 ≤ 대상월.

```text
ym = 전월
for 대상 멤버:
  stat   = get_member_monthly_activity(team, mem, ym)         -- §4
  exmAmt = calcExemption(stat, monthlyFeeAmt(ym)).exmAmt      -- §3 (해당 월 회비단가)
  if exmAmt > 0:
    INSERT fee_due_exm_hist (aply_ym=ym, exm_amt=exmAmt,
        grant_src_enm='rule_attd_quest', exm_cfg_id=NULL, rflt_yn=false, rsn_txt=...)
    -- 멱등: uk_fee_exm_hist_quest 충돌 시 무시(ON CONFLICT DO NOTHING)
  -- exmAmt==0 이면 아무것도 안 함 (회수 분기 없음)
```

- **멱등**: `uk_fee_exm_hist_quest`(월당 1건)로 배치 N회 재실행해도 중복 INSERT 안 됨. 이미 있으면 그대로 둠(재계산이 금액 바꾸지 않음 — 확정값).
- **회비단가 시점**: `calcExemption`에 넘기는 `monthlyFeeAmt`는 **대상 월(ym)에 적용되던** `fee_policy_cfg` 단가(현재 단가 아님). 과거 소급 정확성.
- **잔액 반영 안 함**: 배치는 면제 row만 INSERT. 잔액은 다음 재계산이 §6대로 `rflt_yn=false`를 합산·마킹.
- **재계산 누락 안전망 유지**: 배치를 늦게 돌려 과거 월 면제가 뒤늦게 INSERT돼도, `rflt_yn=false`라 다음 재계산이 §6.1대로 반드시 합산한다(이 안전망은 그대로 가치 있음).

### 8.2 실행 방식 — 수동 트리거 → cron 승격

| 단계 | 방식 | 비고 |
|------|------|------|
| **1차(출시)** | 관리자 페이지 "전월 출석 감면 집계" 버튼 (서버 액션 `withAdmin`) | 칭호도 현재 이벤트 기반이라 일관. 빠른 출시·검증 |
| **2차(승격)** | Vercel cron `/api/cron/dues-exemption` (`"0 0 1 * *"` 매월 1일) | 기존 `/api/cron/keep-alive` 패턴 복제. **1차 버튼이 부르던 로직 그대로 감쌈(재작성 0)** |

> **수동이라 까먹어도 안전**: 배치를 늦게 돌려도 면제가 `rflt_yn=false`로 쌓이고, **그 다음 재계산이
> §6.1대로 반드시 잡는다.** 누락이 영구화되지 않는다 — 이것이 `rflt_yn` 설계의 안전망.
> (현행 cron 인프라: `vercel.json` crons + `app/api/cron/keep-alive` + pg_cron 사용 전례 있음)

### 8.3 당월 처리

배치는 **전월만** 확정한다. 당월(진행 중)은 면제 row를 만들지 않고 **퀘스트 카드(§7)로 실시간 안내**만 한다.
→ 월 중 참석 취소로 횟수가 줄어드는 변동을 면제 내역에 반영했다 회수하는 혼란을 피한다.

## 9. 동작 시나리오 (정합 검증)

| 시점 | 일어나는 일 |
|------|-------------|
| 6월 한 달 | 회원이 정모 참석 + 벙 참석으로 4회 채움. 퀘스트 카드: "현재 감면 1,000원(마감 후 반영)" |
| 7/1 배치 | 전월(6월) 집계 → `calcExemption` → 6월 면제 row INSERT (`rflt_yn=false`) |
| 7월 재계산 | `rflt_yn=false`인 6월 면제 합산 → 잔액 -1,000 보정 → 그 row `rflt_yn=true` 마킹 |
| (배치 깜빡) 4월치를 7월에 돌림 | 4월 면제 row INSERT(`rflt_yn=false`) → 다음 재계산이 `aply_ym` 무관하게 잡아 합산·마킹 ✅ |
| 같은 달 재계산 2번 | 1차가 `true`로 마킹 → 2차는 미반영분만 봄 → 이중 합산 없음 ✅ |
| 마감 후 참석 취소 (§9.1) | **자동 회수 안 함.** 면제 그대로 유지. 필요 시 관리자 수동 조정 |

### 9.1 마감 후 참석 취소 — 자동 회수 없음 (확정)

> 7/1 배치로 6월 면제 확정 → 7/2 재계산으로 잔액 반영 완료. 이후 7/5에 회원이 6월 참석을 취소해도(4→3회):
> **이미 부여된 6월 감면은 그대로 유지된다.** (§6.4 정책)

- **이유**: 참여 독려가 목적이고, 마감 시점에 4회를 채웠다는 사실은 그달의 노력으로 확정됐다. 사후 취소는 드물고, 그 드문 케이스 때문에 회수 자동화(역면제·전체재산정·마감월 재집계)를 만드는 건 과하다.
- **악용 우려**: 마감 *후* 취소는 이득이 없고(면제 유지), 마감 *전* 취소는 아직 면제가 없으니(당월은 §8.3대로 미생성) 무해. → 악용 동기 없음.
- **드문 정정**: 명백한 오류는 관리자가 회비 내역에서 수동 조정(`add-manual-transaction`로 -금액 거래 등). 운영 유연성으로 충분.

## 11. 결정 보류 / 대안

- **규칙 DB화 (대안)**: 관리자가 임계값·금액을 자주 만지고 싶어지면 `DUES_QUEST` → `fee_policy_cfg`
  신규 컬럼(`quest_tier1_cnt` 등)으로 이전. `calcExemption` 호출부는 불변이라 무손실 승급.
- **enum 분리 (확정)**: `fee_grant_src_enm`에 `'rule_attd_quest'` 추가(§5.3). `rule_attd`로 두면
  `rollback-snapshot.ts`가 출석 감면을 휩쓸어가는 부작용이 있어 **분리 확정.**
- **집계 스냅샷 테이블**: 칭호·랭킹 붙어 성능 이슈 생기면 RPC 내부를 스냅샷으로 교체(§4.2).
- **칭호 자동화 연계**: 월 마감 배치(§8)에 칭호 evaluator를 함께 태우는 방안 — 출석 감면 cron 검증 후 확장.
- **과거 backfill**: 제도 도입 월부터 적용. 도입 이전 달은 배치 대상에서 제외.
- **마감 후 회수 안 함 (확정)**: 참여 독려 목적상 한번 마감하면 면제 확정. 정정은 관리자 수동(§6.4/§9.1). → 마감월 재집계·역면제·전체재산정 전부 미도입.

## 부록. 참석 토글 토스트 (선택)

달성 순간 피드백. 잔액·면제와 무관한 순수 UX(선택 기능).

**`justUnlocked` 판정** — "이번 토글로 *처음* 넘은 임계값"을 알려면 **토글 전/후를 비교**해야 한다.
`calcExemption` 결과만으로는 "이미 4회였는데 5회로 늘어 재호출"과 구분 불가하므로:

```text
toggleGatheringAttendance(등록):
  before = get_member_monthly_activity(해당 모임 stt_at 월)   -- 토글 직전 통계
  attd row 등록
  after  = get_member_monthly_activity(같은 월)               -- 토글 직후 통계
  // calcExemption 반환(§3)으로 "단계 레벨" 도출: none(0) < gate(1) < tier1(2) < tier2(3)
  level(r) = !r.gatePassed ? (r.gateDetail.regularAttend||r.gateDetail.hosted ? 1 : 0)
             : r.exmAmt >= 전액 ? 3 : r.exmAmt > 0 ? 2 : 1
  before = level(calcExemption(b, fee));  after = level(calcExemption(a, fee))
  justUnlocked = after > before ? ["none","gate","tier1","tier2"][after] : null
  반환 { attending:true, justUnlocked }
```

| `justUnlocked` | 토스트 |
|----------------|--------|
| `gate` | "🔓 감면 조건 달성! 4회 참석하면 1,000원 감면" |
| `tier1`(4회) | "🎉 출석 4회 — 이번 달 1,000원 감면 예정!" |
| `tier2`(8회) | "🏆 출석 8회 — 이번 달 회비 전액 면제 예정!" |
| `null` | 일반 참석 토스트만 |

- 참석 취소(attending=false) 시엔 토스트 없음. 면제 변동은 배치(§8)·재계산이 처리.
- `GatheringAttendButton`(상세)과 홈 리스트뷰 토글이 동일 액션을 쓰므로 양쪽 동작.
- ⚠️ 토글마다 RPC 2회 추가 호출 비용. 단일 멤버·당월이라 가볍지만(§성능), 부담되면 토글 응답엔
  `after`만 싣고 `justUnlocked`는 클라이언트가 이전 값과 비교하는 방식도 가능.
