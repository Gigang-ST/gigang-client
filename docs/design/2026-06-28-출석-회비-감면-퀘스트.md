# 출석 기반 회비 감면(퀘스트) 설계 문서

> 작성일: 2026-06-28
> 상태: 설계 완료 — 구현 대기
> 관련 문서: `2026-03-26-회비시스템.md`, `.claude/docs/gathering-design.md`

---

## 1. 개요

모임 출석을 독려하기 위해, 한 달 동안의 출석·개설 활동에 따라 **당월 회비를 자동 감면**하는 제도.
"퀘스트"처럼 진행도를 보여주고, 조건을 달성하면 면제 내역에 자동으로 반영한다.

- 월 회비: **2,000원** (`fee_policy_cfg.monthly_fee_amt`)
- **1단계 감면 1,000원** / **2단계 전액(2,000원) 면제**
- 회원이 "내가 얼마나 참석해서 얼마나 감면받았는지"를 한눈에 볼 수 있어야 한다.

## 2. 감면 규칙 (확정)

### 2.1 게이트 조건 (둘 중 하나 충족 필수)

| 게이트 | 판정 |
|--------|------|
| 정모 1회 참석 | 해당 월 `gthr_type_enm = 'regular'` 모임에 **참석**(`gthr_attd_rel` row 존재) ≥ 1 |
| 벙 1회 개설 | 해당 월 본인이 **개설**(`gthr_mst.crt_by = 본인`)한 모임 ≥ 1 (타입 무관) |

> **게이트를 통과하지 못하면 참석 횟수가 아무리 많아도 감면 없음.**
> (예: 일반 벙 5번 참석 + 정모 0 + 개설 0 → 감면 0원)

### 2.2 참석 횟수 (게이트 통과를 전제로 적용)

| 누적 참석 | 감면액 |
|-----------|--------|
| 4회 이상 | 1,000원 |
| 8회 이상 | 2,000원 (전액 면제) |

- **참석 횟수 = 해당 월 `gthr_attd_rel`에 본인 row가 있는, `stt_at`이 그 달에 속한 모든 모임 수** (타입 무관).
- 본인이 개설한 모임은 개설 시 자동으로 참석 처리되므로 **참석 1회에 자연히 포함**된다.
  → "벙 개설하고 7번 더 참석" = 개설한 모임(1) + 추가 참석(7) = **총 8회 → 전액 면제**.

### 2.3 예시

| 활동 | 게이트 | 참석 횟수 | 감면 |
|------|--------|-----------|------|
| 정모 참석 + 일반 벙 3회 참석 | ✅ 정모 | 4회 | 1,000원 |
| 벙 개설(=참석1) + 추가 참석 7회 | ✅ 개설 | 8회 | 2,000원(전액) |
| 일반 벙 5회 참석 (정모X, 개설X) | ❌ | 5회 | 0원 |
| 정모 참석 + 추가 참석 9회 | ✅ 정모 | 10회 | 2,000원(전액) |

> **감면은 단계형(누적 아님).** 8회 도달 시 2,000원이지 1,000+2,000=3,000이 아니다.
> 면제 금액은 월 회비(2,000원)를 초과할 수 없다.

### 2.4 월 귀속 기준

- 모임의 **`stt_at`(KST)이 속한 월**로 귀속. (참석 토글한 시점이 아님)
- 미래 모임에 미리 참석을 눌러도 그 모임의 `stt_at` 월로 집계된다.

## 3. 기존 회비 시스템과의 정합 (핵심)

### 3.1 "입금처리 해줘야 하나?" → **불필요**

기존 `recalculateBalance`의 잔액 공식:

```
newBal = baseBal + totalPaid + totalExempted - totalCharged
```

면제는 **입금(`totalPaid`)이 아니라 별도 항목(`totalExempted`)으로 잔액에 가산**된다.
따라서:

- 이미 당월 재계산이 돌아 회비 2,000원이 부과(`totalCharged`)된 상태여도,
- `fee_due_exm_hist`에 면제 1,000원 row가 추가되면,
- **다음 재계산 때 `totalExempted`에 잡혀 잔액이 자동으로 +1,000원 보정**된다.

→ 가짜 입금(`fee_due_pay_hist`) row를 만들 필요 없음. 면제 이력만 쌓으면 회계가 맞는다.

### 3.1.1 "반영 예정 / 반영됨" 표시 — 별도 컬럼 없이 시각 비교로 해결 (핵심 결정)

> **문제**: 면제를 적립한 순간 거래내역엔 "면제 +1,000원"이 바로 뜨지만, 잔액(`fee_mem_bal_snap`)은
> 다음 재계산 전까진 안 줄어든다. 회원이 "면제됐다는데 잔액은 왜 그대로?"라고 헷갈린다.
>
> **해결**: 즉시 재계산도, 반영여부 컬럼 추가도 하지 않는다. **이미 있는 두 시각만 비교**한다.

- 스냅샷에는 재계산 시각 **`last_calc_dt`(기준일시)** 가 찍힌다 (`recalculate-balance.ts`).
- 면제 내역(`fee_due_exm_hist`)에는 적립 시각 **`crt_at`** 이 있다.
- 화면에서 두 값을 비교만 한다:

| 조건 | 의미 | 회원 표시 |
|------|------|-----------|
| `exm.crt_at` ≤ `snap.last_calc_dt` | 그 재계산에 이미 포함됨 | 표시 없음(=반영됨) |
| `exm.crt_at` > `snap.last_calc_dt` | 아직 잔액에 안 잡힘 | **`반영 예정` 뱃지** |
| 스냅샷 자체가 없음 | 한 번도 정산 안 됨 | **`반영 예정` 뱃지** |

```
회비 내역                        기준일시 6.15 14:00
──────────────────────────────────────────
6.30  면제 +1,000원  [반영 예정]   ← crt_at(6.30) > 기준일시 → 다음 정산 때 잔액 반영
6.10  면제 +1,000원               ← crt_at(6.10) ≤ 기준일시 → 이미 잔액에 들어감(뱃지 없음)
6.05  입금 +2,000원
```

→ 회원은 "반영 예정 = 다음 정산 때 잔액에 들어올 것"으로 직관적으로 이해.
잔액 숫자와 내역이 일시적으로 안 맞아도 헷갈리지 않는다.
**추가 스키마·즉시 재계산 불필요** — `dues/page.tsx`가 이미 `last_calc_dt`와 면제 내역을
함께 조회하므로([dues/page.tsx](app/(info)/profile/dues/page.tsx)) 시각 비교 한 줄이면 된다.

> 같은 원리를 **입금(`fee_due_pay_hist`)·기타 거래에도 적용 가능** — 일관성을 위해 면제뿐 아니라
> "기준일시 이후의 모든 내역"에 `반영 예정` 뱃지를 붙이는 것을 권장(범위는 구현 시 결정).

### 3.2 면제 이력 테이블 재사용

`fee_due_exm_hist`를 그대로 사용한다 (신규 테이블 불필요):

| 컬럼 | 퀘스트 감면 시 값 |
|------|------------------|
| `aply_ym` | 대상 월 `YYYY-MM` |
| `exm_amt` | 1000 또는 2000 |
| `grant_src_enm` | **`rule_attd`** (기존 enum 값 재사용 — "출석 규칙 기반") |
| `exm_cfg_id` | **NULL** (관리자 면제 규칙이 아닌 자동 감면이므로) |
| `rsn_txt` | 예: `"출석 감면: 참석 8회/정모 참석"` |
| `aprv_by_mem_id` / `aprv_at` | 시스템 자동(NULL 또는 시스템 멤버) |

> ⚠️ **중복 방지가 관건.** 같은 월에 면제 row가 중복 INSERT되면 잔액이 잘못 부풀려진다.
> 아래 4.3의 "upsert(갱신)" 전략으로 **월당 퀘스트 면제 row를 정확히 1건**만 유지한다.

### 3.2.1 ⚠️ 재계산 면제 필터 수정 필수 — 당월 소급 면제 누락 버그

> **현재 `recalculateBalance`의 면제 합산은 "월 단위"라, 퀘스트 면제가 누락된다.**

기존 로직은 면제를 `aply_ym >= exmFromYm`(직전 기준일시 +1개월)으로만 필터한다
([recalculate-balance.ts](app/actions/dues/recalculate-balance.ts) `exmFromYm` / `.gte("aply_ym", ...)`).
입금은 시각(`last_calc_at`) 기준으로 거르는데 **면제만 월 단위**라서 다음 케이스가 샌다:

```
6/15 재계산 → 기준일시=6/15, 다음 재계산은 exmFromYm=7월부터 면제를 봄
6/30 퀘스트 면제 적립 (aply_ym='2026-06')
7/01 재계산 → aply_ym(6월) < exmFromYm(7월) → 6월 면제 SELECT에서 누락 → 영영 잔액 반영 안 됨 ❌
```

**수정**: 면제 SELECT 조건을 **기준일시(시각) 기준 합집합**으로 바꾼다.

```
WHERE aply_ym >= exmFromYm        -- 기존: 새 월 귀속 면제
   OR crt_at  >  last_calc_dt     -- 추가: 직전 재계산 이후 적립된 면제 (월 무관) ← 퀘스트 면제 포착
```

- 6/30 적립 6월 면제 → 첫 조건 탈락하지만 `crt_at(6/30) > last_calc_dt(6/15)` → **두 번째로 잡힘** ✅
- 이미 baseBal에 녹은 과거 면제 → 둘 다 탈락 → 이중 합산 없음 ✅
- **dedup 필수**: OR로 양쪽 조건에 동시 매칭되는 row가 생길 수 있으므로, 합산 전
  `exm_hist_id`로 중복 제거(입금 쪽 `filteredPays`와 동일한 패턴).

> 이 수정 없이는 §3.1.1의 `반영 예정` 뱃지가 **영원히 안 사라지는** 면제가 생긴다. **구현 필수 항목.**

### 3.3 `recalculateBalance`의 기존 자동 면제 로직과의 충돌 주의

현재 `recalculateBalance`는 `fee_due_exm_cfg`(관리자 면제 규칙)을 순회하며
`grant_src_enm='rule_attd'`로 `fee_due_exm_hist`를 INSERT한다.
**이 경로는 `exm_cfg_id`가 있는 규칙 기반**이고, 퀘스트 감면은 **`exm_cfg_id IS NULL`**로 구분된다.
→ 중복 INSERT 가드(`exm_cfg_id` eq 비교) 때문에 서로 간섭하지 않지만,
구현 시 **퀘스트 면제 row는 별도 키(`exm_cfg_id IS NULL AND grant_src_enm='rule_attd'`)로 식별**해야 한다.

> 안전을 위해 신규 enum 값 `'rule_attd_quest'`를 추가하는 방안도 고려(§7 대안 A). 본 설계는 `exm_cfg_id IS NULL`로 구분하는 것을 기본안으로 한다.

## 4. 집계 로직

### 4.1 월별 출석 통계 쿼리 (단일 멤버, 단일 월)

```sql
-- 입력: team_id, mem_id, ym_start(해당 월 1일 KST), ym_end(다음 달 1일 KST)
-- regular 참석 수, 본인 개설 수, 총 참석 수
SELECT
  -- 총 참석 (stt_at 기준 월 귀속)
  count(*) FILTER (WHERE a.mem_id = :mem_id)                               AS attend_cnt,
  -- 정모(regular) 참석
  count(*) FILTER (WHERE a.mem_id = :mem_id AND g.gthr_type_enm = 'regular') AS regular_attend_cnt,
  -- 본인 개설 (참석 여부 무관)
  count(DISTINCT g.gthr_id) FILTER (WHERE g.crt_by = :mem_id)             AS hosted_cnt
FROM gthr_mst g
LEFT JOIN gthr_attd_rel a ON a.gthr_id = g.gthr_id AND a.mem_id = :mem_id
WHERE g.team_id = :team_id
  AND g.del_yn = false
  AND g.stt_at >= :ym_start   -- KST 월 경계 (기존 fix_kst_date_boundary 헬퍼 패턴 사용)
  AND g.stt_at <  :ym_end;
```

> 실제 구현은 KST 경계 헬퍼(`20260627190000_rpcs_use_kst_helper.sql` 패턴)를 따른다.
> `hosted_cnt`는 `gthr_attd_rel` JOIN과 무관하게 `gthr_mst.crt_by`로만 센다 (개설=참석 자동이지만 게이트는 개설 사실 자체로 판정).

### 4.2 감면액 계산 (순수 함수, 서버/DB 공용 로직)

```typescript
function calcExemption(stats: {
  attendCnt: number;
  regularAttendCnt: number;
  hostedCnt: number;
}): { gatePassed: boolean; exmAmt: 0 | 1000 | 2000; reason: string } {
  const gatePassed = stats.regularAttendCnt >= 1 || stats.hostedCnt >= 1;
  if (!gatePassed) return { gatePassed: false, exmAmt: 0, reason: "게이트 미충족" };

  if (stats.attendCnt >= 8) return { gatePassed, exmAmt: 2000, reason: "출석 8회 전액 면제" };
  if (stats.attendCnt >= 4) return { gatePassed, exmAmt: 1000, reason: "출석 4회 감면" };
  return { gatePassed, exmAmt: 0, reason: "참석 횟수 부족" };
}
```

> 월 회비 단가가 2,000원이 아닐 수도 있으므로 실제 구현은 하드코딩 대신
> `monthly_fee_amt`를 받아 `exmAmt = min(계산값, monthly_fee_amt)`로 클램프한다.
> (1단계 = 회비의 50%, 2단계 = 전액 — 정책화 권장. §7 대안 B)

### 4.3 면제 이력 반영 (멱등 upsert)

같은 월에 대해 **퀘스트 면제 row를 1건만 유지**한다.

```
1. 해당 (team, mem, aply_ym)의 기존 퀘스트 면제 row 조회
   WHERE exm_cfg_id IS NULL AND grant_src_enm='rule_attd' AND aply_ym=:ym
2. 새 계산값 exmAmt 산출
3. 분기:
   - 기존 없음 + exmAmt>0  → INSERT
   - 기존 있음 + 금액 다름  → UPDATE (exm_amt, rsn_txt 갱신)
   - 기존 있음 + exmAmt=0   → soft delete (del_yn=true)  ※ 게이트 풀렸을 때(참석 취소 등)
   - 변화 없음             → no-op
```

> **부분 유니크 인덱스 추가 권장**:
> `CREATE UNIQUE INDEX uk_fee_exm_hist_quest ON fee_due_exm_hist(team_id, mem_id, aply_ym) WHERE exm_cfg_id IS NULL AND grant_src_enm='rule_attd' AND del_yn=false;`
> → 동시성·중복 INSERT를 DB 레벨에서 차단.

### 4.4 트리거 시점 (언제 면제 이력을 갱신하나)

> **확정**: 즉시 재계산(잔액 보정)은 하지 않는다. 면제 **이력(row)만** 갱신하고,
> 잔액은 다음 `recalculateBalance` 때 일괄 반영한다. 그 사이의 간극은 §3.1.1의
> **`반영 예정` 뱃지**로 회원에게 명확히 안내한다.

| 방식 | 설명 | 채택 |
|------|------|------|
| **A. 참석 토글 시 면제 row upsert** | `toggleGatheringAttendance` 성공 직후, 해당 모임 `stt_at` 월의 본인 면제 row를 upsert(§4.3). **잔액 재계산은 안 함.** | ✅ 기본 |
| B. 재계산 배치에 통합 | `recalculateBalance` 안에서 월 순회 시 출석 통계도 같이 계산해 면제 row 보정 | 보완 |
| C. cron 야간 배치 | 매일 새벽 전월/당월 면제 row 일괄 재집계(누락·backfill) | 정합성 백업용 |

- **A**로 실시간성 확보(토스트 + 면제 내역 즉시 노출) + **B/C**로 정합성 보정.
- A에서 면제 row를 갱신해도 **잔액(`fee_mem_bal_snap`)은 다음 `recalculateBalance` 때만 반영**된다.
  → 면제 내역은 **바로** 뜨지만(뱃지 `반영 예정`), 잔액 차감은 관리자/배치 정산 시점.
  → 토스트로 "이번 달 1,000원 감면 달성!" 알림 (§6).
- **재계산을 영영 안 돌리면 잔액에 반영 안 됨** → 월 1회 이상 정산(수동 or cron C) 필요. (§7 결정사항)

## 5. 회원 UI — 퀘스트 진행도

### 5.1 회비 내역 페이지 (`/profile/dues`)

기존 잔액 카드 아래 **"이번 달 출석 감면" 퀘스트 카드** 추가:

```
┌─────────────────────────────────────┐
│ 이번 달 출석 감면          6월       │
├─────────────────────────────────────┤
│ ① 정모 참석 or 벙 개설   ✅ 달성     │   ← 게이트
│    정모 1/1 · 개설 0                 │
│                                      │
│ ② 참석 횟수                          │
│    [■■■■■■□□]  6/8                   │   ← 진행 바
│    4회 1,000원 ✅  /  8회 전액 ◻      │
├─────────────────────────────────────┤
│ 💰 현재 감면: 1,000원                │
│ 8회까지 2번 더 참석하면 전액 면제!   │
└─────────────────────────────────────┘
```

- 게이트 미충족 시: "①을 먼저 달성해야 감면이 적용돼요" 안내, 참석 바는 회색 비활성 느낌.
- 컴포넌트: `components/profile/dues-quest-card.tsx` (신규)
- 데이터: `dues/page.tsx`에서 §4.1 쿼리(또는 RPC)로 당월 통계 조회 후 props 전달.
- 디자인 토큰 준수: `StatCard`/`CardItem`/타이포그래피 컴포넌트 사용, 진행 바는 `--success`.

### 5.2 면제 내역 표시 + `반영 예정` 뱃지

기존 `fee_due_exm_hist` → 히스토리 리스트에 이미 "면제"로 노출됨(§dues/page.tsx 104~112).
`rsn_txt`("출석 감면: 참석 8회")가 그대로 표시되어 **왜 감면됐는지** 회원이 알 수 있다.

여기에 §3.1.1의 시각 비교로 **반영 상태 뱃지**를 추가한다:

```
거래 내역                          정산 기준일 6.15 14:00
──────────────────────────────────────────────
6.30  면제  +1,000원   [반영 예정]   ← crt_at > last_calc_dt
       출석 감면: 참석 4회/정모 참석
6.10  면제  +1,000원                ← 뱃지 없음(이미 반영)
6.05  입금  +2,000원
```

- 판정: `dues/page.tsx`에서 각 내역의 시각 vs `snap.last_calc_dt` 비교 → `pending: boolean` 플래그를 아이템에 추가.
  - 면제: `exm.crt_at` 비교 (현재 select에 `crt_at` 추가 필요)
  - 입금/기타: `txn`/`pay`의 거래 시각 비교 (권장, §3.1.1 주석)
- 뱃지: `<Badge variant="...">반영 예정</Badge>` — `--info` 톤. 반영된 항목엔 뱃지 없음.
- 상단 잔액 카드 옆에도 보조 문구 가능: "정산 기준일 6.15 · 이후 내역은 다음 정산 시 반영".

## 6. 참석 토글 시 토스트 (퀘스트 피드백)

### 6.1 동작

`toggleGatheringAttendance`가 **참석 등록(attending=true)** 에 성공하면,
서버 액션이 갱신 후의 **당월 퀘스트 상태**를 함께 반환 → 클라이언트가 토스트로 표시.

```typescript
// toggle-attendance.ts 반환 확장
return {
  attending: true,
  quest?: {
    attendCnt: number;
    gatePassed: boolean;
    exmAmt: 0 | 1000 | 2000;
    justUnlocked?: "gate" | "tier1" | "tier2";  // 이번 토글로 새로 달성한 단계
  }
};
```

### 6.2 토스트 메시지 (단계 달성 순간만, "띵"하고)

| justUnlocked | 토스트 |
|--------------|--------|
| `gate` | "🔓 감면 조건 달성! 이제 4회 참석하면 1,000원 감면" |
| `tier1` | "🎉 출석 4회 달성 — 이번 달 1,000원 감면!" |
| `tier2` | "🏆 출석 8회 달성 — 이번 달 회비 전액 면제!" |
| (없음) | 일반 참석 토스트 "참석 완료 (이번 달 N회)" |

- 기존 토스트 인프라(sonner 등) 확인 후 재사용. 1.5~2초 자동 소멸.
- **`justUnlocked`는 "이번 토글로 처음 넘은 임계값"일 때만** 세팅 → 매번 시끄럽게 안 뜸.
- 참석 취소(attending=false) 시엔 토스트 없음(혹은 조용히). 면제 row는 §4.3대로 하향 조정.

### 6.3 모임 상세/홈 양쪽 적용

`GatheringAttendButton`(상세)과 홈 리스트뷰 참석 토글 모두 동일 액션을 쓰므로,
반환값의 `quest`를 받아 토스트를 띄우면 양쪽에서 동작.

## 7. 구현 시 결정 필요 사항 / 대안

- **대안 A — 신규 enum 값**: `fee_grant_src_enm`에 `'rule_attd_quest'` 추가해 관리자 규칙 면제와 명확히 분리.
  (장점: 식별 명확 / 단점: enum·기존 쿼리 마이그레이션) → **권장하나 본 설계 기본안은 `exm_cfg_id IS NULL`**.
- **대안 B — 감면액 정책화**: 1,000/2,000 하드코딩 대신 `fee_policy_cfg`에 퀘스트 단가 컬럼 추가
  (`quest_tier1_amt`, `quest_tier1_cnt`, `quest_tier2_cnt`). 회비 변동·정책 변경에 유연. → v2 권장.
- **과거 모임 backfill**: 제도 도입 전 달은 적용 X (당월부터). 도입 월 경계 명시 필요.
- **참석 취소로 인한 감면 회수**: 월 마감 전 참석을 취소해 4회 미만이 되면 면제도 회수(soft delete).
  월이 지난 과거 달은 동결(재계산 대상에서 제외)할지 정책 결정 필요.
