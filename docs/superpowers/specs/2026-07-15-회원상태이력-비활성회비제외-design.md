## 1. 배경 & 문제

회비 잔액은 "앵커(개시잔액) + 전체 리플레이" 방식으로 매번 처음부터 재계산된다 (`app/actions/dues/recalculate-balance.ts`, `lib/dues/ledger-replay.ts`). 현재 부과는 **`join_dt`(또는 앵커 다음 달)부터 오늘까지 매월 무조건** 걷는다 — 회원이 중간에 비활성/탈퇴했던 기간을 구분하지 못한다.

그 결과:

1. **비활성/탈퇴 기간 회비가 소급 부과된다.** 회원이 5개월 비활성이었다가 관리자가 재활성화하면, 다음 재계산 때 `join_dt`부터 오늘까지 전 구간이 부과되어 비활성 5개월치 회비가 미납으로 잡힌다. 그 기간엔 활동(`withActive` 게이트로 대회 기록·마일리지·프로필 수정 등이 막힘)조차 못 했는데도.
2. **상태 변경 이력이 없다.** 설계 문서(§4 vers 운용)는 회원/소속 변경 시 옛 정본을 `vers>0` 이력으로 보존하도록 규정하지만, 실제 코드(`app/actions/admin/manage-member.ts`)는 `vers=0` 로우를 **in-place UPDATE**만 한다. prd 확인 결과 `team_mem_rel`·`mem_mst`는 전부 `vers=0`(이력 0건)인 반면, `fee_mem_bal_snap`은 vers 이력을 정상 사용(759건). 즉 **회원 도메인만 이력 미구현.** 그래서 "언제부터 언제까지 비활성이었는지"를 시스템이 알 수 없다.
3. **삭제(`del_yn=true`)가 막다른 상태다.** 유니크 제약 `UNIQUE(team_id, mem_id, vers)` 때문에, 삭제된 회원이 재가입하면 `vers=0` INSERT가 충돌(23505)해 되살아나지 못한다.

## 2. 목표 & 범위

### 목표

- **회원 자격 변동(가입·비활성·재활성·탈퇴)이 있었던 달의 회비는 부과하지 않는다** — "온전한 달만 부과".
- **모든 상태/정보 변경을 vers 이력으로 남긴다** — 설계 원래 의도대로. `vers=0`이 현재 정본, 변경 시마다 옛 정본을 `vers=max+1` 이력으로 보존.
- **재활성 시 관리자가 잔액을 초기화(0원)하거나 보존할 수 있다.**
- **삭제된 회원이 재가입 가능하도록** vers를 밀어 `vers=0` 슬롯을 비운다.

### 이번 스펙 범위 (①②③ — 회비 정확성 축)

- ① vers 이력 인프라 (team_mem_rel + mem_mst, 원자적 RPC)
- ② 온전한 달 재계산 (이력 기반 부과 판정)
- ③ 재활성 UX (대상 확대, 잔액 초기화/보존, 앵커 확장)

### 범위 밖 (별도 스펙으로)

- ④ 활동 게이팅 확대 (모임 일정 생성·참가, 프로젝트 참가에 `withActive` 추가 — 현재 미적용)
- ⑤ "활성화 요청" 기능 (비active 회원 → 관리자 알림, 기존 건의/회비확인요청 패턴 재사용)
- ⑤-1 7월 당월 미부과 — **완료**(PR #422)

## 3. 핵심 규칙: "자격 변동 있는 달은 면제"

회비는 월 단위(쪼개기 없음)다. **그 달 1일 00:00 ~ 말일 24:00 동안 회원이 끊김 없이 `active` 소속이었던 달만 부과**한다. 그 달 안에 자격 변동 이벤트(가입/비활성/재활성/탈퇴/삭제)가 하나라도 있으면 그 달은 **면제**.

| 이벤트가 있던 달 | 처리 |
|---|---|
| 가입월 (컷오프 `JOIN_MONTH_EXEMPT_FROM=2026-07-01` 이후 가입자만) | 면제 (중간 진입) — 기존 `firstChargeMonth`가 담당(§6.1 참고). 컷오프 이전 가입자는 가입월도 부과(기존 회원 불변) |
| 비활성 시작월 | 면제 (중간 이탈) |
| 온전히 비활성/탈퇴인 달 | 면제 (내내 비active) |
| 재활성월 | 면제 (중간 복귀) |
| 탈퇴월 | 면제 (중간 이탈) |
| 변동 없이 내내 active인 달 | **부과** |

### 시나리오 검증 (월 2,000원, 5/12 비활성 → 11/20 재활성, 초기화 안 함, 비활성 직전 잔액 −4,000)

| 시점 | 그 달 | 결과 | 잔액 |
|---|---|---|---|
| 5월 | 비활성 시작 | 면제 | −4,000 |
| 6~10월 | 온전히 비활성 | 면제 | −4,000 (보존) |
| 11월 | 재활성 | 면제 | −4,000 |
| 12월~ | 내내 active | 부과 | −6,000(12월), −8,000(1월)… |

## 4. ① vers 이력 인프라

### 4.1 데이터 모델

`team_mem_rel`·`mem_mst`의 기존 스키마·유니크 제약(`UNIQUE(team_id, mem_id, vers)` / `mem_mst`도 유사)을 그대로 두고, **쓰기 패턴만** SCD Type-2로 바꾼다.

- `vers=0` = 현재 정본 (기존 조회 코드 전부 `.eq("vers", 0)` — 변경 불필요)
- `vers>0` = 과거 이력. **클수록 더 최근에 이력으로 밀려난 것**(1=최초로 밀린 가장 오래된 이력, 숫자가 클수록 최근). 변경마다 옛 정본을 `vers = (현재 최대 vers)+1`로 복사.
- **PK(`team_mem_id`)는 정본 로우에 고정**한다. 이력 복사본이 새 PK를 갖는다. → `team_mem_id`를 참조하는 유일한 FK인 `mem_ttl_rel`(칭호)이 항상 살아있는 정본을 가리켜 안전. (prd 확인: team_mem_id 참조 FK는 `mem_ttl_rel` 1개뿐)

> **이력 로우의 PK 방향 결정**: 정본(`vers=0`)의 `team_mem_id`를 **불변**으로 두고, 변경 직전 스냅샷을 *새* `team_mem_id`+`vers>0`로 INSERT한다. 반대로 하면(정본을 새 PK로) 칭호 FK가 매번 끊긴다.

### 4.2 이력 기록 시 담을 필드

각 이력 로우(및 정본)에 "이 상태가 **언제부터** 유효했는가"를 판정할 수 있는 시각이 필요하다(§5 온전한 달 판정의 입력).

- 기존 `crt_at`/`upd_at`(트리거 `team_mem_rel_set_upd_at`가 관리)에 더해, **효력 시작 시각**을 명시적으로 남긴다. 신규 컬럼 `eff_at timestamptz`(효력 발생 시각, KST 저장) 추가를 검토 — `upd_at`은 메모 수정 등으로 덮어써질 수 있어 상태 경계 기준으로 부정확(§8 과거 데이터에서 확인).
  - 대안: 별도 컬럼 없이 이력 로우의 `crt_at`(이력 생성 = 그 변경이 일어난 시각)을 경계로 사용. 단 과거 소급 이력(§8)은 추정 시각을 넣어야 하므로 `eff_at` 명시 컬럼이 더 명확. → **`eff_at` 추가 채택.**

### 4.3 원자적 이력화 RPC

Supabase JS는 여러 문장을 한 트랜잭션으로 못 묶으므로(회비 `recalc_member_balance` 선례와 동일), 상태/정보 변경을 **DB 함수로 원자화**한다.

```text
rpc: apply_team_mem_rel_change(p_team_mem_id, p_changes jsonb, p_eff_at)
  1) 현재 정본(vers=0) SELECT ... FOR UPDATE
  2) 그 스냅샷을 (새 team_mem_id, vers = max(vers)+1)로 INSERT  -- 이력 보존
  3) 정본(vers=0) 로우를 p_changes 로 UPDATE, eff_at = p_eff_at
```

- 적용 지점(전부 이 RPC 경유로 교체): `deactivateMember`, `reactivateMember`, `leaveMemberFromDues`, `rejectMember`, `approveMember`, `toggleAdmin`, `batchDeactivate/Reactivate`, **`deleteMember`(§4.4, 별도 RPC `apply_team_mem_rel_delete`)**. `onboardingCreateMember`(가입 = active 시작 이벤트)는 새 정본을 만드는 최초 INSERT라 이력화할 옛 정본이 없으므로 이 RPC를 경유하지 않고 `team_mem_rel` 직접 INSERT를 유지한다. **이번 스펙은 `team_mem_rel`만 이력화한다 — `mem_mst` 프로필 이력화는 범위 밖(§11 참고).**
- 동시성: 정본 `FOR UPDATE` 락으로 이력 순번(vers) 경합 방지.
- 권한: 두 RPC 모두 `REVOKE ALL FROM PUBLIC` 후 `service_role`에게만 `GRANT`(authenticated 권한 없음). `SECURITY DEFINER`이므로 내부에서 `auth.uid()`가 있는 세션(향후 GRANT 확대 대비 심층 방어)이면 `v2_rls_auth_team_owner_or_admin`으로 팀 owner/admin 여부를 재검증한다.

### 4.4 삭제 = vers 밀기

`deleteMember`를 "정본을 이력으로 밀고 `vers=0` 슬롯을 비우는" 방식으로 변경한다. 전용 RPC `apply_team_mem_rel_delete`는 두 단계로 처리한다:

1. **삭제 직전 상태를 먼저 이력(`vers=max+1`)으로 보존** (change RPC와 같은 복사 패턴, `eff_at`은 그대로 유지) — 그래야 "마지막 상태 전환~삭제" 사이의 active 구간이 타임라인에서 사라지지 않는다.
2. 그다음 정본(`vers=0`)을 `vers=max+2`(1단계 이후 새 max+1), `del_yn=true`, `eff_at=삭제 시각`으로 민다 → `(team_id, mem_id, vers=0)` 슬롯이 비므로 재가입 시 새 정본 INSERT 가능.

- 재가입 경로(`onboardingCreateMember`)는 이미 있는 `vers=0` 정본이 없으므로 정상 INSERT. 과거 회비 잔액이 딸려오지 않도록 재가입 성립 시 `sealBalanceAnchor`로 0원 앵커를 봉인한다(§7-C, §12 참조).

## 5. ② 온전한 달 재계산

### 5.1 상태 타임라인 재구성

재계산 시 회원의 `team_mem_rel` 이력(vers 0..N)을 `eff_at` 기준 시간순으로 정렬해 **active 구간 목록** `activeIntervals: {start, end}[]`을 만든다. (현재 정본이 active면 마지막 구간의 end는 열림/무한.)

### 5.2 부과 월 판정

`buildChargeMonths`에 `activeIntervals`(또는 "이 달 부과 여부" 판정 함수)를 주입한다. 각 부과 후보 월 `M`에 대해:

- `M`의 [1일 00:00, 익월 1일 00:00) **전체 구간이 하나의 active 구간에 완전히 포함**되면 → 부과.
- `M` 안에 상태 경계(구간 시작/끝)가 걸치거나, `M`을 덮는 구간이 비active면 → **면제**(부과 목록에서 제외).

> 순수 함수로 추출: `isFullyActiveMonth(activeIntervals, ym): boolean`. 단위 테스트 용이. `buildChargeMonths`는 이 판정이 false인 달을 `charged`에서 뺀다.

### 5.3 가입월(`firstChargeMonth`)과의 관계

- **가입월 면제는 기존 `firstChargeMonth`(PR #422)가 계속 담당**한다. 컷오프(`JOIN_MONTH_EXEMPT_FROM=2026-07-01`) 이전 가입자는 가입월도 부과해야 하므로(소급 안전), 이 규칙을 상태 타임라인 규칙으로 대체하면 **기존 잔액이 소급으로 틀어진다.** 따라서 두 규칙은 공존:
  - 부과 **시작월** = `firstChargeMonth(join_dt)` (컷오프 규칙 유지)
  - 그 이후 각 달의 **부과 여부** = `isFullyActiveMonth`(비활성/재활성/탈퇴 반영)
- 즉 §3 표의 "가입월 면제"는 기존 로직이, 나머지(비활성/재활성/탈퇴)는 §5.2가 담당. 신규 규칙은 **앞으로 발생하는 상태 변동에만** 적용되어 과거 소급 위험이 없다.

### 5.4 이력 없는 기존 회원

현재 active 회원은 상태 이력이 없다(가입 이벤트만). `activeIntervals`가 "가입~현재 계속 active" 단일 구간이 되어 **기존 동작과 동일**(전 구간 부과) → 소급 영향 0. 이력은 앞으로의 상태 변동부터 쌓인다.

## 6. ③ 재활성 UX

### 6.1 대상 확대

`reactivateMember`(및 batch)를 `inactive` + `left` 대상으로 확대(현재 `inactive`만). `pending`은 재활성이 아니라 승인(`approveMember`) 흐름이므로 제외. 삭제(`del_yn`)도 제외.

### 6.2 비활성 구간 확인 → 이력 소급 생성

재활성 시 관리자에게 비활성 구간을 제시하고 확인받는다.

- **이력이 있으면**(신규 시스템 도입 후 비활성된 회원): 이력 그대로 사용.
- **이력이 없으면**(도입 전 비활성된 기존 회원): `upd_at`으로 **비활성 시작 시각을 추정**해 초기값 제시 → 관리자가 확인·수정 → 확정 시 **비활성 이벤트 + 재활성 이벤트를 vers 이력으로 소급 생성**(§4.3 RPC). "원래 있었어야 할 이력을 지금 채우는" 것.

### 6.3 잔액 초기화 vs 보존

재활성 확정 시 관리자가 선택:

- **보존**(기본): 앵커 없이 리플레이 그대로. 비활성 기간 부과만 §5.2로 빠지고, 기존 잔액(예치금·미납)은 자연 보존. → 미납을 안고 나갔다가 재가입해 털어내는 악용 방지.
- **초기화(0원)**: 재활성 시점에 **개시잔액 0 앵커 스냅샷**을 만들고, 그 이후부터 부과 시작. → 예치금/미납이 모두 0에서 재시작. `left`(회비 미납 강제 탈퇴)의 미납 탕감도 이 옵션으로 처리.

### 6.4 앵커 인정 로직 확장

현재 재계산은 `crt_at < LEDGER_EPOCH(2026-07-02)` 스냅샷만 앵커로 인정한다. 재활성 초기화 스냅샷은 EPOCH 이후라 그대로면 앵커로 안 잡혀 다음 재계산이 `join_dt`부터 다시 계산해 **초기화가 풀린다.**

- 해결: `fee_mem_bal_snap`에 **명시적 앵커 플래그** `anchor_yn boolean`(기본 false) 추가. 앵커 판정을 "`crt_at < EPOCH` **또는** `anchor_yn=true`"로 확장. 재활성 초기화 스냅샷은 `anchor_yn=true`, `bal_amt=0`, `last_calc_dt`=재활성월 초로 생성.
- 재계산 `fromMonth` 결정: 앵커가 있으면 앵커 마감월 다음 달부터(기존 분기 그대로) → 초기화 앵커면 재활성월부터 부과.

## 7. 함정 & 결정 (리뷰에서 도출)

- **A. 원자성 경계**: `apply_team_mem_rel_change`/`apply_team_mem_rel_delete` RPC가 원자화하는 범위는 "상태 변경 + 이력 복사"까지다. 재활성 시 앵커 봉인(`sealBalanceAnchor`)과 잔액 재계산(`recalculateBalance`)은 RPC 밖의 **별도 서버 액션에서 순차 실행**되며 한 트랜잭션으로 묶이지 않는다. 상태 변경 RPC가 성공한 뒤 앵커/재계산이 실패해도 상태 이력 자체는 정합성을 유지하고(이력·정본이 어긋나지 않음), 재계산은 멱등이라 다음 재계산 실행(수동 재시도 또는 배치)이 자가 치유한다 — 부분 실패가 영구 데이터 오류로 남지 않는다.
- **B. 앵커 함정**: §6.4 — 초기화를 단순 스냅샷으로 만들면 리플레이가 되돌린다. `anchor_yn` 필수.
- **C. 삭제 후 재가입 시 회비 딸려옴**: 회비 이력(`fee_*`)은 `team_id+mem_id`로 연결되어 `mem_id`가 같으면 과거 잔액이 살아난다. → 재가입은 "새 시작"이므로 재가입 완료 시 **초기화 앵커(§6.4)** 를 세팅해 과거 회비를 0으로 봉인(재활성 초기화와 동일 메커니즘 재사용).
- **D. `max(vers)+1`**: 단순 "+1"이 아니라 현재 최대 vers 조회 후 +1(이력이 여러 개 쌓인 회원 대비). RPC 안에서 처리.
- **E. `firstChargeMonth`와 소급**: §5.3 — 신규 규칙은 상태 변동에만, 가입월은 기존 규칙. 과거 잔액 불변.

## 8. 과거 데이터 처리

- **삭제된 3명**(prd: `del_yn=true, vers=0` 3건): 마이그레이션으로 `vers=1`로 밀어 `vers=0` 슬롯 정리(재가입 대비). 소규모.
- **기존 inactive/left 86명**: 상태 이력 없음 → **전수 백필하지 않는다.** 이들은 비활성인 동안 재계산 대상(active)이 아니라 잔액이 얼어붙어 영향 없음. 복귀(재활성)하는 극소수만 §6.2로 그 시점에 추정·확인 처리.
  - `upd_at` 신뢰도(prd): left 15명 전원·inactive "탈퇴" 60일+ 32명은 upd_at≈이탈시점으로 쓸만. 메모 편집 흔적 5명은 부정확 가능 → §6.2의 관리자 확인 단계가 보정.
- **기존 active 회원**: 이력 없음(가입 이벤트만) → §5.4로 기존 동작 유지, 소급 0.

## 9. 마이그레이션

1. `fee_mem_bal_snap.anchor_yn boolean NOT NULL DEFAULT false` 추가.
2. `team_mem_rel.eff_at timestamptz` (+ `mem_mst.eff_at`) 추가. 기존 로우는 `crt_at`으로 백필.
3. `apply_team_mem_rel_change` (+ mem_mst용) RPC 생성.
4. 삭제 3명 `vers` 밀기 데이터 마이그레이션.
5. (앱) manage-member·onboarding·update-profile을 RPC 경유로 교체, 재계산에 `activeIntervals` 주입, 재활성 UX 추가.

## 10. 테스트 계획

- **순수 함수**: `isFullyActiveMonth(intervals, ym)` — 월 중간 비활성/재활성/탈퇴, 월초·월말 경계, 여러 번 비활성/재활성, 이력 없는 회원(단일 구간), 가입월과의 상호작용.
- **재계산 통합**: §3 시나리오(−4,000 → 비활성 6개월 → 재활성, 보존/초기화 각각) 잔액 검증.
- **이력 RPC**: 상태 변경 시 vers 순번·정본 PK 불변·칭호 FK 무결성·동시성(FOR UPDATE).
- **소급 안전**: 기존 active/비활성 회원 재계산 후 잔액 불변(diff=0) — 배포 후 prd 대조.
- **삭제→재가입**: vers 밀기 후 재가입 INSERT 성공 + 과거 회비 봉인(초기화 앵커).

### 다관점 리뷰 반영 (수정 완료)
- **[HIGH] 레거시 비활성 소급 부과**: `buildActiveIntervals`의 "첫 active 열기"를 `i===0`(첫 세그먼트가 active일 때만)으로 수정 → 레거시(정본 inactive) 재활성 시 비활성 기간 부과 방지.
- **[MED] fail-unsafe**: `relHist` 조회 실패/빈 결과 시 `[]`(전부 면제)가 아니라 `undefined`(전부 부과)로 처리 + `del_yn=false` 필터(삭제 이력 제외).
- **[초기화 버그]** 재활성 초기화 스냅샷을 `del_yn`+INSERT(유니크 충돌)에서 **vers=0 UPDATE**로 수정.

### 미완 — 후속 필요
- **§6.2 레거시 비활성 구간 정확 재구성 미구현**: 도입 전 비활성된 86명은 `active→inactive` 전이 이력이 없어, 재활성 시 "가입~비활성" 활동 기간의 정확한 부과를 못 한다. `buildActiveIntervals`는 이를 안전 방향으로 처리한다 — **첫 세그먼트가 inactive/left(=레거시)면 그 뒤 재활성 active는 실제 `eff_at`부터만 구간을 연다**(§4.1 코드 주석 참고). 즉 비활성 시작 시각을 모르는 레거시는 "재활성 이후만 부과", 그 이전 비활성 기간 전체가 자동 면제된다 — 잔액 보존을 택해도 그 전제(비활성 기간은 절대 부과되지 않음) 위에서는 안전하다. 다만 **잔액 보존을 택하면 과거 부과-납부 불균형(도입 전 비활성 기간에 이미 발생했던 미납 등)으로 잔액이 왜곡될 수 있음** → 레거시 재활성은 잔액 **초기화** 권장. 비활성 시작일을 정확히 소급 입력해 그 구간의 부과 여부까지 재구성하려면 재활성 UI에 비활성 시작일 입력(upd_at 추정+관리자 확인) 기능이 별도로 필요(미구현).
- **[LOW] left UI 미배선**: 서버는 left 재활성 지원하나 admin-members-client의 활성화 버튼·배지가 inactive만 처리. left 회원 재활성 UI 필요.
- **[해소됨] 삭제 후 재가입 봉인**: `onboardingCreateMember`(`app/actions/onboarding-mem-v2.ts`)가 재가입 시(과거 `fee_mem_bal_snap` 존재 확인) `sealBalanceAnchor`(`lib/dues/seal-anchor.ts`)로 0원 앵커를 봉인하도록 구현 완료 — 과거 fee_* 잔액이 재가입 회원에게 부활하지 않는다. 실패해도 가입 자체는 성립(비치명, 로깅만).
- **mem_mst 프로필 이력**: PK=mem_id 단일이라 vers 이력 불가 → 별도 스펙(이력 테이블). 이번 스펙은 `team_mem_rel`만 이력화한다(§4.3 참고).
