## 1. 배경 / 문제의식

`기강이야기` 탭은 "멤버들의 기록과 이야기"를 담는 자리로 열려 있으나 아직 빈 placeholder다. 크루의 소식(신규 가입·대회 출전·최근 기록·활동 랭킹)을 한 화면에서 보고 **가볍게 반응(응원/장난)하며 즐기는** 참여형 페이지로 만든다.

### 초기 안(뉴스페이퍼 피드)을 폐기한 이유

처음엔 모든 소식을 시간순 기사 카드로 쌓는 "크루 신문" 안을 검토했으나 폐기했다. 진단:

- 넣으려는 소식들(신규 가입, 대회 출전, 최근 기록, 월간 랭킹)은 **자주 발생하지 않고, 건당 정보량이 작은 "감정 이벤트"**다. 기사 카드에 담으면 ① 한 컬럼에 뭉치고 ② 별거 아닌 걸 대단한 뉴스처럼 부풀린다 (그릇이 내용물보다 큼).
- 반대로 증권사 전광판처럼 데이터만 흘리면 감정이 빠져 차갑고 재미없다.

### 핵심 관점: "기사"가 아니라 "감정 이벤트"

| 콘텐츠 | 진짜 목적 | 감정 |
|---|---|---|
| 신규 가입 | 환영 | 👋 반갑다 |
| 다가오는 대회(기강인 참여) | 응원 | 📣 가서 이겨라 |
| 최근 기록 | 축하·자랑 | ⚡ 해냈다 |
| 월간 활동 랭킹 | 경쟁·재미 | 🏆 누가 위냐 |

## 2. 컨셉: 기강 전광판 (스타디움 점보트론)

증권 티커 ❌ 가 아니라 **스타디움 점보트론** ⭕ 을 지향한다. 얼굴을 크게, 축하·응원을 담아, 움직임과 관중 반응이 있는 보드.

### 설계 3원칙

1. **감정별로 존(zone)을 나눈다** — 한 피드에 안 뭉치게, 목적마다 다른 그릇.
2. **정보 크기에 맞는 그릇** — 기록은 작은 리스트로, 큰 카드 남발 금지.
3. **재미는 "스포트라이트 + 리액션 + 모션"에서** — 단순 출력이 아니라 반응.

## 3. 화면 구성

상단 **스포트라이트(전광판)** 가 재미를 담당하고, 아래 **컴팩트 존**들이 정보를 목적별로 작게 흘린다.

### 3.1 스포트라이트 (맨 위, 스와이프 카드)

- 맨 위에 **스와이프 가능한 카드 목록**. 각 카드는 **주인공의 프로필 카드 + 이유 배너**.
- "지금 크루가 다 같이 봐야 할 순간"만 큐레이션해서 하나씩 크게 돌린다. 자잘한 건 안 올라오고 아래 존으로 간다.
- 자동 전환 + 스와이프 + 전환 도트.

**로테이션 세트**

| 이유 배너 | 언제 뜨나 | 헤드라인 예시 | 리액션 |
|---|---|---|---|
| 👋 환영 | 신규 가입 | "강남역 사는 **김기강**, 오늘 기강 합류!" | 👏 환영 |
| 🔥 속보 | PB·완주·입상 | "**이기강**, 하프 개인최고 **1:32:10**" | 🔥 대박 |
| 📣 응원 | 대회 임박(D-7 이내) | "이번 주말 춘천마라톤, 기강인 **5명** 출격" | 💪 응원 |
| 👑 이달의 기강 | 월간(상시 채움) | "이달 열심히 달린 **최기강**" | 👏 짝짝 |

**카드 한 장 구성(anatomy)**

```
┌───────────────────────────┐
│ 🔥 속보              ● ○ ○  │  이유 배너 + 전환 도트
│      [ 큰 아바타 ]          │  얼굴이 주인공 (응원은 아바타 스택)
│   이기강, 하프 개인최고      │  명조 헤드라인 한 줄
│        1:32:10            │  큰 숫자 / D-day / 역
│   💪 대박 12               │  감정 맞춤 리액션 1개 + 카운트
└───────────────────────────┘
   자동 전환 · 스와이프 · 탭 → 프로필
```

**선별·운영 규칙**

- **선별 범위**: 최근 7일 이내 + 유의미한 것(신규·PB·완주·임박대회)만. 최대 5~6장.
- **우선순위**: 시간 민감 우선(대회 D-day 임박 > 신규 > 큰 기록), 나머지 최신순.
- **이달의 기강은 5위 내 랜덤** — 똑같은 1위만 반복되지 않게. (프로젝트 "한마디"의 `RandomReview` 랜덤 로테이션 패턴 선례 있음.)
- **빈 상태 폴백**: 하이라이트가 없으면 크루 슬로건("No time to be weak", 홈 헤더 문구)으로 대체. 빈칸 금지.

### 3.2 컴팩트 존 (스포트라이트 아래)

| 존 | 내용 | 소스 |
|---|---|---|
| 🎉 새 얼굴 | 신규 가입 아바타 가로 스크롤, 탭 → 환영 | `mem_mst`(가입일) + `mem_onbd_prf` |

## 4. 참여 모델

### 4.1 리액션 (1차) — 신규 테이블 `rctn_mst`

범용(polymorphic) 리액션. 기존 `cmnt_mst`의 `entity_type`/`entity_id` 패턴을 그대로 답습.

```
rctn_mst
  rctn_id     uuid pk
  team_id     uuid
  entity_type text     -- 스포트라이트/존 아이템 종류 (record | newbie | race_entry | ranking ...)
  entity_id   text
  mem_id      uuid
  rctn_cd     text     -- cheer | welcome | fire | clap | boo
  crt_at      timestamptz default now()
  unique (entity_type, entity_id, mem_id)   -- 1인 1리액션, 교체 가능(upsert)
```

- 리액션 세트(감정별): 👏 환영(welcome) · 🔥 대박(fire) · 💪 응원(cheer) · 😂 ㅋㅋ(lol) · 😈 야유(boo).
- RLS: 팀 멤버만 자기 `mem_id`로 insert/update/delete, 팀 공개 read.
- **기강 포인트 트리거 주의**: `rctn_mst`에는 포인트 적립 트리거를 붙이지 않는다(리액션은 적립 대상 아님).

### 4.2 댓글 (후속) — 기존 `cmnt_mst` 재사용

- `entity_type` = 아이템 종류, `entity_id` = 아이템 id, `prnt_id` = 대댓글, 멘션 = `cmnt_mention_rel`.
- 알림 연동(`insertNoti`)은 댓글 도입 시 함께.

## 5. 데이터 / 백엔드

- **집계 RPC 신설**: `get_team_story_feed(team_id, cursor, limit)` — 스포트라이트/존 소스를 DB에서 union·정렬·페이징. 홈의 `get_public_team_*` 패턴과 동일. 각 아이템에 리액션 카운트 + (로그인 시)내 리액션을 조인.
- **오버레이 방식**: 공개 데이터 목록은 캐시 가능, 리액션 카운트·내 리액션은 동적 오버레이 — 홈의 `is_attending` 클라이언트 오버레이 패턴 그대로.
- **서버/클라 패턴**: `page.tsx`(서버, 데이터 조회) → 클라이언트 피드 컴포넌트(스와이프·리액션 상호작용). `getCurrentMember()`로 로그인/멤버 판별.
- **날짜**: `@/lib/dayjs`만 사용, KST 기준. 문자열 조작 금지.

## 6. 의존성: 멤버 프로필 카드 (별도 설계)

> 이 카드의 **세부 구성은 별도 문서에서 설계**한다(작성자 진행). 여기서는 전광판이 무엇에 의존하는지만 명시.

- 스포트라이트 카드 = **재사용 가능한 멤버 프로필 카드 + 이유 배너**. 즉 프로필 카드가 전광판의 원자 단위이자 **선행 의존성**이다.
- 기존 [components/profile/profile-card.tsx](../../../components/profile/profile-card.tsx)는 "내 컬렉션" 편집이 달린 **소유자용 자기 카드**라 그대로 못 쓴다 → 남을 보여주는 **쇼케이스 카드를 신규 제작**.
- 데이터: **`get_public_member_card(mem_id, team_id)` RPC가 이미 DB에 존재하나 현재 미사용** → 이걸 살려 쓴다.
- 확장 예정(별도 설계에서 확정): **"한마디(자기소개)"** — 멤버가 직접 입력하는 한 줄. 저장 필드 신설 + 프로필 편집 입력란 + 카드 표시가 필요(마일리지 후기의 "한마디"와는 별개 개념). 각오/공약도 이 카드에 붙여 확장 가능.

## 7. 디자인 방향

디자인 시스템([DESIGN.md](../../../DESIGN.md))에서 크게 벗어나지 않되, 개성은 **폰트와 코너 톤 한 곳**에 몰아준다. 톤 온도는 "적당히".

- **Signature = 명조 헤드라인**: 헤드라인만 제목 폰트(Nanum Myeongjo)로 → 폰트 자체가 "전광판/제호" 느낌을 내 별도 장식 없이 개성이 난다. 본문·메타는 Pretendard.
- **색**: CSS 변수 토큰만. 종목 기록엔 `sport-*` 악센트. 하드코딩 색 금지.
- **타이포**: 매직넘버 금지 — `H2`/`Body`/`Caption`/`Micro`/`SectionLabel` 사용.
- **레이아웃**: `PageHeader`(기강이야기) + eyebrow. 스포트라이트(스와이프) → 컴팩트 존 세로 스택. 카드는 `CardItem`.
- **모션**: 스포트라이트 자동 전환/스와이프, 리액션 탭 피드백. 과함 금지(`prefers-reduced-motion` 존중).

## 8. 제약 / 주의사항

1. **기강 포인트는 히든 운영** — UI·유저에게 "기강포인트" 표기·원장 노출 **금지**. 🏆 순위 존은 **유저에게 보이는 활동 지표**(예: 이달 거리·출석·활동 수)로만 표현한다. 랭킹 노출 지표는 구현 전 확정 필요(11절).
2. **참가 기록 이름 보존** — 회원 탈퇴·삭제 후에도 과거 기록의 이름은 남아야 한다. `mem_mst` 삭제·이름 덮어쓰기 금지 원칙 준수.
3. **프라이버시** — 소스는 팀 공개 범위(`get_public_*`)만 사용. 각오/가입목적 텍스트 공개 여부는 미정(11절).

## 9. 범위 / 단계

- **1차**: 멤버 프로필 카드(쇼케이스) — *별도 설계 후* 구현. 전광판의 선행 재료.
- **2차**: 기강 전광판 — 스포트라이트(4종 로테이션, 5위 랜덤) + 컴팩트 존 4개 + `get_team_story_feed` RPC + **리액션**(`rctn_mst`).
- **3차**: 댓글(`cmnt_mst` 재사용) + 알림 연동 + 각오/공약 카드 확장.
- **나중**: 대회 후기(기록 입력에 후기 기능 생기면) · 마일리지 "한마디" 이전.

## 11. 열린 결정 (미확정)

1. **🏆 순위 노출 지표** — "기강포인트"는 노출 금지이므로, 유저에게 보이는 어떤 활동 지표로 랭킹을 매길지(거리/출석/활동 수 등) 확정 필요.
2. **각오/가입목적 공개 여부** — 보류. 공개로 결정되면 스포트라이트·프로필 카드에 인용으로 추가.
3. **멤버 프로필 카드 세부 구성** — 별도 설계 문서에서 확정(작성자 진행).
