## 1. 배경 / 문제의식

`기강이야기` 탭은 "멤버들의 기록과 이야기"를 담는 자리로 열려 있으나 아직 빈 placeholder다. 크루의 소식(신규 가입·대회 출전·최근 기록·활동 랭킹)을 한 화면에서 보고 **가볍게 반응(응원/장난)하며 즐기는** 참여형 페이지로 만든다.

### 초기 안(뉴스페이퍼 피드)을 폐기한 이유

처음엔 모든 소식을 시간순 기사 카드로 쌓는 "크루 신문" 안을 검토했으나 폐기했다. 진단:

- 넣으려는 소식들(신규 가입, 대회 출전, 최근 기록, 월간 랭킹)은 **자주 발생하지 않고, 건당 정보량이 작은 "감정 이벤트"**다. 기사 카드에 담으면 ① 한 컬럼에 뭉치고 ② 별거 아닌 걸 대단한 뉴스처럼 부풀린다 (그릇이 내용물보다 큼).
- 반대로 증권사 전광판처럼 데이터만 흘리면 감정이 빠져 차갑고 재미없다.

### 핵심 관점: "기사"가 아니라 "감정 이벤트"

| 콘텐츠 | 진짜 목적 | 감정 |
|---|---|---|
| 신규 가입 | 환영 | 👋 반갑다 |
| 다가오는 대회(기강인 참여) | 응원 | 📣 가서 이겨라 |
| 최근 기록 | 축하·자랑 | ⚡ 해냈다 |
| 월간 활동 랭킹 | 경쟁·재미 | 🏆 누가 위냐 |

## 2. 컨셉: 기강 전광판 (스타디움 점보트론)

증권 티커 ❌ 가 아니라 **스타디움 점보트론** ⭕ 을 지향한다. 얼굴을 크게, 축하·응원을 담아, 움직임과 관중 반응이 있는 보드.

### 설계 3원칙

1. **감정별로 존(zone)을 나눈다** — 한 피드에 안 뭉치게, 목적마다 다른 그릇.
2. **정보 크기에 맞는 그릇** — 기록은 작은 리스트로, 큰 카드 남발 금지.
3. **재미는 "스포트라이트 + 리액션 + 모션"에서** — 단순 출력이 아니라 반응.

## 3. 화면 구성

상단 **스포트라이트(전광판)** 가 재미를 담당하고, 아래 **컴팩트 존**들이 정보를 목적별로 작게 흘린다.

### 3.1 스포트라이트 (맨 위, 스와이프 카드)

- 맨 위에 **스와이프 가능한 카드 목록**. 각 카드는 **주인공의 프로필 카드 + 이유 배너**.
- "지금 크루가 다 같이 봐야 할 순간"만 큐레이션해서 하나씩 크게 돌린다. 자잘한 건 안 올라오고 아래 존으로 간다.
- 자동 전환 + 스와이프 + 전환 도트.

**로테이션 세트**

| 이유 배너 | 언제 뜨나 | 헤드라인 예시 | 리액션 |
|---|---|---|---|
| 👋 환영 | 신규 가입 | "오늘 합류한 **김기강**, 환영해요!" | 👏 환영 |
| 🔥 속보 | PB·완주·입상 | "**이기강**, 하프 개인최고 **1:32:10**" | 🔥 대박 |
| 📣 응원 | 대회 임박(D-7 이내) | "이번 주말 춘천마라톤, 기강인 **5명** 출격" | 💪 응원 |
| 👑 이달의 기강 | 월간(상시 채움) | "이달 열심히 달린 **최기강**" | 👏 짝짝 |

**카드 한 장 구성(anatomy)**

```
┌───────────────────────────┐
│ 🔥 속보              ● ○ ○  │  이유 배너 + 전환 도트
│      [ 큰 아바타 ]          │  얼굴이 주인공 (응원은 아바타 스택)
│   이기강, 하프 개인최고      │  명조 헤드라인 한 줄
│        1:32:10            │  큰 숫자 / D-day / 역
│   💪 대박 12               │  감정 맞춤 리액션 1개 + 카운트
└───────────────────────────┘
   자동 전환 · 스와이프 · 탭 → 프로필
```

**선별·운영 규칙**

- **선별 범위**: 최근 7일 이내 + 유의미한 것(신규·PB·완주·임박대회)만. 최대 5~6장.
- **우선순위**: 시간 민감 우선(대회 D-day 임박 > 신규 > 큰 기록), 나머지 최신순.
- **이달의 기강은 5위 내 랜덤** — 똑같은 1위만 반복되지 않게. (프로젝트 "한마디"의 `RandomReview` 랜덤 로테이션 패턴 선례 있음.)
- **빈 상태 폴백**: 하이라이트가 없으면 크루 슬로건("No time to be weak", 홈 헤더 문구)으로 대체. 빈칸 금지.

| 존 | 내용 | 소스 |

| 🎉 새 얼굴 | 신규 가입 아바타 가로 스크롤, 탭 → 환영 | 공개 허용 필드만(이름·아바타·가입일), 공개 프로젝션 RPC 경유. 각오·가입목적은 공개 가능(11절), 거주지 등 위치는 제외 |

## 4. 참여 모델

### 4.1 리액션 (1차) — 신규 테이블 `rctn_mst`

범용(polymorphic) 리액션. 기존 `cmnt_mst`의 `entity_type`/`entity_id` 패턴을 그대로 답습.

```
rctn_mst
  rctn_id     uuid pk
  team_id     uuid
  entity_type text     -- 스포트라이트/존 아이템 종류 (record | newbie | race_entry | ranking ...)
  entity_id   text
  mem_id      uuid
  rctn_cd     text     -- welcome | fire | cheer | clap | lol | boo (정본 6종 — 아래 리액션 세트와 일치)
  crt_at      timestamptz default now()
  unique (team_id, entity_type, entity_id, mem_id)   -- 1인 1리액션, 교체 가능(upsert). team_id 포함 — entity_id가 팀 간 재사용될 수 있어(예: ranking 기간키) 팀별로 분리
```

- 리액션 세트(정본 6종, 감정별): 👏 환영(welcome) · 🔥 대박(fire) · 💪 응원(cheer) · 👏 짝짝(clap) · 😂 ㅋㅋ(lol) · 😈 야유(boo). 스키마 `rctn_cd` 주석·§3.1 로테이션 표·이 목록이 동일한 6개 값을 쓰며, 이 목록이 유일한 정본이다.
- RLS: 팀 멤버만 자기 `mem_id`로 insert/update/delete, 팀 공개 read.
- **기강 포인트 트리거 주의**: `rctn_mst`에는 포인트 적립 트리거를 붙이지 않는다(리액션은 적립 대상 아님).

### 4.2 댓글 (후속) — 기존 `cmnt_mst` 재사용

- `entity_type` = 아이템 종류, `entity_id` = 아이템 id, `prnt_id` = 대댓글, 멘션 = `cmnt_mention_rel`.
- 알림 연동(`insertNoti`)은 댓글 도입 시 함께.

### 4.3 라이브 프레즌스 — "관중석" (후속, 단계 출시)

> 2026-07-22 리스크 검증 워크플로 결론: **"물리 놀이터"가 아니라 "온기 신호"로 축소해서 시작할 때만
> 가치가 있다** — 조건부 승인. 아래 원칙들이 그 조건이며, 하나라도 깨지면 기능을 강등한다.

전광판이 있는 스타디움에 **관중석**이 생긴다 — 지금 `/story`를 보고 있는 멤버의 아바타가
**탭바 위 고정 높이(64~80px) 전용 밴드**에 나타난다. 아바타는 콘텐츠 위로 절대 떠다니지 않는다
(스포트라이트·존 가림과 오탭 방지 — 밴드 밖 pointer-events 차단, 스크롤 중 애니메이션 일시정지).

**단계**

| Phase | 내용 | 착수 조건 |
|---|---|---|
| **A. 프레즌스 칩** | 아바타 스택 + "○○님 외 2명이 함께 보는 중" 정적 칩. Supabase Realtime **Presence** | 전광판 본체(1~2차) 출시 후 |
| **B. 찌르기 + 리액션 파도** | ① 아바타 탭 → 튕김(양쪽 화면에서 보임, broadcast 1건). ② 스포트라이트 리액션 탭 → **같은 페이지를 보고 있는 사람 화면에 이모지 파티클이 떠오른다**(휘발, 저장 없음 — `rctn_mst` insert와 별개의 채널 이벤트) | A 실사용 확인 후 |
| **C. 자율 산책** | 상시 구르기 ❌ — "가끔 산책 + 숨쉬기(idle bob)" 톤. 공유 seed 결정론 로컬 시뮬레이션 — 위치 동기화 없음 | A·B의 관찰된 사용량으로 자격 획득 후 |

**빈 방(N=0~1)이 기본 상태다** — 동접 0~3명이 일상이므로 라이브 프레즌스는 겹쳤을 때의 보너스이지 전제가 아니다.

- **발자국**: 최근 24시간 다녀간 멤버를 반투명 고스트 아바타로 표시 — 방이 비어 보이지 않게.
- **마스코트 1기 상주** — 항상 누군가는 관중석에 있다.
- "0명" 카운트는 절대 표시하지 않는다.

**찌르기 4원칙** (실명 크루에서 표적 행동·괴롭힘 전이를 구조적으로 차단)

1. **익명** — 누가 쳤는지 표시하지 않는다.
2. **휘발** — 로그·히스토리를 남기지 않는다. 양쪽이 동시에 페이지에 있을 때만 보인다.
3. **무알림** — 푸시·인앱 알림을 보내지 않는다 (오프라인에 맞은 걸 나중에 알게 되는 설계 금지).
4. **옵트아웃** — "내 아바타 숨기기" 설정 제공.
- \+ 쌍별 쿨다운 3초로 연타 차단.

**기술 원칙** (배터리·데이터·iOS PWA)

- 네트워크로는 **이산 이벤트만**: 입장/퇴장 = Presence, 찌르기·리액션 파도 = broadcast 1건. 위치 스트리밍 없음 — 데이터는 사실상 0에 수렴.
- rAF 물리는 `visibilitychange` + IntersectionObserver로 화면 밖·백그라운드에서 완전 정지. 가능한 모션은 CSS 애니메이션으로.
- iOS standalone PWA는 백그라운드 전환 시 웹소켓을 즉시 끊는다 → **퇴장 판정 30~60초 유예**로 join/leave 깜빡임(flapping) 흡수, `pageshow`에서 재구독. FAB존은 `env(safe-area-inset-bottom)` 위에 명시적 적층, standalone 실기기 QA 통과가 출시 조건.
- 기존 리얼타임 선례: [notification-bell-icon.tsx](../../../components/notifications/notification-bell-icon.tsx) · [comment-section.tsx](../../../components/comment/comment-section.tsx)의 postgres_changes 구독 — Presence·broadcast 확장은 현실적.

**접근성** (강등이 아니라 동등한 1급 상태로 디자인)

- `prefers-reduced-motion` → 자율 이동 완전 오프, 정적 아바타 스택으로 대체.
- 물리 레이어는 `aria-hidden`, 스크린리더에는 "지금 ○○님 외 2명이 함께 보고 있어요" 텍스트 요약 + 정적 버튼 목록 병행.
- 티커류 자동 흐름 콘텐츠는 도입하지 않는다 (WCAG 2.2.2 — 아래 "보류" 참조).

**일몰 기준**: 출시 4주 후 주간 상호작용이 기준치 미만이면 정적 칩(Phase A)으로 강등한다. 기준치는 출시 전에 합의해 이 문서에 기록.

**검토 후 보류**

- 콘텐츠 전체 풀스크린 페이퍼 스와이프(스토리 모드) — 신문 피드를 폐기한 것과 같은 이유(§1 — 콘텐츠 밀도가 그릇보다 작음). 페이지 넘기는 손맛은 §3.1 스포트라이트 스와이프가 이미 담당.
- 뉴스 티커 — 수십 명 규모에선 같은 항목이 반복 순환하며 "콘텐츠 없음"을 오히려 광고하고, 컴팩트 존과 소스가 중복.

## 5. 데이터 / 백엔드

- **집계 RPC 신설**: `get_team_story_feed(team_id, cursor, limit)` — 스포트라이트/존 소스를 DB에서 union·정렬·페이징. 홈의 `get_public_team_*` 패턴과 동일. 각 아이템에 리액션 카운트 + (로그인 시)내 리액션을 조인.
  - **응답 형식**: 아이템마다 `{ zone, entity_type, entity_id, event_at, payload(존별 표시 데이터), rctn_count, my_rctn }`. 클라이언트가 `zone`으로 스포트라이트/컴팩트 존에 배치하므로 `zone` 필드로 그룹핑하거나 존별 배열로 내린다.
  - **정렬·cursor**: `event_at DESC, entity_id DESC` 고정(`entity_id`를 안정적 tie-breaker로). cursor = `(event_at, entity_id)` 복합값 — 동시각 아이템이 페이지 경계에서 중복·누락되지 않게 한다.
- **오버레이 방식**: 공개 데이터 목록은 캐시 가능, 리액션 카운트·내 리액션은 동적 오버레이 — 홈의 `is_attending` 클라이언트 오버레이 패턴 그대로.
- **서버/클라 패턴**: `page.tsx`(서버, 데이터 조회) → 클라이언트 피드 컴포넌트(스와이프·리액션 상호작용). `getCurrentMember()`로 로그인/멤버 판별.
- **날짜**: `@/lib/dayjs`만 사용, KST 기준. 문자열 조작 금지.

## 6. 의존성: 멤버 프로필 카드 (별도 설계)

> 이 카드의 세부 구성은 ✅ [멤버 프로필 카드 설계서](2026-07-22-멤버-프로필-카드-design.md)에서 확정됨. 여기서는 전광판이 무엇에 의존하는지만 명시.

- 스포트라이트 카드 = **재사용 가능한 멤버 프로필 카드 + 이유 배너**. 즉 프로필 카드가 전광판의 원자 단위이자 **선행 의존성**이다.
- 기존 [components/profile/profile-card.tsx](../../../components/profile/profile-card.tsx)는 "내 컬렉션" 편집이 달린 **소유자용 자기 카드**라 그대로 못 쓴다 → 남을 보여주는 **쇼케이스 카드를 신규 제작**.
- 데이터: `get_public_member_card(mem_id, team_id)` RPC — record-card 브랜치 잔재(dev 드리프트)는 2026-07-22 정리 완료, 프로필 카드 설계서의 **v2로 재정의**해 정식 마이그레이션으로 배포한다.
- 확장 예정(별도 설계에서 확정): **"한마디(자기소개)"** — 멤버가 직접 입력하는 한 줄. 저장 필드 신설 + 프로필 편집 입력란 + 카드 표시가 필요(마일리지 후기의 "한마디"와는 별개 개념). 각오/공약도 이 카드에 붙여 확장 가능.

## 7. 디자인 방향

디자인 시스템([DESIGN.md](../../../DESIGN.md))에서 크게 벗어나지 않되, 개성은 **폰트와 코너 톤 한 곳**에 몰아준다. 톤 온도는 "적당히".

- **Signature = 명조 헤드라인**: 헤드라인만 제목 폰트(Nanum Myeongjo)로 → 폰트 자체가 "전광판/제호" 느낌을 내 별도 장식 없이 개성이 난다. 본문·메타는 Pretendard.
- **색**: CSS 변수 토큰만. 종목 기록엔 `sport-*` 악센트. 하드코딩 색 금지.
- **타이포**: 매직넘버 금지 — `H2`/`Body`/`Caption`/`Micro`/`SectionLabel` 사용.
- **레이아웃**: `PageHeader`(기강이야기) + eyebrow. 스포트라이트(스와이프) → 컴팩트 존 세로 스택. 카드는 `CardItem`.
- **모션**: 스포트라이트 자동 전환/스와이프, 리액션 탭 피드백. 과함 금지(`prefers-reduced-motion` 존중).

## 8. 제약 / 주의사항

1. **기강 포인트는 히든 운영** — "기강포인트" 표기·원장(`pt_txn_hist`) 자체는 UI에 노출하지 않는다. 다만 🏆 순위 존은 **이달 참가왕**(참가 횟수) + **기강활동지수**로 표현하며, 활동지수는 유저에게 공개한다(11절 확정, 추후 전면 공개 예정).
2. **참가 기록 이름 보존** — 회원 탈퇴·삭제 후에도 과거 기록의 이름은 남아야 한다. `mem_mst` 삭제·이름 덮어쓰기 금지 원칙 준수.
3. **프라이버시** — 표시 필드는 공개 허용목록(`get_public_*` 프로젝션)에서만 가져온다. **각오·가입목적은 공개**(11절 확정). 단 거주지(가까운 역) 등 위치 정보는 헤드라인·목록에 노출하지 않는다.
4. **프레즌스 프라이버시** (§4.3) — "누가 언제 접속해 있는지"는 실명 크루에서 사회적으로 읽히는 정보다. 관중석 표시는 이 페이지 안에서만 유효(전역 온라인 상태 아님), 서버에 접속 이력을 저장하지 않으며(휘발 Presence 채널만), "내 접속 표시 끄기" 옵트아웃을 제공한다.

## 9. 범위 / 단계

- **1차** — ✅ **2026-07-22 구현 완료(dev)**: 멤버 프로필 카드(쇼케이스). 전광판의 선행 재료.
- **2차** — ✅ **2026-07-22 구현 완료(dev)**: 기강 전광판 — 스포트라이트(스와이프·자동전환·전환도트) + 컴팩트 존 4개 + `get_team_story_feed` RPC + **리액션**(`rctn_mst`).

### 2차 구현 노트 (실측 반영)

- **기간 창은 실측으로 정했다** — dev 기준 기록이 90일 5건·30일 0건이라, 설계 원안(최근 7일)이나 30일로 자르면 속보 존이 **상시 빈다**. 최종: 기록 90일 / 대회 60일 / 신규 30일. 크루 이벤트 빈도가 낮다는 게 이 페이지의 근본 제약이다.
- **`/story`를 공개 경로로 열었다** — RPC가 anon 허용(랭킹과 동일 정책)인데 `lib/supabase/proxy.ts`의 `publicPaths`에 없어 비로그인이 로그인 페이지로 튕겼다. 목록에 `/story` 추가.
- **리액션 캐시 전략** — `getStoryFeed`는 `p_mem_id` 없이 조회해 `unstable_cache`에 태운다(사용자별 캐시 분기 방지). 내 리액션은 버튼이 낙관적으로 관리하고, 토글 시 `revalidateTag("story-feed", "max")`로 카운트를 갱신한다.
- 리액션 세트는 아이템 종류별로 1종 고정(신규=👏환영 / 기록=🔥대박 / 대회=💪응원). 6종 전체를 고르게 하는 UI는 도입하지 않았다 — 감정이 이미 아이템에 정해져 있어 선택지가 오히려 군더더기.
- **3차**: 댓글(`cmnt_mst` 재사용) + 알림 연동 + 각오/공약 카드 확장.
- **후속 (전광판 출시 후)**: 관중석 프레즌스 Phase A → B → C (§4.3) — 단계별 착수 조건 충족 시에만.
- **나중**: 대회 후기(기록 입력에 후기 기능 생기면) · 마일리지 "한마디" 이전.
