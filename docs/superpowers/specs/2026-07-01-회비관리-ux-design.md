# 회비 관리 UX 혁신 — 설계 문서 (P1)

- **작성일**: 2026-07-01
- **상태**: 확정 (P1 구현 대기)
- **범위**: 관리자 "회비 관리" 운영자 워크플로 재설계. 데이터 모델 본체는 유지, 매칭 학습 엔진·Triage Inbox·상태 단순화·잔액 원장 랜딩을 신설.
- **후속**: P2(프로젝트성 모금), P3(회원 본인 잔액 뷰 + 미납 푸시 독려)는 별도 스펙.

---

## 1. 배경 & 문제 정의

기강은 회원(~100명)이 모임 계좌로 입금한 은행 거래내역 엑셀을 매월 1회 업로드해 회비를 정산한다. 데이터 모델(원시거래 `fee_txn_hist` → 확정 납부원장 `fee_due_pay_hist` → 면제 `fee_due_exm_*` → 잔액 스냅샷 `fee_mem_bal_snap`)은 감사성 있게 잘 분리돼 있다. **병목은 백엔드가 아니라 운영자 워크플로**다.

### 운영 현실 (인터뷰)

- 회비는 월 **2,000원**. 회원은 몰아서 낸다(예: 10,000원 = 5개월, 6,000원 = 3개월). 잔액 수학(`낸돈 + 면제 − 부과`)이 몰아내기를 자동 흡수하므로 입금을 월별로 쪼갤 필요 없음.
- **들어오는 돈이 전부 회비가 아니다.** 모임 식대 대납·단체복 구매 등 **프로젝트성 입금**이 무더기로 섞인다(예: 단체복 구매자 수만큼 30,000원 입금). 이건 회비가 아니라 별도로 "누가 얼마 냈는지"만 보면 된다.
- 운영자가 관리하는 것은 오직 **(a) 각 회원이 회비를 잘 냈는가 (b) 더 냈나(예치금)/덜 냈나(미납)** 뿐. 지출 상세는 추적하지 않는다.

### 매칭이 자동으로 안 되는 이유 (핵심 페인)

1. **입금자명을 보내는 사람이 자유 선택** — "회비 홍길동", "홍길동", 닉네임 등 제각각.
2. **가입명도 제각각** — 실명/이메일 아이디/닉네임/영문 약자.
3. **동명이인** 다수(~100명 규모).
4. **비회비(프로젝트성) 입금 혼입** — 매칭 전에 회비 여부부터 갈라야 함.
5. **금액 제각각** — 2,000원 배수가 아닌 몰아내기·단체복값 등.

매칭이 매월 반복되는데 **학습이 안 되는 것**이 최대 손실. 같은 사람이 같은 닉네임으로 보내도 매달 다시 손으로 찾는다.

### 현재 UX의 구체적 불편 (코드 분석)

- 거래내역 화면이 단일 거대 테이블(1,668줄): 정렬 6종 + 필터 4종 + 행마다 인라인 컨트롤 3종.
- 확정과 재계산이 **2단계로 분리**(버튼 두 번).
- 상태 표현 5종(미확정/확정/미반영/계산대기/반영)으로 난해.
- 회비 관리가 7개 탭(회비계산·거래내역·회원별잔액·면제관리·회비정책·지출·제외)으로 난립.

## 2. 목표 & 비목표

### 목표 (P1)

- 매월 운영자가 손대는 거래 수를 **시간이 갈수록 줄이는** 매칭 학습.
- 업로드 후 **"확인필요"만** 집중 처리하는 Triage Inbox.
- 확정 + 재계산을 **한 동작으로** 통합.
- 상태 표현 **5종 → 3종** 단순화.
- 7개 탭을 **3면**(잔액 원장 · Triage Inbox · 프로젝트)으로 재편. P1은 앞 2면.

### 비목표 (P1에서 제외)

- 프로젝트성 모금의 대상자·인당금액·미납 추적 → **P2**.
- 회원 본인 잔액 뷰, 미납자 푸시 독려 → **P3**.
- 지출(출금) 상세 회계/영수증 관리 → 범위 밖(운영자가 추적 안 함).
- 기존 잔액 스냅샷/납부원장 스키마의 구조 변경 → 하지 않음(컬럼 추가만).

## 3. 아키텍처 — 7페이지 → 3면

```
회비 관리
├── 📒 잔액 원장 (랜딩)   ← 회원별 더/덜 냄 한눈, 면제·정책 인라인       [P1]
├── 📥 Triage Inbox       ← 엑셀 업로드 후 새 입금만 처리              [P1]
└── 🎯 프로젝트            ← 일회성 모금, 태그별 대상자·미납 추적        [P2]
```

- 기존 `회비계산`·`거래내역` → **Triage Inbox**로 흡수.
- 기존 `회원별잔액`·`면제관리`·`회비정책` → **잔액 원장**으로 흡수(면제·정책은 인라인 액션/설정).
- 기존 `지출`·`제외` → Inbox의 분류 결과(`제외` 버킷)로 자연 통합. 별도 조회는 Inbox 필터로 제공.

라우트는 기존 `/app/(info)/admin/dues/` 하위를 재편한다. 세부 라우트 구조는 구현 계획(writing-plans)에서 확정.

## 4. 핵심 엔진 — 매칭 학습 (모든 면의 토대)

### 4.1 신규 테이블 `fee_payer_alias`

```
alias_id      uuid pk
team_id       uuid  (fk teams)
raw_name_norm text  (정규화된 입금자명)
mem_id        uuid  (fk members)
hit_cnt       int   default 0   (자동매칭 성공 누적)
last_used_at  timestamptz
created_by    uuid
created_at    timestamptz default now()
UNIQUE(team_id, raw_name_norm)
```

- 운영자가 Inbox에서 "이 이름 기억하기"를 체크하고 확정하면 1행 기록/갱신.
- 다음 업로드부터 이 매핑으로 자동 매칭 → ⚠️ 큐에서 빠짐.
- RLS: 같은 team 관리자만 read/write (`verifyAdmin` 패턴). 기존 회비 테이블 RLS 정책과 동일 형태.

### 4.2 정규화 함수 (단일 출처)

입금자명 정규화 규칙을 **한 곳**(`lib/dues/normalize-payer-name.ts`)에 정의해 업로드 파서와 별칭 조회가 공유:

- 앞뒤 공백 제거, 내부 공백 단일화.
- 접두/접미 노이즈 토큰 제거: `회비`, 숫자, 특수문자.
- 결과를 소문자화(영문) + 한글 자모 정규화(NFC).

### 4.3 매칭 파이프라인 (업로드 시)

각 입금 거래에 대해 순서대로:

1. **별칭 정확일치** (`fee_payer_alias.raw_name_norm`) → 단일 회원 ⇒ **자동완료(회비, 매칭)**, `hit_cnt++`.
2. **회원 실명/닉네임 정확일치** → 단일 ⇒ 자동완료. 후보 2+ ⇒ ⚠️ 동명이인 큐.
3. **부분포함·자모 유사** → 후보 Top-N(신뢰도 점수 포함) ⇒ ⚠️ 큐.
4. 매칭 0 ⇒ ⚠️ 미매칭 큐.

자동 분류(회비/제외)는 기존 규칙 유지: 출금→제외(지출), 예금이자·타행자동이체→제외(기타), 그 외 입금 기본값→회비. **동명이인·미매칭 회비는 절대 자동완료하지 않는다**(항상 ⚠️ 큐).

### 4.4 `fee_txn_hist` 분류 확장

- `fee_item_cd`에 `project` 의미를 추가(P1에서는 선택 시 "보류" 버킷, P2에서 `project_id` FK 활성).
- `project_id uuid nullable` 컬럼만 미리 추가(P1 미사용, P2 연결). 마이그레이션 위험 0.

## 5. Triage Inbox (P1)

### 5.1 레이아웃

```
┌─────────────────────────────────────────────┐
│  📥 거래내역 업로드        [엑셀 선택] [지난 업로드▾] │
├─────────────────────────────────────────────┤
│  이번 업로드: 132건  ✅자동완료 98  ⚠️확인필요 27  🚫제외 7 │
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░  74% 자동 처리됨            │
├─────────────────────────────────────────────┤
│  ⚠️ 확인필요 27건  ── 묶음 보기 [켜짐]            │
│                                               │
│ ┌──[무더기] 30,000원 · 1.12 · 5건 ──────────┐ │
│ │ 단체복값으로 추정 (같은 금액·같은 날)        │ │

│ │ → 한 번에:  [회비] [🎯프로젝트(P2)] [제외]  │ │
│ └───────────────────────────────────────────┘ │
│                                               │
│ ┌──[단건] 10,000원 · 1.08 · "민초러버" ──────┐ │

│ │ ☑ 이 이름 기억하기 (다음부터 자동)          │ │
│ └───────────────────────────────────────────┘ │
│            … 카드 큐 계속 …                     │
├─────────────────────────────────────────────┤
│  [27건 모두 처리하면 활성] → ✅ 확정 + 회비 재계산 │
└─────────────────────────────────────────────┘
```

### 5.2 동작

- **세 버킷**: ✅자동완료(펼쳐 확인만) · ⚠️확인필요(처리 대상) · 🚫제외.
- **후보 탭 한 번 = 매칭.** 키보드 단축키: ↑↓ 후보 이동, 1/2/3 분류 선택, Enter 다음 카드.
- **"이 이름 기억하기"** 체크 시 확정에서 `fee_payer_alias` 기록.
- **묶음**: 같은 날·같은 금액 N건 자동 그룹핑 → 일괄 분류. 무더기 안의 개별 회원 매칭은 카드 내에서 각각.
- **확정 + 재계산 통합**: 큐가 비면 하단 버튼 1회로 확정과 잔액 재계산을 원자적으로 실행. (P1: "프로젝트" 선택 건은 보류 버킷에 두고 잔액에 영향 주지 않음 — P2에서 처리.)
- 이미 처리된 과거 거래(중복 업로드)는 자동완료/제외에서 회색 처리.

### 5.3 상태 단순화 (5종 → 3종)

UI는 기존 컬럼(`match_st_cd`, `is_cfm_yn`, 스냅샷 반영 여부)에서 **파생**해 3종으로 표시(스키마 변경 없음):

| 표시 상태 | 의미 (파생 규칙) |
|---|---|
| 처리대기 | 미확정 — Inbox ⚠️ 큐 대상 |
| 확정 | 확정됨, 아직 잔액 미반영 |
| 반영완료 | 확정 + 잔액 스냅샷 반영 |

확정과 재계산이 통합되므로 실무상 "확정"은 과도기 상태이며, 정상 흐름에서는 처리대기 → 반영완료로 이어진다.

## 6. 잔액 원장 (랜딩, P1)

### 6.1 레이아웃

```
┌─────────────────────────────────────────────┐
│  📒 회비 현황              기준 2026.06  [정책⚙]│
│  미납 23명  ·  정상 71명  ·  예치 6명           │
├─────────────────────────────────────────────┤
│  [전체] [미납만] [예치만]        🔍이름 검색     │
│ ─────────────────────────────────────────────│
│  김철수    -4,000원  (2개월 미납)   [면제][독려]│
│  이영희    +6,000원  (3개월 예치)               │
│  박민수      0원      정상                       │
│   …                                            │
├─────────────────────────────────────────────┤
│  [미납자 전체 푸시 독려]  ← P3에서 활성          │
└─────────────────────────────────────────────┘
```

### 6.2 동작

- 회원 행 탭 → 납부원장 드릴다운(입금 이력 + 월별 부과 차감 타임라인).
- **면제 등록 인라인**: 기존 별도 탭 → 행 액션 다이얼로그(full/part, 기간, 사유). 기존 `create-exemption` 서버액션 재사용.
- **정책 설정**: 월 회비 금액(기본 2,000원) 등은 상단 [정책⚙]에서. 기존 `fee_policy_cfg` 재사용.
- "독려"·"미납자 전체 푸시"는 P1에서 자리만(disabled), P3에서 활성.

## 7. 에러 / 엣지 케이스

- **중복 업로드**: 기존 `UNIQUE(team_id, txn_dt, txn_tm, txn_amt, raw_name)` + 파일 해시(`fee_xlsx_upd_hist.file_hash`)로 차단. 처리 완료 과거 거래는 회색 표시.
- **확정 취소**: 반영완료 건 취소 시 "재계산 영향" 경고 후 스냅샷 롤백(기존 `rollback-snapshot` 재사용).
- **별칭 충돌**: 같은 `raw_name_norm`이 다른 회원에 재매칭되면 최신 매핑으로 갱신 + 운영자 경고 토스트.
- **동명이인**: 별칭으로도 안 갈리면 자동완료 금지, 항상 ⚠️ 큐.
- **빈 업로드/형식 오류**: 파서가 검증 후 사용자 친화 메시지(어느 행이 문제인지).
- **프로젝트 분류(P1)**: 선택 시 보류 버킷으로만 두고 잔액 영향 없음. P2 도입 전까지 명확히 "프로젝트 기능 준비중" 표기.

## 8. 테스트 전략

- **정규화·퍼지 매칭 단위 테스트(vitest)**: "회비 김철수" / "김철수회비" / "민초러버" / 동명이인 / 영문 약자 케이스.
- **별칭 학습 회귀**: 별칭 기록 후 다음 업로드에서 자동매칭률 상승 검증.
- **잔액 재계산 시나리오**: 몰아내기(10,000=5개월), 면제 적용, 선납/미납 경계.
- **서버액션 원자성**: 확정 + 재계산이 하나의 트랜잭션으로 성공/실패.
- **상태 파생 로직**: 5종 원천 → 3종 표시 매핑 테이블 테스트.

| 잔액 원장 클라이언트 | 회원별 잔액·면제·정책 인라인 | 기존 스냅샷/면제/정책 |

각 단위는 단독 테스트 가능하도록 인터페이스를 좁게 유지. 기존 1,668줄 단일 거래내역 클라이언트는 버킷/카드 단위로 분해한다.
