# 회비 거래처리 화면 개편 (Inbox v2 — 통합 테이블) 설계

> 상위 로드맵의 **SP1**. 후속 SP2(프로젝트 추적)·SP3(회원용 잔액+수동 푸시)는 별도 스펙.

**작성일**: 2026-07-02
**브랜치(예정)**: `feature/dues-inbox-table`
**베이스**: `dev`

---

## 목표

회비 거래내역 처리(업로드→분류) 화면을 **카드 큐에서 단일 통합 데이터 테이블로 개편**한다. ~100건 규모를 스캔·정렬·일괄처리하기 좋은 스프레드시트형 UI로 바꾸고, **동명이인 후보를 생년월일로 구분**할 수 있게 한다. 백엔드 조회·확정 흐름은 그대로 재사용하고 **표현 계층만 교체**한다.

## 배경 (현재 상태)

- 처리 화면 `app/(info)/admin/dues/inbox/` = `inbox-client.tsx`(3버킷 카드 큐) + `inbox-card.tsx`(카드 1건).
- 데이터: `getInboxTxns()`(`lib/queries/dues.ts`) → `{ members, txns: InboxTxn[] }`. 미확정(`is_cfm_yn=false`) 거래를 팀 회원·별칭으로 매칭해 `bucket`(autoDone/needsReview/excluded) 부여.
- 확정: `confirmAndRecalc({ items, aliasLearn })` → 확정 + 별칭학습(확정 성공 후) + 전체 활성회원 재계산. `router.refresh()`로 갱신.
- 회원 데이터 `mem_mst`에 **`birth_dt`(nullable) 컬럼 존재** — 현재 조회·표시 안 함.
- 운영자는 **주로 PC/노트북**에서 이 작업을 함(2026-07-02 확인) → 넓은 테이블 적합.

## 범위

**포함**
1. 카드 큐 → **단일 통합 테이블** (필터 칩 + 검색 + 인라인 편집 + 다중선택 일괄처리 + 키보드).
2. **생년월일 구분(A)**: `getInboxTxns`가 `birth_dt` 조회 → 후보/회원 타입에 포함 → **동명이인일 때만** 이름 옆 표시.

**제외 (명시)**
- 프로젝트 선택 열 → **SP2**에서 이 테이블에 열 추가.
- 확정된 과거 내역 화면(`transactions/`)·제외 전용 화면(`excluded/`) 등 다른 admin/dues 라우트 개편.
- 열 헤더 클릭 정렬(초기 정렬만 제공, 클릭 정렬은 후순위).
- 별도 모바일 레이아웃(모바일은 가로 스크롤로 그대로 노출).
- React Testing Library 도입(현 vitest는 순수 로직 테스트만) — 인터랙션은 라이브 QA로 검증.

## 데이터 계층 변경

### `getInboxTxns()` — `birth_dt` 스레딩 (`lib/queries/dues.ts`)

`mem_mst` 조회에 `birth_dt` 추가하고, memId→birthDt 맵을 만들어 **후보·회원 목록에 붙인다**. `matchPayer`의 순수 로직은 건드리지 않는다(candidates는 그대로 받고 getInboxTxns에서 birthDt만 부착).

```ts
// 변경: select에 birth_dt 추가
.from("mem_mst").select("mem_id, mem_nm, birth_dt")...

// memId → birthDt 맵
const birthById = new Map<string, string | null>(
  (mems ?? []).map((m) => [m.mem_id, m.birth_dt]),
);

// candidates·members에 birthDt 부착 (아래 타입 참조)
```

### 타입 변경 (`lib/queries/dues.ts`)

```ts
export type MemberOption = { memId: string; name: string; birthDt: string | null };

export type InboxTxn = {
  txnId: string;
  txnDt: string;
  txnTm: string | null;
  amt: number;
  rawName: string;
  memId: string | null;
  matchStatus: "matched" | "unmatched" | "ambiguous";
  feeItemCd: string | null;
  bucket: "autoDone" | "needsReview" | "excluded";
  candidates: { memId: string; name: string; score: number; birthDt: string | null }[];
};

// 반환 members 타입: { memId; name; birthDt }[]  (= MemberOption[])
export async function getInboxTxns(): Promise<{ members: MemberOption[]; txns: InboxTxn[] }>;
```

- `database.types.ts`는 `mem_mst.birth_dt`를 이미 포함하므로 추가 브릿지 불필요.

### 동명이인 판정 헬퍼 (신규, 순수·테스트 대상)

`lib/dues/homonyms.ts`

```ts
/** 회원 목록에서 2명 이상이 공유하는 이름 집합. 생년월일 표시 여부 판단에 쓴다. */
export function duplicateNames(members: { name: string }[]): Set<string> {
  const seen = new Map<string, number>();
  for (const m of members) seen.set(m.name, (seen.get(m.name) ?? 0) + 1);
  return new Set([...seen].filter(([, n]) => n >= 2).map(([name]) => name));
}
```

전체 팀 회원(`members`) 기준으로 계산한다 — 한 거래의 후보에 동명이인이 하나만 보여도, 검색으로 고를 다른 동명이인이 팀에 있으면 구분이 필요하기 때문.

### 생년월일 표시 규칙

- 이름이 `duplicateNames`에 속할 때만 이름 뒤에 `(YY.MM.DD)` 표시. 아니면 표시 안 함.
- `birthDt`가 `null`이면 생략(이름만).
- 포맷은 `dayjs(birthDt).format("YY.MM.DD")` (프로젝트 규칙, 문자열 slice 금지).

## UI 계층

### 파일 구성

- **교체**: `inbox-client.tsx` → `inbox-table.tsx`(테이블 클라이언트), `inbox-card.tsx` → `inbox-row.tsx`(행 1건).
- **추출**: 회원 검색 콤보박스를 `member-combobox.tsx`로 분리(행에서 재사용, 생년월일 표기 포함).
- **공유 상수/타입**: `Decision`, `ItemCd`, `ITEM_LABEL`, `ITEM_ORDER`는 `inbox-row.tsx`에 유지(현행과 동일 위치 감각).
- `inbox/page.tsx`: `<InboxClient>` → `<InboxTable>` import·렌더로 교체(props 동일: `members`, `txns`).

### 상태 모델 (현행 유지 — 파생 렌더 패턴)

`decisions`는 `overrides`(사용자 변경분) + `defaultDecision(t)` 파생으로 계산한다. **effect 내 setState 금지 규칙**을 지키기 위해 P1의 `useMemo` 파생 패턴을 그대로 계승한다.

```ts
function defaultDecision(t: InboxTxn): Decision {
  return { memId: t.candidates[0]?.memId ?? null, itemCd: "due", remember: false };
}
// decisions = useMemo(review.map(t => [t.txnId, {...defaultDecision(t), ...overrides[t.txnId]}]))
```

### 테이블 레이아웃

```
[전체 42] [확인필요 8] [자동 30] [제외 4]        (검색: 입금자·금액)
--------------------------------------------------------------
[v] 날짜   입금자      금액     매칭 회원          분류      상태
--------------------------------------------------------------
[v] 07.01 회비홍길동  10,000  <홍길동 92% v>     <회비 v>  확인
[ ] 07.01 김철수       2,000  김철수 (자동)      회비      자동
[ ] 07.01 (출금)      -5,000  --                 제외      제외
--------------------------------------------------------------
선택 1건: [회비로] [프로젝트로] [제외로]        [42건 확정 + 회비 재계산]
```

**필터 칩**: `전체 / 확인필요 / 자동 / 제외`. 기본 = **확인필요**. 각 칩에 건수 배지(현행 Badge 색: 자동=success, 확인필요=warning, 제외=muted).

**행 정렬**: 서버가 `txn_dt` 오름차순 제공. 화면에서는 현재 필터 내에서 그 순서 유지. (열 클릭 정렬은 범위 밖.)

**검색**: 입금자 원본명 부분일치 + 금액 문자열 부분일치. 필터 칩과 AND.

**열 & 인라인 컨트롤**:
| 열 | 내용 |
|---|---|
| ☑ | 행 선택 체크박스(현재 필터의 "전체 선택" 헤더 체크 포함) |
| 날짜 | `dayjs(txnDt).format("YY.MM.DD")` |
| 입금자 | `rawName`(원본) |
| 금액 | `amt.toLocaleString()`원 |
| 매칭 회원 | **needsReview**: 후보 칩(`이름 nn%`, 동명이인이면 `이름 (YY.MM.DD) nn%`) + `MemberCombobox`(직접선택). **autoDone/회비**: 배정 회원명 **읽기 전용**. **제외/프로젝트**: `—` |
| 분류 | `회비/프로젝트/제외` 드롭다운(needsReview는 편집, autoDone/excluded는 저장값 **읽기 표시**) |

> autoDone·excluded 행의 재분류·재매칭은 SP1 범위 밖(확정 payload는 P1처럼 txnId만 넘겨 저장값 유지). 자동매칭이 틀린 예외 케이스 수정은 후속 과제로 둔다.
| 상태 | 배지: `확인`(warning)/`자동`(success)/`제외`(muted) |

- **확인필요 미결정 행**: 행 배경 warning 틴트 → 결정(회비면 memId 지정, 그 외 분류 선택)하면 해제.
- **회비+회원 확정 시** 행 내 `이 이름 기억하기` 토글(별칭 학습, 현행과 동일).

**다중선택 일괄처리**: 선택된 행에 `[회비로][프로젝트로][제외로]` 일괄 적용(`bulkSetItemCd` 재사용, 여러 txnId에 `itemCd` 패치). `회비로`는 `itemCd=due`만 세팅(회원은 행별로).

**키보드**: 행에 `tabIndex=0`. 활성 행에서 `↑/↓` 이동, `1/2/3` = 회비/프로젝트/제외, `Enter` = 다음 행 포커스. 내부 입력(콤보박스 등)에서의 타이핑은 `e.target === e.currentTarget` 검사로 가로채지 않음(현행 규칙 계승).

### 확정 흐름 (변경 없음)

`onSubmit`은 현행 `inbox-client.tsx`와 동일:
- `items` = autoDone(txnId만) + excluded(txnId만) + review(txnId+feeItemCd+memId).
- `aliasLearn` = review 중 `remember && due && memId`.
- `confirmAndRecalc({ items, aliasLearn })` 호출(**recalcMemIds 생략** → 전체 활성회원 재계산). 성공 시 `router.refresh()`.
- 확정 버튼: `pending || !allDecided || totalToConfirm===0` 이면 비활성. `allDecided` = 모든 needsReview 행이 결정됨(회비는 memId 필수).
- **확정은 현재 필터 칩과 무관하게 전체 배치(autoDone+review+excluded)를 대상으로 한다** — 필터는 보기(view)일 뿐, 확정 범위를 좁히지 않는다.

## 에러 처리

- 조회 에러는 `getInboxTxns`가 이미 `throw` → 빈 화면을 정상으로 오인하지 않음(현행 유지).
- 확정 실패 시 서버 메시지를 캡션으로 노출(현행 `setMsg`).
- `birth_dt`가 null이거나 dayjs 파싱 불가한 값이면 생년월일 부분만 생략하고 이름은 정상 표시(방어).

## 테스트 (vitest, `lib/__tests__/`)

- `duplicateNames`: 중복 없음 → 빈 Set / 동명이인 2명 → 그 이름 포함 / 3명 이상·복수 그룹 / 이름 트림·대소문자는 정규화 안 함(정확 일치 기준) 등 케이스.
- 인터랙션(필터·인라인 편집·일괄·확정 payload)은 RTL 미도입으로 **라이브 QA로 검증**(P1과 동일 방침). `onSubmit`의 payload 구성 로직이 순수 함수로 분리 가능하면 별도 테스트 추가(선택).

## 마이그레이션 / 배포

- **DB 스키마 변경 없음**(birth_dt 기존 컬럼). 마이그레이션 불필요.
- 라이브 QA(dev): 업로드 → 필터/검색 → 인라인 분류·회원지정 → 동명이인 생년월일 노출 → 일괄 → 확정 → 원장 반영 확인.

## 후속 (별도 스펙)

- **SP2 프로젝트 추적**: 프로젝트 마스터(태그) 테이블 + 이 테이블에 "프로젝트" 열(어느 프로젝트로 온 돈인지 선택, `fee_txn_hist.project_id` 채움) + 대상/참여자·모금액 집계.
- **SP3 회원용**: 회원 본인 납부/미납/예치 조회 화면 + 운영진 수동 푸시 발송(자동 넛지 아님).
