## 확정된 결정

1. **활동량 집계 주기 = 매달.** 설계서(`docs/design/2026-07-04-기강포인트제도.md` §6)는 연 단위 윈도우를 명시하지만, 운영 판단으로 월 단위로 간다. 원장(`pt_txn_hist`)은 그대로 두고 집계 윈도우만 자르므로 리셋 배치는 없다.
2. **내역에 지수 증감량을 공개한다.** 포인트 제도가 사실상 읽기 개방으로 전환된다(설계서 로드맵 2단계). 배점 룰은 내역에서 역산 가능해진다 — 의도된 결정.
3. **"포인트"라는 단어는 계속 쓰지 않는다.** 화면 명칭은 **"활동량"**으로 통일한다(현재 "활동지수"/"활동량" 혼용을 정리).

## 6. 응원 — 무한 카운트

### 데이터

```sql
ALTER TABLE public.rctn_mst ADD COLUMN rctn_cnt integer NOT NULL DEFAULT 1;
ALTER TABLE public.rctn_mst ADD CONSTRAINT ck_rctn_cnt CHECK (rctn_cnt BETWEEN 1 AND 99);
```

유니크 제약 `(team_id, entity_type, entity_id, mem_id)`는 **유지한다**. 1인 1행이되 행 안에서 카운트가 오르는 구조라, 누가 몇 번 눌렀는지가 남고 1인 상한(99)이 제약으로 자연히 걸린다. 행을 여러 개 쌓는 설계는 감사성은 같지만 테이블이 무한히 커지고 상한 강제에 별도 집계가 필요해 채택하지 않는다.

피드 RPC의 `rctn` CTE: `count(*)` → `sum(rctn_cnt)`. `my_rctn`에 더해 `my_cnt`(내가 누른 횟수)를 함께 내린다.

### 증가 RPC

증가는 원자적이어야 한다(읽고-쓰기 왕복은 동시 연타에서 유실된다). `bump_story_rctn(p_team_id, p_entity_type, p_entity_id, p_mem_id, p_rctn_cd, p_delta)` — `INSERT ... ON CONFLICT DO UPDATE SET rctn_cnt = LEAST(rctn_mst.rctn_cnt + EXCLUDED.rctn_cnt, 99)`. 갱신 후 행의 `rctn_cnt`를 반환해 클라이언트가 상한 도달을 알 수 있게 한다.

### 서버 액션

`toggleStoryReaction` → `bumpStoryReaction({ entityType, entityId, rctnCd, delta })`.
- `delta`는 1~20으로 검증(한 번의 flush가 담을 수 있는 최대치)
- `entityType` / `rctnCd`는 기존과 동일하게 허용목록 검증
- **`revalidateTag`를 호출하지 않는다.** 여럿이 연타하면 `story-feed` 캐시가 계속 무효화된다. 낙관적 업데이트로 즉시 반영하고 서버 값은 기존 5분 주기에 수렴시킨다 — 응원 숫자에 실시간 정확도는 요구되지 않는다.

토글(취소)은 사라진다. 누른 건 되돌릴 수 없다 — 카운트업 UI의 계약이다.

### 클라이언트

`StoryReactionButton`을 다시 쓴다.
- 탭 → 즉시 `count + 1`, 로컬 `pending`에 누적
- 700ms 디바운스 후 `bumpStoryReaction({ delta: pending })` 1회 전송
- `pagehide` / `visibilitychange(hidden)`에서 즉시 flush(이탈 시 유실 방지)
- 실패 시 pending 만큼 롤백 + 토스트

### 모션 — 3층

| 층 | 내용 |
|----|------|
| 숫자 롤업 | `key={count}`로 재마운트, `translateY(100%) → 0` + fade. Oswald `tabular-nums` |
| 이모지 버스트 | 탭마다 이모지 1개가 위로 튀며 좌우 랜덤(±24px), 0.7s에 `scale 1→1.6` fade-out. 동시 최대 8개, 타이머로 제거 |
| 콤보 | 1.2초 내 연타 3회 이상이면 우측에 `×N` 배지 팝 + 버튼 `scale(1.04)` 유지. 콤보 끊기면 복귀 |

`prefers-reduced-motion`이면 롤업만 남기고(그것도 fade로) 버스트·콤보는 생략한다. keyframes는 `globals.css`에 둔다(기존 `board-*` / `lede-in`과 같은 자리).

## 8. 신규멤버 = 간단 프로필 카드

### 간단 카드의 정의 (여기서 확정)

- **상세**(`MemberCardDetail`) = "이 사람의 실적" — 스크린 존 + RECORDS / ACTIVITY / TITLES
- **간단**(`MemberCardCompact`) = "이 사람이 누구인지" — 아바타 · 이름 · 칭호 · 한마디 · **러닝 프로필**

신규멤버는 실적이 없다. 그래서 러닝 프로필이 간단 카드의 본체가 된다.

```
┌────────────────────────────────────────┐
│ ┌────┐  민준  [칭호]              7.19 │  meta 슬롯 = 가입일
│ │아바│  "완주가 목표입니다"              │  intro_txt
│ └────┘  6'00"/km · 8km · 합정역         │  러닝 프로필 (신설)
└────────────────────────────────────────┘
```

- 셋째 줄은 `getMemberIntro().rows`를 `·`로 이어붙인다(기존 함수 재활용)
- `intro_txt`도 러닝 프로필도 없으면 가입 목적 칩으로 폴백, 그것도 없으면 줄 자체를 생략
- 탭하면 기존대로 `MemberCardDialog` 상세가 열린다

### props 타입 좁히기

`MemberCardCompact`가 현재 `MemberCardData` **전체**를 요구한다. 그래서 피드가 컴팩트 카드를 그리려면 상세 카드 payload를 통째로 실어야 한다.

props를 실제로 쓰는 필드만 담은 타입(`MemberCardCompactData`)으로 좁힌다: `mem_nm`, `avatar_url`, `badge_effect`, `frame_cd`, `intro_txt`, `primary_title`, `running_profile`. `MemberCardData`가 이 타입을 구조적으로 만족하므로 기존 호출부는 수정 없이 동작한다.

### RPC

`newbies` CTE에 `mem_onbd_prf` LEFT JOIN(페이스·거리·역·목적)과 칭호/프레임을 추가하고, 위 7개 필드만 payload에 싣는다.

### 레이아웃

카드가 되면 행 높이가 커진다. `StorySection initial={3}`은 그대로 둔다.

### (d) `actv_rank`를 월 집계로

```sql
WHERE pt.team_id = p_team_id
  AND pt.aply_dt >= date_trunc('month', (now() AT TIME ZONE 'Asia/Seoul')::date)
  AND pt.aply_dt <= (now() AT TIME ZONE 'Asia/Seoul')::date
```

두 번째 조건이 필수다. 설계서 §"모든 유저-노출 집계는 `aply_dt <= 오늘(KST)` 필터 필수" — 대회 포인트는 **개최일**에 귀속되므로, 필터가 없으면 11월 대회를 신청한 사람이 7월 랭킹에 미리 잡힌다. 내역 조회에도 같은 조건을 적용한다.

## 묶음 A — 지면 수정 (4·5·7·9)

### 4·9. 최근 기록

`recent_recs`의 `race_dt >= today - 90` 필터를 없앤다. "최근 90일"이 아니라 **최신순 10건**이면 된다 — 활동이 뜸한 달에도 지면이 비지 않는다. 프론트의 3건 → 10건 접기는 이미 있다. payload에 `race_dt`를 실어 목록에 대회일자를 표시한다(기간 제한이 사라지면서 필수 정보가 됐다).

### 5. 자동 전환 재개

`manual` 불리언을 타이머로 바꾼다. 손이 닿으면 10초 멈추고, 반응이 없으면 3초 주기를 되살린다. 영구 정지하면 한 번 만진 뒤로 전광판이 죽은 화면이 된다.

### 7. 리드 슬롯 — 종류당 한 칸

슬롯 4개 고정: 다가오는 대회 1 / 새 얼굴 1 / 기록 1 / 이달의 참가왕 1. 새 얼굴·기록은 최근 30일.

가장 최근 1명(1건)이 대표로 크게 서고, 나머지는 우측 레일에 작게 선다. 레일은 `w-12` + 세로 괘선(`border-l`), 32px 아바타 + 10px 이름, 최대 3명 + "외 N명". 레일의 얼굴도 탭하면 각자의 카드가 열린다 — 지면에서 빠지는 사람이 없게.

**참가왕 하단 섹션은 제거한다.** 활동량이 월 집계가 되면서 월 랭킹 두 개가 연달아 서면 같은 걸 두 번 본 것처럼 읽힌다. 참가왕은 리드 슬롯에만 남긴다.

## 기강 기상대 (아이디어 2)

신문에는 일기예보란이 있다. 이 지면이 신문이므로 크루 총량도 같은 형식으로 싣는다: 왼쪽에 큰 상태(아이콘 + 한 단어), 오른쪽에 촘촘한 수치 격자, 아래에 8주 추세 막대. **먼저 분위기를 말하고 근거를 뒤에 붙인다** — "회원 42명"만 나열하는 통계 블록과 다른 점.

단어는 프로필 카드의 개인 컨디션과 **같은 4단계**를 쓴다(기강 그 자체 / 기강 잡아 / 기며든다 / 실종). 개인 카드에서 본 말을 크루 박스에서 다시 만나면 두 지표가 같은 척도라는 게 설명 없이 전달된다.

판정은 절대 건수가 아니라 **이번 주 ÷ 직전 4주 평균**이다(크루 규모마다 절대값이 달라서). 기준선이 0인 초기 크루는 절대량으로 폴백한다.

### 데이터

`get_team_overview(p_team_id)` — 별도 RPC로 뺀다. 피드 함수는 이미 CTE가 열 개를 넘고, 관심사(개별 소식 vs 크루 총량)와 캐시 수명(5분 vs 1시간)이 다르다.

반환: `mem_cnt`(활동 회원 수) + `weeks`(최근 8주 월요일 시작 KST, 각 주의 모임/참석 연인원/기록). 마지막 원소가 이번 주이고, 상한을 `LEAST(주끝, now())`로 잘라 미래 일정이 이번 주 수치를 부풀리지 않게 한다.

## 검증

- `pnpm run lint` / `pnpm run build` 통과
- 연타 후 새로고침해도 카운트가 유지되는지(디바운스 flush 유실 없음)
- 1인 99회 상한 도달 시 UI가 더 오르지 않는지
- 비로그인 상태에서 응원 시 에러 메시지가 뜨는지
- `prefers-reduced-motion: reduce`에서 버스트·콤보가 멈추는지
- 신규멤버 중 온보딩 프로필이 비어 있는 사람의 카드가 깨지지 않는지
- 내역이 비어 있는 멤버의 시트 빈 상태
- 스와이프 후 10초 지나면 자동 전환이 되살아나는지
- 최근 30일 신규·기록이 0명/0건일 때 해당 리드 슬롯이 통째로 빠지는지
- 신규 크루(4주 기준선 0)에서 기상 판정이 절대량 폴백으로 넘어가는지

## 미적용 사항

마이그레이션 4건은 **어느 DB에도 적용하지 않았다.** dev/prd 반영 시점은 별도 판단.
미적용 상태로 배포돼도 크래시는 없다 — `my_cnt`·카드 필드·`race_dt`가 없으면 0/빈 값으로 degrade하고,
기상대는 `weeks`가 비어 `null`을 반환해 상자 자체가 접힌다. 응원만 RPC 부재로 에러 토스트가 뜬다.
