# Lighthouse 보고서 읽는 법

> 이 문서는 Lighthouse 성능 분석 보고서를 혼자 해석할 수 있도록 작성된 학습 가이드입니다.

---

## 1. Lighthouse란

구글이 만든 웹 성능 측정 도구. Chrome DevTools 또는 GitHub Actions 같은 자동화 환경에서 실행할 수 있다.

**핵심 질문:** "사용자가 이 페이지를 처음 열었을 때 얼마나 빨리 쓸 수 있게 되는가?"

**측정 환경 (중요):**
- 네트워크: 4G 속도로 강제 제한
- CPU: 실제보다 4배 느리게 강제 설정

개발 컴퓨터나 WiFi 환경에선 어떤 사이트든 빠르게 느껴지기 때문에 의도적으로 느린 환경을 시뮬레이션한다. 실제 한국 사용자 체감은 이보다 빠를 수 있다.

---

## 2. 브라우저가 페이지를 여는 순서

Lighthouse 지표를 이해하려면 브라우저가 페이지를 여는 순서를 알아야 한다.

```
① 서버에 HTML 요청
② HTML 받음                          ← TTFB 측정 지점
③ HTML 파싱 → CSS/JS 파일 목록 발견
④ CSS/JS 다운로드
⑤ JS 실행 (React 하이드레이션 등)
⑥ 화면에 뭔가 처음 그려짐             ← FCP 측정 지점
⑦ 가장 큰 콘텐츠가 화면에 그려짐      ← LCP 측정 지점
⑧ 사용자가 클릭/스크롤 가능해짐       ← TTI 측정 지점
```

---

## 3. 핵심 지표 (Core Web Vitals)

### TTFB (Time To First Byte)
서버가 HTML을 처음 보내주기까지 걸린 시간. ①→② 구간.
- 기준: ≤ 800ms
- 높으면: 서버/DB 문제

### FCP (First Contentful Paint)
화면에 뭔가 처음 그려지는 순간. 빈 흰 화면이 끝나는 시점.
- 기준: ≤ 1.8s

### LCP (Largest Contentful Paint)
화면에서 가장 큰 콘텐츠가 그려지는 순간. 사용자가 "로딩됐다"고 느끼는 순간.
- 기준: ≤ 2.5s
- 높으면: JS 블로킹 또는 이미지/리소스 로드 지연

### TBT (Total Blocking Time)
JS가 메인 스레드를 막아서 사용자 입력을 처리하지 못한 총 시간.
- 기준: ≤ 200ms
- 계산 방식: 50ms 초과 작업의 초과분만 합산

```
작업이 386ms 걸렸다면 → 336ms만 TBT에 합산
작업이 30ms 걸렸다면  → 0 (50ms 이하는 정상 범위)
```

### CLS (Cumulative Layout Shift)
페이지 로드 중 요소들이 갑자기 위치가 바뀌는 정도.
- 기준: ≤ 0.1
- 높으면: 이미지에 크기 미지정, 광고 삽입, 늦게 로드되는 폰트 등

### TTI (Time To Interactive)
사용자가 실제로 상호작용 가능해지는 시점.
- 기준: ≤ 3.8s

### SI (Speed Index)
페이지가 시각적으로 얼마나 빨리 채워지는지를 나타내는 복합 지표.
- 기준: ≤ 3.4s

---

## 4. Performance 점수 계산 방식

```
Performance =
  FCP   × 10%
  SI    × 10%
  LCP   × 25%
  TBT   × 30%  ← 비중 가장 큼
  CLS   × 25%
```

**결론:** TBT와 LCP가 점수에 가장 큰 영향을 준다. 이 두 지표를 먼저 고쳐야 점수가 많이 오른다.

---

## 5. LCP 분해 읽는 법

LCP가 높을 때 Lighthouse는 원인을 4단계로 쪼개서 보여준다.

```
TTFB         → 서버가 HTML 주는 시간
Load Delay   → 리소스 다운로드 대기 시간
Load Time    → 리소스 다운로드 시간
Render Delay → JS가 메인 스레드를 막아 렌더 지연
```

**Render Delay가 크면** → TBT 문제와 같은 원인 (JS 블로킹)

### 메인 스레드란

브라우저는 메인 스레드 하나로 JS 실행과 화면 그리기를 번갈아 처리한다.

```
[JS 실행] → [화면 그리기] → [JS 실행] → [화면 그리기]
```

JS가 실행되는 동안엔 화면을 못 그린다. JS가 오래 걸릴수록 사용자는 빈 화면을 더 오래 본다.

---

## 6. TBT 분해 읽는 법

긴 작업 목록에서 **같은 파일이 반복 등장하면** 그 파일이 주범이다.

파일 이름으로 정체 파악:
- `8d867abb...js` 같은 해시값 → 빌드 도구가 만든 청크 → 번들 분석 필요
- `googletagmanager.com` → 서드파티 스크립트 → 로드 타이밍 조정
- `turbopack-*.js` → Next.js 런타임

---

## 7. 주요 원인 패턴과 해결책

### 레거시 폴리필

**폴리필이란:** 구형 브라우저에 없는 기능을 JS 코드로 직접 구현해서 심어두는 것.

**문제:** `browserslist` 설정이 넓으면 Next.js/SWC가 아무도 쓰지 않는 구형 브라우저용 폴리필을 번들에 자동 포함한다. 폴리필은 크기도 크고 실행 시간도 길다.

**확인 방법:** TBT 긴 작업 목록에서 같은 해시 파일이 반복 등장하고, 그 파일이 200KB+이면 폴리필 의심.

**해결:** `browserslist` 설정을 현대 브라우저 기준으로 좁히기.

```
Chrome >= 90
Safari >= 16
Firefox >= 90
Edge >= 90
```

### 서드파티 스크립트 타이밍

**문제:** GTM, Google Analytics 같은 분석 스크립트가 페이지 초기 로드 시 실행되면 메인 스레드를 점령한다.

**해결:** 사용자 경험에 직접 필요하지 않은 스크립트는 페이지 로드 완료 후에 실행되도록 타이밍 조정 (`lazyOnload`).

### 불필요한 JS 번들 유입

**문제:** 특정 페이지에서 필요 없는 라이브러리가 공유 청크를 통해 유입되는 경우. 예: 홈페이지에 폼이 없는데 Zod(폼 검증 라이브러리)가 로드됨.

**원인 구조:**
```
로그인 페이지 → Zod import
→ Next.js가 공유 청크로 묶음
→ 홈 로드 시 공유 청크도 같이 로드
→ 홈에서 Zod 실행
```

**해결:** 동적 import(`dynamic()`)로 코드 스플리팅.

### 미사용 JS

Lighthouse는 다운로드했지만 실행되지 않은 코드도 측정한다. 다운로드 자체도 시간이 걸리므로 쓰지 않는 코드는 제거하거나 지연 로드해야 한다.

## 8. 보고서를 혼자 읽는 순서

### ① 종합 점수 확인

| 점수 | 평가 |
|------|------|
| 90+ | 좋음 |
| 70~89 | 보통 |
| 70 미만 | 문제 있음 |

### ② 어떤 지표가 낮은가

```
TBT 높음   → JS가 메인 스레드를 막고 있음
LCP 높음   → 화면에 주요 콘텐츠가 늦게 그려짐
FCP 높음   → 첫 화면 자체가 늦게 시작됨
CLS 높음   → 레이아웃이 로드 중 흔들림
TTFB 높음  → 서버/DB 문제
```

### ③ 비중 높은 지표부터 우선순위 잡기

TBT(30%), LCP(25%), CLS(25%) 순으로 고쳐야 점수 효율이 높다.

### ④ LCP가 높으면 — 4단계 분해 확인

```
Render Delay가 크다 → JS 블로킹 → TBT 문제와 연결
Load Time가 크다   → 이미지/리소스 크기 문제
TTFB가 크다       → 서버 문제
```

### ⑤ TBT가 높으면 — 긴 작업 목록 확인

- 같은 파일이 반복 등장하면 그 파일이 주범
- 파일명으로 서드파티인지 내 코드인지 구분
- 서드파티 → 로드 타이밍 조정
- 내 코드/빌드 청크 → 번들 분석

### ⑥ JS 번들 목록 확인

- 크기가 큰 파일과 실행 시간이 긴 파일 식별
- 미사용 비율이 높은 파일 → 코드 스플리팅 또는 제거

---

## 9. 스스로 판단하는 질문들

보고서를 보며 이 질문을 던지면 된다:

1. **TTFB가 낮은데 LCP가 높은가?** → 서버가 아니라 JS 문제. Render Delay 확인.
2. **TBT 주범이 내 코드인가, 서드파티인가?** → 서드파티면 타이밍 조정, 내 코드면 번들 분석.
3. **같은 파일이 긴 작업 목록에 반복 등장하는가?** → 그 파일 제거/분리가 최우선.
4. **미사용 JS가 많은가?** → 코드 스플리팅 또는 트리쉐이킹 필요.
5. **CLS가 높은가?** → 이미지에 width/height 미지정 또는 늦게 삽입되는 요소 확인.

## 10. 실전 예시 — 기강 앱 분석 흐름

```
Performance 49  → 심각
↓
TBT 1,810ms (비중 30%), LCP 5.6s (비중 25%)가 주범
↓
LCP 분해 → Render Delay 85% → JS 블로킹
↓
TBT 긴 작업 목록 → 8d867abb...js가 1위·3위·6위·7위·9위 반복
↓
해당 파일 = 레거시 폴리필 224KB (browserslist 설정 문제)
↓
TTFB 0ms → 서버/Supabase 리전 변경은 효과 없음
↓
해결 우선순위:
  Phase 1: browserslist 현대화 → TBT ~840ms, 점수 ~60~65
  Phase 2: GTM lazyOnload → TBT ~510ms, 점수 ~68~72
  Phase 3: Zod/Supabase 번들 분리 → 점수 ~74~78
```
