1. 핵심 테이블 2개

"칭호의 정의"와 "누가 어떤 칭호를 가졌는지"를 따로 관리합니다.

ttl_mst
칭호 마스터 — 어떤 칭호가 있는지 정의
  • ttl_idPK칭호 고유 ID
  • team_idFK어느 팀의 칭호인지 (팀별로 칭호가 따로 존재)
  • ttl_kind_enmauto(자동) / awarded(수여)
  • ttl_ctgr_cd카테고리 (러닝/철인/트레일/사이클/수여)
  • ttl_nm칭호 이름 (예: "신세계", "맛객")
  • ttl_rank카테고리 내 등급 (자동만, 수여는 0)
  • emoji_txt표시 이모지 (🏃 ⛰️ 🚴 🏅)
  • cond_rule자동 부여 조건 (JSON, 예: 풀마라톤 3시간 이내)
  • base_pt수여 칭호 기본 점수
  • use_yn운영에서 사용 중인지
mem_ttl_rel
회원-칭호 관계 — 누가 어떤 칭호를 보유 중인지
  • mem_ttl_idPK관계 고유 ID
  • team_idFK칭호가 속한 팀
  • team_mem_idFK팀-회원 관계 키 (team_mem_rel 참조)
  • ttl_idFK어떤 칭호인지 (ttl_mst 참조)
  • granted_at언제 부여됐는지
  • expires_at만료 시점 (수여만 사용)
  • granted_by_mem_idFK누가 줬는지 (수여 칭호)
  • granted_pt부여 시 점수 (수여만)
  • granted_rsn_txt부여 사유 텍스트
team_mst  ─ 1:N ─  team_mem_rel  ─ 1:N ─  mem_ttl_rel  ─ N:1 ─  ttl_mst
팀이 칭호 카탈로그를 보유 → 회원이 그중 하나를 보유

2. 칭호 카테고리 (TTL_CTGR_CD)

새로 만든 공통코드 그룹. 칭호는 5개 카테고리로 분류됩니다.

4. 수여 칭호 4종

기강단장이 직접 수여하는 칭호. 점수와 만료일은 수여 시점에 입력합니다.

🎖️ 서브현근 수여

  • 모임장보다 풀코스 빠른 회원

🎖️ 기강단장 수여

  • 기강 모임장

🎖️ 행동대장 수여

  • 운영진

🎖️ 맛객 수여

  • 맛집을 잘 아는 회원

5. 실제 동작 시나리오

홍길동이 풀마라톤 3시간 25분 기록을 등록하면 어떻게 되는지 따라가 봅니다.

예시 — "신세계" 칭호 자동 부여

1홍길동이 프로필에서 풀마라톤 3:25:00 기록 입력 → rec_race_hist 에 저장.
2서버 액션 recomputeAutoTitles(teamId, memId) 실행 — 4개 카테고리를 다시 계산.
3러닝 카테고리: 등급 9→1 순으로 검사. 최고존엄(3h 이내)·천상천하(3h10m 이내) 미충족 → 신세계(3h30m 이내) 충족.
4mem_ttl_rel 의 기존 러닝 칭호 정본 행을 vers > 0 으로 이력화하고, 신세계를 새 정본으로 INSERT.
5화면 노출 시 보유자 수와 활성 회원 수로 점수를 동적 계산 (DB에 점수 저장 안 함).

6. 알아두면 좋은 정책

  • 팀별 분리 — 같은 회원이 여러 팀에 속하면 팀마다 칭호가 따로 계산됩니다.
  • 자동 칭호는 카테고리당 1개 — 등급이 올라가면 이전 칭호는 이력으로 빠지고 새 등급이 정본이 됩니다.
  • 점수는 동적 — 자동 칭호 점수는 활성 회원 수 ÷ 보유자 수로 매번 계산. 희귀할수록 높음.
  • 수여 칭호 점수는 정적 — 부여 시 입력한 granted_pt 값을 그대로 사용.
  • 가입일 기준은 팀별 가입일 — team_mem_rel.join_dt 사용. 글로벌 가입일은 사용하지 않음.
  • 자동 부여는 서버에서만 — 회원이 자기 칭호를 임의로 INSERT 못하도록 RLS로 차단. 서버 액션(서비스 롤)에서만 부여.
